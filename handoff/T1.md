## Summary

Implemented market-wide and segment-level halt cascade for the securities-service. When a market or segment transitions to HALTED status, all instruments within that scope are automatically halted. When restored to ACTIVE, all instruments are restored.

## Design Decisions

1. **Filter by SegmentID client-side**: `InstrumentStore` has no `ListBySegment` method. Rather than add one, `cascadeSegmentStatusToInstruments` calls `List(InstrumentFilters{})` and filters in-memory. This is correct for MVP scale; a `ListBySegment` index can be added to the store later without changing handler logic.

2. **Cascade is best-effort**: Errors during individual instrument status updates or audit log writes are silently skipped. The market/segment status update has already succeeded; a partial cascade failure does not roll back the status. This matches the behavior of circuit breakers in trading systems (halt-first, reconcile later).

3. **Non-halted statuses do not cascade**: Only `MARKET_HALTED`/`MARKET_ACTIVE` and `SEG_HALTED`/`SEG_ACTIVE` trigger cascade. `MARKET_SUSPENDED`, `MARKET_CLOSED`, `SEG_SUSPENDED` do not cascade — a suspended market may be undergoing scheduled maintenance while instruments remain tradeable on other venues.

4. **Audit actor is "system"**: Cascade halts are system-initiated, not operator-initiated. The original operator action is already audited by the upstream market/segment status update handler.

5. **TenantID in audit entries**: Set to empty string `""` in cascaded audit entries. The tenant context is available via the `X-GarudaX-Tenant` request header but is not propagated into the cascade helpers for MVP. Follow-up: extract tenant from request context and pass through.
