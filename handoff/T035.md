## Summary

Designed the market-data-service for OHLCV candle aggregation from matching engine trade streams. Produced a comprehensive spec document covering TimescaleDB schema design (hypertable + hierarchical continuous aggregates for 6 intervals), gRPC API contracts (6 RPCs: GetCandles, StreamCandles, GetTicker, GetTickers, GetTrades, StreamTrades), real-time streaming architecture, data retention policies, and gateway integration endpoints. Created the protobuf contract file and a SQL migration (V8) with the full TimescaleDB schema.

## Key Decisions

1. **Separate service from matching engine** — The matching engine's `MarketDataService` (T007) serves raw order book snapshots and trade streams. This service consumes those streams and produces derived aggregated views (candles, tickers). This separation keeps the matching engine latency-critical path clean.
2. **TimescaleDB hierarchical continuous aggregates** — 1m candle is the base aggregate over the trades hypertable; 5m/15m aggregate over 1m; 1h aggregates over 1m; 4h/1d aggregate over 1h. This avoids redundant computation and leverages TimescaleDB 2.9+ hierarchical support.
3. **Dual ingestion: gRPC primary, Kafka fallback** — Primary data source is gRPC `StreamTrades` from matching engine for lowest latency. Kafka topic `exchange.trades.*` is used for replay/recovery after restarts or connection drops, ensuring no trade gaps.
4. **Append-only trades hypertable** — Follows the established pattern (T004, T007) with DELETE/UPDATE rules blocked. Trade busts are stored as separate `BUST` type events and excluded from continuous aggregates via WHERE clause.
5. **In-memory candle accumulators for real-time streaming** — Current (incomplete) candles are maintained in-memory per instrument per interval, updated on each trade, and broadcast to StreamCandles subscribers. This gives sub-50ms latency without waiting for continuous aggregate refresh.
6. **Decimal-as-string in protobuf** — Consistent with T007/T008 pattern for financial precision.
7. **Service named `MarketDataAggregateService`** — Avoids collision with matching engine's `MarketDataService` in the proto namespace.
8. **Ports 50057/8087** — Following the established port allocation convention.
