# Handoff — R019: compliance-service MCSD/FRC money→decimal

**Status:** DONE
**Agent:** Coder
**Phase:** R0-extension (money-float follow-ups from R006 audit)

---

## Summary
Converted the residual `float64` money paths in compliance-service's MSE
integration (MCSD) and regulatory reporting (FRC) packages to the shared
`github.com/garudax-platform/decimal` fixed-point type, and deleted both local
`round2` float helpers. Money is now computed and stored as `Decimal(18,4)` with
banker's rounding and overflow checking; non-money floats (risk scores,
percentages, price-change ranking, the dual-purpose corporate-action ratio,
share counts) were correctly left as `float64` per the R006 audit scope.

## Key Decisions
1. **Wired the shared module via filesystem `replace`** — added
   `require github.com/garudax-platform/decimal v0.0.0` +
   `replace ... => ../shared/pkg/types/decimal` to `compliance-service/go.mod`,
   matching the validated pattern used by securities-service and the six engines
   (R002/R003/R004). No `go.sum` change needed — the shared module is zero-dep.
2. **Money fields converted to `decimal.Decimal`:**
   - `integration/mcsd.go`: `DvPInstruction.SettlementAmount`,
     `TransferStatus.SettlementAmount`, `Entitlement.CashEntitlement`.
   - `reporting/frc.go`: `InstrumentVolume.Value`, `DailyTradingSummary.TotalValue`,
     `SettlementFail.FailValue`, `SettlementFail.PenaltyAmount`,
     `SettlementFailsReport.TotalPenalty`.
   `InstrumentVolume.Value` (the per-instrument notional that sums into
   `TotalValue`) was converted too — though not individually enumerated in the
   task, it is the same money path and converting it avoids a lossy per-row
   `float→decimal` boundary inside the sum.
3. **Dividend cash entitlement now exact** — `round2(float64(qty) * RatioOrAmount)`
   became `decimal.NewFromFloat(action.RatioOrAmount).TryMulInt64(qty)`. The
   `Try*` form is used because `RatioOrAmount` arrives from untrusted JSON;
   overflow/NaN now returns an error from `NotifyCorporateAction` instead of
   silently corrupting. `RatioOrAmount` itself stays `float64` because it is
   dual-purpose: a per-share **cash amount** for DIVIDEND but a **share ratio**
   for splits/rights (used to compute `int64` `ShareEntitlement`). It is consumed
   into decimal only on the money (DIVIDEND) branch.
4. **`round2` deleted from both files.** Its only money use (FRC `TotalPenalty`)
   is gone — the penalty total is now an exact `Decimal` sum, no post-rounding.
   Its one non-money use (FRC `LargeTraderPosition.PercentOutstanding`, a
   percentage) was replaced with `math.Round(pct*100)/100`, preserving the prior
   half-away-from-zero 2-dp behaviour without a money helper.
5. **CSV render unchanged in format** — `renderCSV` still emits `value` as
   fixed 2-dp via `strconv.FormatFloat(iv.Value.Float64(), 'f', 2, 64)`
   (`Float64()` is the sanctioned display-only accessor). Test row
   `MSE:APU,APU,1,10,50.00,0.5000` still matches.

## Behavioural deltas (intended)
- Dividend/penalty money now carries up to 4 dp instead of being float-rounded to
  2 dp — strictly more precise and consistent with the platform's 4-dp money
  reconciliation. Existing test values (2500, 1250, 20.00) are exact and unchanged.
- `TransferStatus.SettlementAmount` keeps its `,omitempty` tag, which is now a
  no-op for a struct type, so FoP transfer status JSON will include
  `"settlement_amount":0`. This is cosmetic and money-irrelevant; no test asserts
  its absence.
- Handlers (`mcsd_handlers.go`, `frc_handlers.go`) needed **no** code change —
  they decode/encode these structs as JSON, and `Decimal` implements
  `Marshal/UnmarshalJSON` accepting both JSON numbers and strings.
