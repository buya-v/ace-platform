# Handoff — R023: gateway reporting generator → decimal

**Status:** DONE
**Agent:** Coder
**Phase:** R-series (financial-correctness remediation)

---

## Summary
Migrated the latent money math in `gateway/internal/reporting` from `float64` to the
shared fixed-point `decimal.Decimal` type, closing the residual the R022 re-audit
mis-classified as "gateway reporting = non-money". Settlement-statement P&L, fee totals,
net amount, and market-summary OHLC/settlement prices are now computed and persisted in
`Decimal(18,4)` with half-even rounding and no float drift, silent overflow, or truncation
bias. The HTTP wire shape is unchanged (Decimal marshals as a bare JSON number).

## Key Decisions
1. **Money vs quantity split (per the R020 recipe + task scope).** Converted to `decimal.Decimal`:
   `PositionInput.{AvgPrice,MarkPrice}`, all `MarginInput` fields, all `FeeInput` fields,
   `TradeInput.Price`, both `PnLDetail` P&L fields, `DailyStatement.NetAmount`, and the five
   `MarketSummary` price fields (`OpenPrice/HighPrice/LowPrice/ClosePrice/SettlementPrice`).
   Kept as `float64` (genuine contract counts / ratios, not money — same call the recipe made
   for `fees.Volume30D` and the risk percentages): `PositionInput.Quantity`,
   `TradeInput.Quantity`, `PositionSnapshot.{LongQty,ShortQty}`, `MarketSummary.{Volume,OpenInterest}`,
   and `LargeTraderPosition.{NetPosition,GrossPosition,PctOfOpenInterest}`.
   `GenerateLargeTraderReport` is therefore unchanged — it operates purely on quantities/ratios.
2. **`roundTo4` retained, scoped to non-money values only.** It now rounds only the surviving
   float quantity/ratio fields (volume, open interest, net/gross position, pct-of-OI). Documented
   inline so the platform-wide "surviving `roundTo4` = missed money residual" sweep doesn't flag it
   as a false positive — the money path no longer touches it (Decimal is exact to 4 dp by construction).
3. **P&L computed multiply-then-aggregate in Decimal.** `unrealized = (mark - avg) * qty` uses
   `Decimal.Sub(...).MulDecimal(decFromFloat(qty))`; `MulDecimal` already rounds half-to-even to 4 dp,
   so the old `roundTo4` step was dropped. `netAmount = totalUnrealized.Sub(totalFees)`.
4. **DB shim mirrors R020 exactly.** Added a package-local `decFromFloat` in `store.go`;
   write money via `.Float64()`, read via `decFromFloat(...)`. **No migration was needed** — the
   reporting columns are already `DECIMAL(18,4)` (exact NUMERIC) in `V22__reporting_tables.sql`, so a
   value round-trips `Decimal → Float64 → NUMERIC(18,4) → Float64 → Decimal` losslessly for all
   realistic settlement amounts. (The R011 `*_raw` int64 column option was considered and judged
   unnecessary; see Follow-ups for the extreme-magnitude caveat.)
5. **Wire contract preserved.** No `json:` tags changed. Verified by new tests that `net_amount`,
   `open_price`, `settlement_price`, etc. serialize as bare JSON numbers, and that quantity fields
   (`volume`, `open_interest`) stay plain numbers.

## Blockers Found
None. The generator remains latent (no live caller — `handlers.go` serves pre-stored reports via the
`Store`; `main.go` wires a `NoOpStore`). Migrating before wiring, per the R022 directive.

## Suggested Follow-ups
- **R024-adjacent:** when the report-generation path is actually wired to a binary/scheduler, feed it
  registered-instrument data so its tests stop passing vacuously (same pattern as the engine Kafka gap).
- **Extreme-magnitude exactness (optional):** for notionals above ~1e11 with 4 dp, the `Float64()` DB hop
  loses the least-significant digit. If MSE-scale equities notionals approach that range, switch the
  reporting money columns to raw `int64` (`*_raw`) storage with `Raw()`/`DecimalFromRaw()` — a schema
  migration deferred here to stay on the validated R020 pattern.
- **CI sweep:** under R014, confirm the `roundTo2|roundTo4` float-helper grep treats this file's
  documented `roundTo4` (quantity/ratio only) as expected, not a residual.
