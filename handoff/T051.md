## Summary

Designed the complete event-driven integration layer for all 9 ACE platform services using Apache Kafka. The spec covers 9 event topics, their JSON schemas (aligned with existing Go struct types), consumer group mappings, partition strategies, idempotency patterns, DLQ design, retention policies, security ACLs, and operational monitoring. Deliverable is `docs/T051_kafka_event_spec.md`.

## Key Decisions

1. **Partition keys split by domain** — trade/market/settlement events partition by `instrument_id` (aligns with matching-engine's single-threaded-per-instrument design), while compliance/auth/margin-call events partition by `participant_id` (aligns with per-participant processing).
2. **16 partitions for high-throughput topics, 4 for low-throughput** — trade execution and clearing novation get 16 partitions for parallelism during peak trading; compliance and auth get 4 since they're infrequent.
3. **Extended clearing-engine's existing idempotency pattern** — the `processedTrades map[string]bool` pattern in `src/clearing-engine/internal/engine/engine.go` is standardized as the idempotency approach for all consumers, with a recommendation for database-backed dedup for critical services.
4. **JSON wire format initially** — chose JSON for debuggability and simplicity. Noted Protobuf as a future optimization for high-throughput topics.
5. **KRaft mode (no Zookeeper)** — modern Kafka deployment without Zookeeper dependency.
6. **Per-service SASL credentials with strict ACLs** — each service can only write to its owned topics and read from subscribed topics, preventing accidental cross-writes.
7. **3 retry attempts with exponential backoff before DLQ** — balances fast recovery from transient failures against flooding failing consumers.
