## Summary

Implemented the warehouse-service as a zero-dependency Go module at `src/warehouse-service/`. The service provides electronic warehouse receipt (eWR) lifecycle management (issue, transfer, cancel, pledge, release), facility registry, quality inspection workflow, delivery processing (initiate → complete/fail), double-entry inventory tracking, and facility capacity management. Ports: gRPC 50058, health HTTP 8088.

## Key Decisions

1. **Inspection-gated receipt issuance**: Receipt issuance requires a passed inspection record, enforcing quality control as specified in the T037 spec. Grade defaults to the inspection's assigned grade if not explicitly provided.
2. **Lot uniqueness enforcement**: One active receipt per facility/commodity/lot combination, tracked via an in-memory index that is updated on cancel and delivery completion.
3. **Double-entry inventory via event sourcing**: All inventory changes (deposits on issuance, withdrawals on cancel/delivery) are recorded as events. Current inventory is derived by aggregating events, matching the SQL view pattern from V8 migration.
4. **Capacity enforcement at receipt creation**: Available capacity is computed from active/pledged/delivery-pending receipts, preventing over-capacity at issuance time.
5. **Delivery failure reverts receipt to ACTIVE**: Failed deliveries restore the receipt to ACTIVE status so it can be re-delivered or otherwise managed.
6. **Append-only audit trail**: All receipt state transitions generate ReceiptEvent records for full audit trail (ISSUED, TRANSFERRED, PLEDGED, RELEASED, DELIVERY_INITIATED, DELIVERED, CANCELLED).
7. **In-memory store with SQL-ready interface**: The store layer mirrors the SQL schema from V8 migration. In production, swap the in-memory implementation for parameterized SQL queries against the `warehouse` schema.
8. **Collateralization as status transition**: Pledge moves receipt ACTIVE→PLEDGED (blocking transfer/cancel), release moves PLEDGED→ACTIVE. The clearing member ID must match for release.
