## Summary

Added a self-contained Go test module at **`tests/reconciliation/`** that exercises
the *real* shared money type (`github.com/garudax-platform/decimal`, the code R003
wired into the six engines and R004 wired into securities-service) via a filesystem
`replace` in `go.mod` — no network, no go.sum. It covers all four areas the task
called for:

1. **Mul/Div round-trip & rounding-direction invariants** — property-based.
2. **Overflow on large notionals errors instead of wrapping** — property-based + focused.
3. **Divide-by-zero errors** (Try* → error, convenience → panic; never a zero result).
4. **Cross-service reconciliation** — a trade priced/quantified the same way produces
   identical settlement cash in securities-service and the settlement-engine
   (the previous float-vs-int boundary), including a regression that pins *why* the
   float→decimal migration was needed.

**22 test functions, all PASS.** Clean under `go vet`, `gofmt -l`, `-race`, and
`-count=2` (deterministic — all randomized sweeps use a fixed seed).

## Key decisions

1. **`math/big` as an independent oracle.** Every Mul/Div property computes the exact
   integer result in `big.Int`, applies half-to-even rounding independently, and
   compares. This catches *wrong rounding* and *silent overflow* in one sweep: if the
   oracle value fits int64 the API must return it exactly; if it doesn't, the API must
   return `ErrOverflow` (never a wrapped value). The oracle is deliberately not the
   decimal package's own arithmetic.

2. **Everything lives under `tests/`.** The settlement formulas the task asks to
   reconcile live in service `internal/` packages, which other modules cannot import.
   Rather than touch service source (scope boundary: only `tests/` and `handoff/`), the
   two one-line formulas are *reproduced* from source (with file:line citations in the
   test comments) and both call the genuinely-shared `decimal` package. The reconciliation
   therefore tests the real money type both services depend on. See "Suggested follow-ups"
   for adding in-package `_test.go` if the scope is later widened.

3. **Float-boundary regression is concrete.** `4.015` is fractionally below 4.015 in
   float64, so `4.015 * 100` truncates to `401.4999` (one tick short of `401.5000`).
   The test asserts the old float path produced `401.4999` and that both services'
   decimal path now yields exactly `401.5000` — documenting the bug R004 removed.

4. **Deterministic randomness.** Fixed seed + per-test offsets, 20k iterations each.
   Failures are reproducible; CI is stable (verified with `-count=2`).

## Gaps / untestable

- **Cross-module import of service `internal/` settlement code** is not possible from a
  separate test module, so the reconciliation reproduces the formulas rather than calling
  the live functions. If a regression changes a service's formula without changing the
  shared type, this suite won't catch it (the per-service `_test.go` follow-up would).
- **Live process-to-process reconciliation** (securities-service ↔ settlement-engine over
  the wire) is out of scope and belongs to the e2e tier; the channel-based Kafka stubs do
  not bridge processes (see CLAUDE.md pattern, 2026-03-29).
