## The one test change made in this task (in scope: `tests/`)

`tests/kafka-e2e/propagation_test.go` — the `watcher` matched the correlation ID with strict `==`. The settlement engine re-keys its downstream event with the **deterministic idempotency key `cycle-<tradeID>`** (the documented R024 pattern) and carries that as `correlation_id` on `settlement.completed`, so the strict matcher **missed a genuinely-propagated event** (verified directly on the broker: `settlement.completed.correlation_id == "cycle-"+want`). Changed the match to **containment** on the unique nanosecond token, which keeps the assertion specific while tolerating the documented prefix. This makes the test reflect *real propagation* rather than the exact correlation encoding. Root cause was proven before the change (the event was visibly present in the `settlement.completed` topic for the exact correlation), so the change is not papering over a failure.

## Newly-surfaced downstream defects (file as R028 — NOT bug class #3, NOT R024 regressions)

This is the first time the chain was exercised by a **real gateway-submitted** trade against a **live broker**, which surfaced three data/config/topic gaps *downstream* of the (working) bus:

- **D1 — matching publishes trades with empty participant/order IDs.** The real matching-engine `trades.executed` payload has `buyer_participant_id=""`, `seller_participant_id=""`, `buy_order_id=""`, `sell_order_id=""`. Clearing **received** the event (cross-process propagation works) but **rejected** it: `clearing: novation failed: novation: both buyer and seller participant IDs are required`. The synthetic `kafka-e2e` flow only works because it hand-sets `P-BUY`/`P-SELL`. Fix: populate participant + order IDs on the matched `types.Trade` before publish in matching-engine.
- **D2 — margin has no risk parameters seeded for the demo instrument.** Margin **consumed** `clearing.novated` (all 3 synthetic events) but failed: `margin engine: calculation failed for P-BUY: params: no risk parameters for instrument WHT-HRW-2026M07-UB`. So `margin.call-issued` is never produced for the real/synthetic flow (kafka-e2e treats margin as best-effort, so this does not fail it). Fix: seed margin risk params for `WHT-HRW-2026M07-UB` in the deployed margin-engine.
- **D3 — DLQ topics do not exist.** When a consumer exhausts retries it publishes to `ace-commodities.dlq.<topic>`, but those topics aren't created/auto-created: `DLQ publish failed: [3] Unknown Topic Or Partition`. Failed events are therefore neither processed nor durably dead-lettered (dropped after retries). Fix: pre-create `ace-commodities.dlq.*` (or enable auto-create for the DLQ path).

Severity: **P2** (the propagation invariant — bug class #3 — is satisfied; these block the *economic completion* of a real trade, not the event transport). Suggest scheduling **R028** to drain D1–D3 so a real gateway-submitted trade settles end-to-end on the bus.
