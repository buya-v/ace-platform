## Key Decisions
1. Order book steps (section 9) are public — no auth headers, matching the admin-ui pattern where these are public gateway endpoints.
2. POST/PUT endpoints (settlement trigger, circuit breaker set, halt, resume) use inline `(status) => (status >= 200 && status < 300) ? 'PASS' : 'FAIL'` validators instead of `okValidator` — same logic but explicit per task spec.
3. Settlement trigger `extractState` handles both `cycle_id` and `id` response shapes for resilience.
4. Monitoring section step IDs use `mon-` prefix — the readiness checklist already had `mon-1/2/3` IDs but those are `ChecklistItem` IDs, not `StepDefinition` IDs. Since `getAllSteps()` only collects from `Section.steps` (not checklist items), there's no collision.
