# Handoff — R004: Remove float money math in securities-service

**Status:** DONE
**Agent:** Coder
**Phase:** R0 financial-correctness remediation

---

## Summary
Completed the migration of securities-service money fields from `float64` to the
shared `decimal.Decimal` type (`../shared/pkg/types/decimal` via go.mod replace).
The prior partial commit (28f40af) had converted the type definitions and the
`pg_store` scan/write boundary; this change finishes the work by fixing every
arithmetic/comparison site in the matching engine, auction engine, settlement
engine, and ~25 HTTP handler DTOs, and by migrating all affected unit tests.
`go build ./...`, `go vet ./...`, and `go test ./...` (7 packages) all pass; core
money packages also pass `-race`.

## Key Decisions
1. **Decimal-native arithmetic for all in-scope money math.** Position cost/MV/PnL
   (`engine.go`, `auction.go`), settlement `NetAmount = price.MulInt64(qty)`
   (`settlement/engine.go`, `equities.go`), and report/firm-view value aggregates
   now use `Add`/`Sub`/`MulInt64`/`DivInt64`/`MulDecimal`. These use the panic-on-overflow
   convenience methods (loud failure on corrupt money, per the R002 spec).
2. **Weighted-average cost preserves precision by dividing last.** AvgCost =
   `(oldQty*oldAvgCost + tradeQty*tradePrice) / newQty`, computed as
   `oldAvgCost.MulInt64(oldQty).Add(price.MulInt64(qty)).DivInt64(newQty)`.
3. **Accrued-interest decomposition reworked for precision (important).** The first
   pass folded `days*rate/basis` into one `NewFromFloat` factor, which rounds to 4dp
   and lost too much precision (ACT/365 test asserts a 0.0001 tolerance). Both
   `calcAccruedInterest` (bond handler) and `calculateAccruedInterest` (settlement)
   now compute `parValue * couponRate * days / basis`, converting only `couponRate`
   (a genuine ratio) via `NewFromFloat` and keeping `days`/`basis`/`quantity` as
   exact integer `MulInt64`/`DivInt64`, **dividing by the basis LAST**. This keeps
   half-even rounding only at the final money scale.
4. **`Float64()` only at genuine non-money boundaries.** Circuit-breaker engine
   (float64 reference prices — out of scope), tick-size validation, surveillance
   thresholds/scores, `math.Remainder`, price-range query filters, market-data
   `PriceLevel`/`TickerData` display floats, `TradeCorrection.Original/CorrectedPrice`,
   `TradingParameterSet.MaxOrderValue`, and `Entitlement.EntitlementValue` all stay
   float64 per the task scope; Decimal is converted to float only at these edges.
5. **Decimal is a comparable struct, so `==`/`!=` work directly.** Order/auction
   sort and `pricesCross` use `.Equal/.LessThan/.GreaterThan/.GreaterThanOrEqual`;
   map keys that were `map[float64]` (auction price ladder) became `map[types.Decimal]`.
6. **Test ergonomics via a `decLit` helper.** Each affected test package got a tiny
   `dechelper_test.go` (`decLit(f float64) types.Decimal`). Numeric money literals in
   fixtures/assertions were wrapped with `decLit(...)`; printf/`%f` args on Decimals
   use `.Float64()`. This was applied with a compiler-guided script and verified.
7. **Wire contract preserved.** `Decimal.MarshalJSON` emits a JSON number and
   `UnmarshalJSON` accepts number-or-string (R002), so request DTOs that switched
   `float64 → types.Decimal` (bond `par_value`, csd `settlement_amount`, service-desk
   `price`) and all JSON responses keep the exact external shape the SPAs/e2e consume.
   Existing handler tests that assert `result["..."].(float64)` still pass.
8. **Kept the diff focused.** Reverted ~24 files that `gofmt -w` had only reformatted
   (pre-existing alignment drift unrelated to R004). `types.go` retains a gofmt
   realignment because it is a primary deliverable and must stay gofmt-clean.

## Suggested Follow-ups
1. **R005 reconciliation:** securities `SettlementObligation.NetAmount`
   (`price.MulInt64(qty)` + bond accrued) should be reconciled against the
   settlement-engine cash leg. A starter `TestCrossServiceReconciliation` lives in
   the shared decimal module.
2. **Out-of-scope money-ish floats** intentionally left as `float64` (not in the task
   field list) — convert in a future pass if desired: `Entitlement.EntitlementValue`
   (+ `handlers_corporate_actions.go:233` dividend math), `TradeCorrection.Original/
   CorrectedPrice`, `OffBookTrade.Price`, `ReferencePrice.Price`, `PriceLevel.Price`,
   `TickerData.*Price`, `CustodyBalance.AvgCost`, `TradingParameterSet.MaxOrderValue`,
   `CircuitBreaker`/`PostTradeParams` rate fields.
3. **pg_store boundary** uses `decFromFloat`/`.Float64()` against numeric columns. For
   full precision at rest, consider migrating to `*_raw int64` columns + `Raw()`/
   `DecimalFromRaw()` (no proto/DB migration needed per R002) in a later DB task.
4. **Reviewer note:** the float↔Decimal shims to scrutinise are the `.Float64()` edges
   in decision #4 and the accrued-interest precision decomposition in #3.
