## Summary

Created a comprehensive Go-based end-to-end integration test suite at `tests/e2e/` that exercises the full ACE Platform trading flow through the gateway HTTP API. The tests are zero-dependency (standard library only) and skip gracefully when the gateway is not running.

**20 top-level test functions** covering:

- `TestAuthInvalidToken` — Garbage token and expired token return 401
- `TestPasswordChange` — Change password and verify login with new password
- `TestTokenRefresh` — Refresh token flow + invalid refresh token rejection

## Key Decisions

1. **Standard library only** — Uses `net/http`, `encoding/json`, `testing`. Zero external dependencies.
2. **Graceful skip pattern** — Every test checks gateway reachability first (`t.Skip` if unavailable). Backend service unavailability (503/502) also triggers skip for individual steps.
3. **Token field flexibility** — Checks both `access_token` and `AccessToken` variants since Go JSON marshaling may use either depending on struct tags.
4. **E2E_BASE_URL configurable** — Defaults to `http://localhost:8080`, overridable via environment variable.
5. **Test isolation** — Each test generates unique emails using `uniqueEmail()` with nanosecond timestamps to prevent cross-test interference.

## Gaps / Notes

- **No WebSocket tests** — `golang.org/x/net` is unavailable and raw HTTP upgrade with `net` stdlib would add complexity for marginal value. Can be added when websocket support is a priority.
- **Backend services use StubBackendClient** — When the gateway runs with its default stub backend, all forwarded requests return 503. The e2e tests handle this by skipping those steps. Full end-to-end validation requires all 9 services running behind the gateway.
- **No settlement trigger endpoint** — The gateway routes don't include a POST endpoint for triggering settlement cycles (only GET for listing/getting). The task description mentioned `POST /v1/settlement/cycle` but this doesn't exist in the current gateway routes. Tests cover the available GET endpoints.
