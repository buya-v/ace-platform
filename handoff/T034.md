# Handoff — T034: API Gateway Service

**Status:** DONE
**Agent:** Coder

## Summary

Implemented the ACE API Gateway as a zero-dependency Go module at `src/gateway/`. The gateway provides REST-to-gRPC translation for all 6 backend services (matching-engine, clearing-engine, margin-engine, settlement-engine, auth-service, compliance-service) with JWT authentication, token-bucket rate limiting, structured JSON logging, WebSocket upgrade support, and graceful shutdown. All 51 REST endpoints from the T033 spec are mapped, plus 3 WebSocket endpoints and 2 health endpoints.

## Key Decisions

1. **Zero-dependency Go module** — Following the established pattern, the gateway uses only Go stdlib. JWT validation uses HMAC-SHA256 with `crypto/hmac`. WebSocket upgrade uses raw HTTP hijacking. No external dependencies.
2. **Interface-based backend client** — Defined `proxy.BackendClient` interface so the forwarding layer can be tested with mocks. The `StubBackendClient` returns 503 until real gRPC clients are wired in.
3. **Custom HTTP router with path parameter extraction** — Built a simple pattern-matching router that handles `{param}` placeholders in URL patterns. Path params are extracted and placed in query parameters for handler access.
4. **Middleware chain** — RequestID → Logging → BodyLimit → Auth → RateLimit → Router. Public paths skip auth via configurable whitelist (exact paths and prefix matches).
5. **Token bucket rate limiting per user per endpoint group** — 9 rate limit groups matching the T033 spec (order_submit=50/s, order_cancel=100/s, etc.). Falls back to IP-based limiting for unauthenticated requests.
6. **Port 8080 HTTP, Port 8090 health** — Following the T033 spec. Health server runs on a separate port for Kubernetes probes.
7. **JWT claims forwarded as metadata** — Validated claims are attached to context and forwarded to backends as `x-user-id`, `x-participant-id`, `x-roles` metadata fields.

## Blockers Found

None. The T033 spec was comprehensive enough to implement all endpoints without ambiguity.

## Suggested Follow-ups

- **Real gRPC client implementation** — Replace `StubBackendClient` with actual gRPC clients using the connection pool in `proxy/pool.go`. This requires the matching-engine, clearing-engine, etc. to expose gRPC services.
- **Circuit breaker** — The T033 spec calls for circuit breakers per backend (3 failures → 30s open). This should be added to the `BackendClient` implementation.
- **CORS headers** — Not included since the spec doesn't mention CORS, but browser clients will need it.
- **WebSocket streaming** — Currently WebSocket handlers only send heartbeats. Real trade/book streaming requires connecting to backend gRPC server-streaming RPCs.
- **Integration testing** with actual backend services running.
