## Key Decisions

1. **Append-only receipt event model**: All receipt state changes are recorded in `warehouse.receipt_events` — receipts are never deleted, only transitioned to terminal states (CANCELLED, DELIVERED). This provides a complete audit trail.
2. **Double-entry inventory tracking**: Physical commodity movements use an event-sourcing model in `warehouse.inventory_events` with positive/negative quantities. Current inventory is derived via aggregation, never stored as a mutable balance.
3. **Receipt collateralization as status change**: Pledge/release is modeled as a receipt status transition (ACTIVE↔PLEDGED) with the margin engine notified via Kafka events. The warehouse service does NOT compute collateral values — that's the margin engine's responsibility.
4. **Facility capacity computed, not stored**: Available capacity is derived from `total_capacity - SUM(active receipt quantities)` via a database view, avoiding stale cached values.
5. **Inspection-gated issuance**: Receipt issuance requires a PASSED inspection record. This enforces quality control at the database level via the inspection_id foreign key.
6. **Port 50058/8088**: Following the established port convention — gRPC on 50058, health HTTP on 8088.
7. **Partial delivery via receipt splitting**: When a receipt covers more than the delivery obligation, the receipt is split atomically — obligation quantity delivered, remainder stays as new ACTIVE receipt.
8. **Idempotency keys table**: All mutating RPCs support idempotency via a `warehouse.idempotency_keys` table with 24-hour TTL to prevent duplicate operations.
