# Handoff — R025: Fix DB init (zero-pad migration filenames + reconcile duplicate market_data.trades)

**Status:** DONE

## Summary
Fixed the P1 infra defect (from R015's fresh full-stack run) that prevented a clean
Postgres from initializing. Two sub-defects, both in `infrastructure/db/migrations/`:
(1) renamed every migration to zero-padded 3-digit form (`V1__`→`V001__`, … `V33__`→`V033__`)
so `docker-entrypoint-initdb.d` lexicographic ordering equals version ordering; and
(2) neutralized the stale duplicate `market_data.trades` definition in
`V008__market_data_timescaledb.sql` so the authoritative `V016__market_data_tables.sql`
(traded_at shape, the shape the service code uses) wins. **Verified end-to-end on a fresh
TimescaleDB volume via Docker: 0 init errors, all `ace_*` schemas + `auth`/`platform` present,
`ace_market_data.trades` is a hypertable on `traded_at`.**

## Key Decisions
1. **Zero-pad to 3 digits (`V001__`…`V033__`), not 2.** Highest current version is 33, and
   the project plans more (V029–V033 are recent). 3 digits is future-proof and unambiguous.
   Used `git mv` so history is preserved. Version gaps (no V2–V5; R017 dropped the empty stubs)
   are intentional and tolerated by both `initdb.d` ordering and Flyway.
2. **Neutralized all of V008's conflicting body, not just the `CREATE TABLE`.** V008 created
   `market_data.trades(executed_at …)` PLUS OHLCV continuous aggregates (`ohlcv_1m`..`ohlcv_1d`),
   retention policies, and grants to role `garudax_marketdata_svc`. All of those depend on the
   `executed_at` shape and conflict with V016's `traded_at` shape. They are also **dead**:
   `grep -rn ohlcv src/` returns nothing (the service uses V016's plain `candles` table, not the
   continuous aggregates), and `garudax_marketdata_svc` is a misspelling of the canonical
   `garudax_market_data_svc` (V001/V030). So V008 is reduced to a harmless idempotent no-op that
   only ensures the timescaledb extension + `market_data` schema exist before V016 runs. A long
   header comment documents exactly what was removed and why.
3. **Kept V016 as authoritative (unchanged).** Confirmed via
   `src/market-data-service/internal/store/postgres.go`: every query uses `ace_market_data.trades`
   with `id` + `traded_at` + `ON CONFLICT (traded_at, id)`, plus `ace_market_data.candles` /
   `tickers` — all the V016 shape.
4. **docker-compose needs no change.** It bind-mounts the whole `migrations/` directory
   (`./infrastructure/db/migrations:/docker-entrypoint-initdb.d`), so renaming files inside it
   does not break the mount. No Flyway config, deploy manifest, or k8s manifest references
   migration filenames (verified by grep).
5. **Updated filename references in code comments + the runbook (comment-only, no logic change).**
   `src/platform-service/internal/types/types.go`, `…/store/store.go` (V29→V029) and
   `src/warehouse-service/internal/store/postgres.go` (V9→V009) had doc-comments naming the old
   files; updated to keep them accurate. These are comment-only edits — no application logic
   changed (honoring "do not change application source" while satisfying "update any references").
   `docs/DEMO_RUNBOOK.md` infra-readiness table updated to mark DB migrations Working (R025).

## Verification
Ran a true fresh-DB bring-up (Docker daemon + `timescale/timescaledb:latest-pg15` image both
available in the worker env):
```
docker run -d --name r025-verify-pg -e POSTGRES_* \
  -v $(pwd)/infrastructure/db/migrations:/docker-entrypoint-initdb.d:ro -p 55432:5432 \
  timescale/timescaledb:latest-pg15
```
Results:
- `PostgreSQL init process complete; ready for start up.` — init finished.
- **`grep -c "ERROR:"` over the full init log = 0.** Only benign NOTICEs (idempotent
  `CREATE SCHEMA IF NOT EXISTS` skips) and TimescaleDB "best practices" WARNINGs.
- Schemas present: `ace_clearing, ace_compliance, ace_exchange, ace_margin, ace_market_data,
  ace_participants, ace_reference, ace_securities, ace_settlement, ace_warehouse, auth, platform`.
- `ace_market_data.trades` columns: `id, instrument_id, price, quantity, trade_value,
  aggressor_side, trade_type, sequence_number, traded_at` (V016 shape — **no `executed_at`**).
- `ace_market_data` tables: `candles, tickers, trades` (all V016).
- No `ohlcv*` continuous aggregates remain (V008 successfully neutralized).
- `timescaledb_information.dimensions`: `trades | traded_at` (hypertable on the right column —
  this is exactly the `CREATE INDEX ... (traded_at)` step that previously aborted init).
- Throwaway container removed after verification (no impact on the running stack).

Confirmed the bug pre-fix: `LC_ALL=C ls *.sql` previously sorted `V10__…V33__` and `V6__…V9__`
all before `V1__`, so `V11__matching_engine_tables.sql` (needs the `exchange` schema from V1) ran
first. After zero-padding, `LC_ALL=C ls` order == version order (`V001, V006, V007, V008, V009,
V010, … V033`).
