# Handoff — R008: Fix engine concurrency bugs

**Status:** DONE
**Agent:** Coder
**Phase:** R0 backlog (engine concurrency)

---

## Summary
Fixed three classes of concurrency bug across the clearing, margin, and settlement engines: handler callbacks invoked while holding the engine mutex (deadlock/contention risk), an unguarded concurrent map read in the settlement engine, and unsynchronized writes to handler fields. Each engine now snapshots its handler under the lock and invokes it after unlock, guards the settlement instrument-config read with the read lock, and synchronizes all `Set*Handler` writes with the existing mutex. Added `-race`-clean concurrency tests proving no deadlock and no data race.

## Key Decisions
1. **Snapshot-under-lock, invoke-after-unlock (not `atomic.Value`).** Fix (3) offered `atomic.Value` or set-before-serve+document. I instead guard each handler field with the struct's *existing* mutex and snapshot it inside the critical section, then call the handler after `Unlock()`. This solves fix (1) and fix (3) together with one consistent idiom, avoids `atomic.Value`'s gotchas for `func` types (typed-nil store, type-consistency panics), and needs no new fields. Set-before-serve is no longer required — handlers may now be set concurrently and safely.
2. **Extracted `*Locked` helpers** (`clearTradeLocked`, `calculateMarginLocked`, `runSettlementCycleLocked`, `runMultiInstrumentCycleLocked`) that own the `defer Unlock()` and return the handler (+ a value snapshot of the data passed to it). The thin public method invokes the handler outside the lock. This preserves every existing error path and the exact handler-firing semantics (e.g. settlement still does NOT fire the handler on the early P&L-failure path; margin fires `marginHandler` before the margin-call evaluation).
3. **Settlement `sync.Mutex` → `sync.RWMutex` + split accessor.** `getInstrumentConfig` now takes `RLock` so it is safe standalone (it is also called directly by tests with no lock held). Its only in-engine caller, `runMultiInstrumentCycleLocked`, already holds the **write** lock, so it calls a new lock-free `lookupInstrumentConfig` to avoid a non-reentrant `RLock`-under-`Lock` deadlock. `GetCycle`/`GetAllCycles` were downgraded to `RLock` for concurrent reads.
4. **Margin engine `CalculateMargin` now calls `callService.Evaluate` outside `e.mu`.** `Evaluate` has its own mutex and (after this change) fires its own handler outside *its* lock; running it without `e.mu` held means the margin-call handler never fires under any engine lock. `margincall.issueOrUpdateCall` changed its return from `(*MarginCall, error)` (error always nil) to `(*MarginCall, bool)` where the bool flags a newly-issued call that must fire the handler.
