# T067 — Playwright Admin Dashboard Verification Tests

## Summary

Created `tests/playwright/admin-verification.spec.ts` — a 13-test Playwright E2E suite that verifies demo runbook outcomes are visible in the admin dashboard.

## Test Coverage

| # | Test | What it verifies |
|---|------|-----------------|
| 1 | Setup: Run demo registration steps | Runs reg-1 through reg-6 in demo runner, verifies all complete |
| 2 | Verify Participants in Admin | Logs into admin, navigates to Participants, checks for trader emails |

| 13 | Verify Market Phase page | Checks instrument state cards |

## Key Implementation Details

- **Serial execution**: Tests use `test.describe.serial` — Test 1 sets up data, Tests 2+ verify admin views
- **SPA navigation**: Admin UI stores auth tokens in React state (in-memory). After login, navigation uses sidebar NavLink clicks instead of `page.goto()` to preserve the session
- **Login helper**: `loginAsAdmin()` fills the form, intercepts the `/auth/login` API response for diagnostics, and waits for dashboard redirect
- **Sidebar helper**: `navigateAdminSidebar()` expands collapsed sidebar sections and clicks NavLinks
- **Soft assertions**: All content checks use `expect.soft()` so individual failures don't block subsequent tests
- **Screenshots**: All 13 tests capture screenshots to `tests/playwright/test-results/admin-*.png`

## Gaps / Notes

- Admin login depends on the demo runner having registered the admin user first (Test 1). If the backend auth service is down, all admin tests will soft-fail on login
- Tests 4, 6, 7, 10, 11 use soft assertions because the pages may show empty states (no positions, no margin calls, etc.) depending on backend state
- The admin UI uses in-memory auth (no cookies/localStorage) — each test that needs admin access must re-login via `loginAsAdmin()` since `test.describe.serial` provides a fresh page per test

## Suggested Follow-ups

- Add localStorage-based token persistence to admin-ui so tests can share login state across tests without re-authenticating
