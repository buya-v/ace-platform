# Handoff — R011: Make gateway tenant enforcement real

**Status:** DONE
**Agent:** Coder
**Phase:** R2 (tenancy)

---

## Summary
Closed the live P1 multi-tenancy defect: the gateway `tenantBypassPrefixes` list
had grown to whitelist nearly every tenant-scoped business path, so
`TenantMiddleware` skipped header validation and invoked handlers with no tenant
in context (200 where 401/403 is required). Narrowed the bypass list to genuine
platform-level prefixes only, and now forward the resolved tenant to backends as
the canonical `X-GarudaX-Tenant` header in `BackendRequest.Metadata`. Existing
single-tenant clients keep working by sending the `ace-commodities` header (as
admin-ui/web-ui already do; the e2e suites now send it too).

## Key Decisions
1. **Bypass list narrowed to platform-level only.** `tenantBypassPrefixes` is now
   exactly `/platform/`, `/api/v1/platform/`, `/api/v1/auth/` (plus the existing
   exact-match health paths `/healthz`, `/readyz`, `/metrics`). Every business
   prefix the task named (orders, clearing, margin, settlement, warehouse,
   participants, compliance, market-data, securities, admin) — and the
   previously-exempt `ws`, `tickets`, `bot`, `instruments`, `screening`,
   `risk-scores`, `securities/demo` — now require the `X-GarudaX-Tenant` header.
   This matches the repeatedly-stated learned-pattern action verbatim.
2. **`/api/v1/auth/` KEPT bypassed despite appearing in the task's removal
   list.** The task parenthetical listed `/auth` among "trading endpoints," but
   (a) the platform directive states "Auth is platform-level; all other schemas
   are per-tenant," (b) login/register/refresh happen before a tenant is
   resolved, and (c) the authoritative learned pattern lists `/api/v1/auth/` as
   one of the genuine platform-level bypasses to KEEP. Favoured the directive +
   learned pattern over the parenthetical. `/api/v1/admin/` is NOT bypassed
   (admin actions like HaltInstrument are tenant-scoped).
3. **Added an optional `RouteChecker` to `TenantMiddleware` for a pre-enforcement
   404 guard.** Because tenant runs BEFORE auth in the chain, an unknown path
   without a header would otherwise return 401 TENANT_REQUIRED instead of 404 —
   reintroducing the documented "auth-before-routing 401-vs-404" bug at the
   tenant layer. The middleware now returns 404 for paths matching no registered
   route before enforcing tenant, mirroring the Auth middleware's existing guard.
   Signature is `TenantMiddleware(validTenants []string, routeChecker ...RouteChecker)`
   — variadic so the existing `TenantMiddleware([]string{...})` call sites and
   tests compile unchanged; `main.go` passes the router (`rt`) as the checker.
4. **Tenant forwarded as `X-GarudaX-Tenant` (not a bespoke `tenant_id` key).**
   `BackendRequest.Metadata` is forwarded to backends as HTTP headers
   (`proxy/httpclient.go`), so injecting under the shared `tenant.HeaderName`
   means downstream services that adopt the shared tenant middleware (e.g.
   securities-service) read the exact value the gateway validated. Re-exported
   the constant as `middleware.TenantHeaderName` and added a small
   `addTenantMetadata(r, meta)` helper called in both metadata-building sites
   (`handler.forward` and `handler.SubmitOrder`).
5. **e2e clients send the default tenant header.** Both e2e request chokepoints
   (`tests/e2e/e2e_test.go` `apiClient.do`, `tests/e2e/lifecycle_test.go`
   `lifecycleClient.do`) now set `X-GarudaX-Tenant: ace-commodities`, so the
   full-stack suite behaves like a real single-tenant client and does not
   regress to 401 on every authed call. The `unknown_endpoint_returns_404` test
   still gets 404 (valid header → route checker → not found).

## Blockers Found
- **None blocking.** Two follow-ups worth flagging:
  - **Browser WebSocket (`/api/v1/ws/`) now requires the header.** Browsers
    cannot set custom headers on the WS handshake, so live web-ui streams will
    need tenant via query-param/subprotocol resolution. Deferred — see follow-ups.
  - **Demo-reset paths (`/api/v1/admin/demo/`, `/api/v1/securities/demo/`)** now
    require a tenant header; the demo-runner client must send one.

## Suggested Follow-ups
- **R012 (tenant tests):** add gateway HTTP + gRPC enforcement coverage. The unit
  tests here cover bypass/enforce/404/forward; R012 should add full-stack e2e
  assertions (missing header → 401, unknown tenant → 403, cross-tenant isolation)
  and wire the shared gRPC interceptors onto backend gRPC servers so the tenant
  propagated via `X-GarudaX-Tenant` is enforced backend-side too.
- **WebSocket tenant resolution:** add a `/api/v1/ws/` tenant source that reads a
  query param or subprotocol (browsers can't send the header on WS handshake),
  then either keep `ws` enforced via that path or add a WS-specific resolver.
  Track as a dedicated task; do NOT simply re-add `/api/v1/ws/` to the bypass list.
- **Demo paths:** ensure the demo-runner sends `X-GarudaX-Tenant`, or add a
  scoped demo-reset bypass if demo reset is genuinely platform-wide.
- **Reconcile the task spec vs directive on `/auth`:** the task removal list
  named `/auth`; this handoff kept it bypassed per the platform directive +
  learned pattern. If the reviewer disagrees, removing the `/api/v1/auth/` entry
  is a one-line change — but note the auth public-path config and login flow.
