# T045 — Settlement Engine Test Coverage Improvement

## Summary

Added comprehensive test coverage for three business-critical packages in `src/settlement-engine/`:

- **payment** — 10 new tests: empty instructions, all-failed processing, confirmed/failed timestamp verification, sequential reference uniqueness, copy safety on GetPayments, multi-amount settlement, large batch (100 instructions), mixed success/failure across participants
- **pnl** — 12 new tests (including 10-case table-driven): batch error propagation (closes the 85.7% gap), empty batch, zero-sum multi-instrument verification, new position short loss, fractional prices, record field validation, NetByParticipant edge cases (empty, single, zero-sum, multi-instrument)
- **valuation** — 10 new tests (including 8-case table-driven): multiple instruments, price overwrite, 3-day price chaining, non-consecutive day gap, wrong instrument/date errors, instrument/date preservation, multi-instrument HasPreviousPrice

## Coverage Results

| Package | Before | After |

| `internal/pnl/` | 96.3% | 100.0% |

| **Total** | **98.9%** | **100.0%** |

**Total tests across 3 packages: 53** (32 new + 21 existing)

Note: The task description listed coverage at 27-28% per package, but actual baseline coverage was already 96-100%. The only real gap was `CalculateBatch` error path at 85.7% in pnl. That is now closed.

## Key Test Additions

- **CalculateBatch error propagation** — the only actual coverage gap; verifies errors from missing prices propagate correctly through batch processing
- **Table-driven P&L scenarios** — 10 cases covering long/short, profit/loss, new positions, zero diff, single contract, large positions, negative prices
- **Table-driven ValuePosition** — 8 cases including zero price, fractional prices, large positions
- **Price chaining** — verifies 3-day consecutive pricing and non-consecutive day gap behavior
- **Zero-sum invariant** — multi-instrument batch verification that total variation margin across matched positions is zero

## Gaps Identified

None — all three packages are at 100% statement coverage.

## Suggested Follow-ups

- The settlement-engine has additional packages (`internal/engine/`, `internal/settlement/`) that were not in scope. Consider coverage improvement tasks for those if needed.
- Consider adding concurrent access tests for `valuation.Store` and `payment.InMemoryGateway` since both use mutexes — while the locking code is covered, race condition tests under `go test -race` would add confidence.
