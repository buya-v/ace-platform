# Handoff — R014: Re-enable CI build+test gate

**Status:** DONE
**Agent:** Coder
**Phase:** R-series (pipeline infrastructure / deferred backlog)

---

## Summary
Re-enabled the GitHub Actions CI pipeline as an active **build + test gate**. `ci.yml`
now builds and unit-tests every Go module in the repo (17 modules, including the
nested `shared/pkg/*` modules and `tests/reconciliation`) under `-race` with coverage
reporting, on every PR and push to `main`/`develop`, fronted by a single required
`CI Gate` status check. The four cloud/infra workflows (`deploy`, `docker-build`,
`terraform`) were activated but **guarded to manual `workflow_dispatch` only** so they
never auto-deploy on merge; `security` was activated for PR/push/scheduled scanning.
All 69 tests in `tests/cicd/test_workflows.py` (the CI contract) now pass — they were
20-failing before this change.

## Key Decisions
1. **Treated `tests/cicd/test_workflows.py` as the acceptance contract.** It asserts all
   five workflows exist as active `.yml`, that `ci.yml` has `lint`/`test`/`build`/`ci-gate`
   jobs, and that `.github/branch-protection.json` requires `CI Gate` + `Security Gate`.
   This is what "re-enable CI" concretely means here, so I satisfied it rather than only
   renaming the one CI file. branch-protection.json already existed and was correct — left
   as-is.
2. **Reconciled the contract with "keep cloud-deploy disabled/guarded" by guarding triggers,
   not by keeping files `.disabled`.** `deploy.yml`, `docker-build.yml`, and `terraform.yml`
   now trigger on `workflow_dispatch` **only** (auto `push`/`workflow_run`/`pull_request`
   triggers removed, with an explanatory comment). They cannot run on PR/merge, so no
   image push, EKS deploy, or `terraform apply` ever happens automatically; the deploy/apply
   jobs additionally keep their GitHub `environment` approval gates. This satisfies both the
   file-existence contract and "this is a build+test gate, not a deploy."
3. **Dynamic module discovery instead of the old hardcoded `src/*/` loop.** A `discover`
   job emits a JSON matrix from `find src tests/reconciliation -name go.mod`. The old
   `ci.yml.disabled` only walked `src/*/` one level deep, which **missed the nested
   `src/shared/pkg/tenant` and `src/shared/pkg/types/decimal` modules** (and `go test ./...`
   from `src/shared` won't cross module boundaries). Building all modules every run also
   avoids the false-green risk of change-detection.
4. **Bumped Go to 1.25 with `GOTOOLCHAIN: auto`.** The old workflow pinned `go 1.22`, but
   12 of 17 modules declare `go 1.25.0` and `shared` declares `1.24.0` — the gate would have
   failed to build them. Same fix applied to `security.yml` (CodeQL/govulncheck).
5. **Coverage is measured and reported but NOT gated.** The old `ci.yml.disabled` failed
   builds below 60% *total* coverage; verified locally that real services (e.g. gateway) have
   intentionally-untested packages (`websocket`, `types`, `cmd` at 0%) that drag total
   coverage well under 60%, so that gate would wrongly block all merges. Scope here is
   build+test, so coverage is emitted via `::notice` only. (Per the learned coverage-
   methodology pattern, business-logic coverage is the meaningful metric, tracked elsewhere.)
6. **Lint is non-gating.** `lint` runs `gofmt` (warn-only) + `go vet ./...` with
   `continue-on-error: true` and is excluded from `ci-gate`. golangci-lint was dropped (v1.57
   predates go1.25 and would break the run); `go vet` gives static checking without a brittle
   external pin.
7. **E2E handled per the task: separate manual job.** `e2e-fullstack` runs `tests/e2e`
   only on `workflow_dispatch` (`continue-on-error`), excluded from the gate — the suite
   skips gracefully without a live stack, so gating on it would be vacuous. `tests/load` is
   excluded entirely (needs a running system).
8. **`-race` on every module** (not just the four engines) — strictly stronger, and aligns
   with the "race detector is the verification tier for engine concurrency fixes" pattern.

## Blockers Found
- **None blocking.** Note: the host toolchain is go1.22.5, so the `go 1.25` modules can only
  be built in CI (where setup-go installs 1.25) or locally via `GOTOOLCHAIN=auto` network
  fetch. I validated the exact command chain (build + vet + `go test -race` + coverage) end-
  to-end on the go1.22 modules buildable offline: `shared/pkg/types/decimal` (92.1%),
  `tests/reconciliation`, and the real `gateway` service (1.22.0, includes the R011 tenant
  middleware) — all green.

## Suggested Follow-ups

  contract is now green. When standing up the stack, the manual `e2e-fullstack` job (or
  Docker Compose + `GATEWAY_URL`) exercises the e2e tier for real.
- **Branch protection is config-as-data only.** `.github/branch-protection.json` must be
  applied to the actual GitHub repo (REST `PUT .../branches/main/protection` or a settings-
  sync action) for the `CI Gate`/`Security Gate` required checks to actually block merges.
  The workflow file alone does not enforce it.
- **Frontend not in the gate (deliberate).** The 6 SPAs (web-ui, admin-ui, demo-runner, …)
  are not built/tested by this Go-scoped gate, and per the current learned pattern they have
  ~26 failing fixture-drift tests. A separate (non-blocking first) `frontend` job should be
  added *after* the R013-class fixture refresh lands, so it doesn't red-wall every merge.

  `docker-build`/`deploy`/`terraform`) is a dedicated deploy task — out of scope here. The
