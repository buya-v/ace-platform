## Summary

Re-enabled an active GitHub Actions build+test+validate gate (`ci.yml`) that runs
on every PR/merge to `main`/`develop`, builds and unit-tests every discovered Go
module, and gates merges on a single `CI Gate` status check. This rework fixes the
fatal interaction that got the first attempt REJECTED: a required status check
(`Validate`) that no longer reported on PRs, which would have bricked **all** merges
to `main`. Cloud-mutating steps (image push, EKS deploy, `terraform apply`) remain
guarded; only read-only infra validation/plan now runs on PRs.

## Key Decisions

1. **`terraform.yml` â€” restored `pull_request` trigger; guarded only the mutating
   apply jobs.** Changed `on:` from `workflow_dispatch`-only back to
   `pull_request: [main, develop]` + `workflow_dispatch`. The read-only
   `terraform-validate` (job name **`Validate`**) and `terraform-plan` jobs now run
   on PRs again â€” they mutate nothing (`init -backend=false` / `plan` only). The
   three mutating `terraform-apply-{dev,staging,prod}` jobs changed from
   `if: push && ref==main` (auto-apply on merge) to
   `if: github.event_name == 'workflow_dispatch'`, so they run **only** on manual
   dispatch and each still carries its `environment:` approval gate. This is the
   root-cause fix: it is why `Validate` reports on PRs again.

2. **`branch-protection.json` â€” required contexts now exactly equal the PR-reporting
   checks.** The required set for `main` is `["CI Gate", "Security Gate", "Validate"]`.
   After fix #1, all three report on a normal PR (`CI Gate` from `ci.yml`,
   `Security Gate` from `security.yml`, `Validate` from `terraform.yml`). No required
   context is unreachable, so merges are not blocked. (No edit was needed to the file
   itself â€” fix #1 made its existing contents correct. The danger was the *reverse*:
   a required check that never reports.) `develop` requires only `CI Gate`, which also
   reports on PR. **This file is data only â€” it must be `PUT` to the repo via the
   GitHub API / Settings â†’ Branches to take effect.**

3. **Added Terraform validation to the CI gate.** New `terraform-validate` job in
   `ci.yml` runs `terraform fmt -check -recursive` + `init -backend=false` +
   `validate` against `infrastructure/terraform`, and is wired into `ci-gate`'s
   `needs` + result check. Malformed/cross-referenced HCL (the T003/T054/R017 bug
   class) now fails the merge gate. No cloud credentials and no state mutation, so it
   is safe to run on every PR.

4. **Removed the two dead `deploy.yml` promote jobs.** `promote-staging` /
   `promote-prod` gated on `github.event.workflow_run.head_branch == 'main'`, a
   context that can never be set now that `deploy.yml` is `workflow_dispatch`-only â€”
   they were permanently unreachable. Removed them (and documented that staging/prod
   are reached by dispatching `deploy.yml` directly with the `environment` input).
   This also eliminates a dangerous implicit devâ†’stagingâ†’prod auto-promotion path.

5. **Lint and e2e stay non-gating.** `lint` is `continue-on-error`; `e2e-fullstack`
   runs only on `workflow_dispatch` (the suite gracefully skips without a live stack).
   The `CI Gate` aggregates `build` + `test` + `terraform-validate` only â€” a true
   build+test+validate gate, not a deploy.
