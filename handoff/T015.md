## Summary

Designed the complete KYC/AML compliance architecture for the ACE Platform. Produced a 13-section spec covering participant onboarding workflows (5 participant types, 11-state lifecycle), document verification pipeline, watchlist screening with pluggable provider interface, quantitative risk scoring model (6 weighted factors, 4 tiers), and ongoing transaction monitoring (6 rules). Created gRPC protobuf API contracts (3 services: OnboardingService, ScreeningService, ComplianceAdminService with 20 RPCs, 9 enums, 30+ message types) and a SQL migration adding 7 new tables to the `compliance` schema.

## Key Decisions

1. **Append-only screening results, risk scores, and SAR filings** — follows T004's immutability pattern with PostgreSQL RULES blocking UPDATE/DELETE. Ensures tamper-evident audit trail per AML/CFT requirements.
2. **Risk-based tiered approach** — 4 risk tiers (Low/Medium/High/Prohibited) drive onboarding approval gates and monitoring frequency. Low/medium auto-approve; high requires compliance officer; prohibited auto-rejects.
3. **Pluggable screening provider interface** — watchlist screening uses an adapter pattern (`ScreeningProvider` Go interface) so sanctions data providers (Dow Jones, Refinitiv, etc.) can be swapped without changing core logic.
4. **Fuzzy name matching with Jaro-Winkler + Double Metaphone** — handles Cyrillic transliteration challenges for Mongolian names; threshold at 0.85 for escalation, 0.95 for auto-reject.
5. **Separation of compliance check from order flow** — exchange engine calls `CheckParticipantStatus` RPC (cached in Redis with 60s TTL) rather than embedding compliance logic, keeping services decoupled.
6. **Risk score versioning** — scores are never overwritten; model version changes trigger batch re-scoring. Enables audit of historical risk assessments.
7. **6 transaction monitoring rules** — covering single-trade limits, volume spikes, structuring detection, wash trading, dormant account activation, and cross-border payments.
