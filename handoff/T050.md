## Summary

Built the ACE Admin Dashboard as a React 18 + TypeScript + Vite SPA at `src/admin-ui/`. The dashboard provides exchange operators and compliance officers with views for system health monitoring, participant/KYC management, margin calls, settlement status, circuit breaker controls, warehouse receipts, compliance alerts, and audit log. Role-based access control restricts compliance officers to Participants, Compliance, and Audit sections only.

## Key Decisions

1. **Role-based route guards via `RoleGuard` component**: Admin-only routes (monitoring, margin, settlement, circuit-breakers, warehouse) are wrapped in a `<RoleGuard allowedRoles={['admin', 'exchange_admin']}>` layout route. Compliance officers accessing these routes are redirected to `/dashboard`.

2. **Polling intervals aligned with spec**: Health checks poll every 30s, margin calls every 10s, participants every 30s, settlements every 15s, circuit breakers every 10s, warehouse every 60s, compliance alerts every 30s. Audit log uses manual refresh (no auto-polling).

3. **In-memory JWT token management**: Access token stored in module-level variable (not localStorage) per security best practice. 401 responses trigger automatic logout via `onUnauthorized` callback.

4. **CSS Modules for styling**: Each page and component has its own `.module.css` file, avoiding global style conflicts. Added `vite-env.d.ts` with CSS module type declarations for TypeScript compatibility.

5. **Dev server on port 3001**: As specified in task requirements, Vite dev server configured for port 3001 with API proxy to `localhost:8080`.

6. **Zero heavy dependencies**: Only React, React Router, and Vite dev tooling. State management via React Context + useReducer. Data fetching via native `fetch` with typed wrappers.

## Suggested Follow-ups

- **Gateway Enhancement**: The `GET /api/v1/admin/health` aggregated health endpoint needs to be added to the gateway service to support the System Health dashboard.
- **E2E Testing**: Component-level integration tests with React Testing Library would increase confidence in form interactions (KYC approve/reject, circuit breaker halt/resume).
- **WebSocket Upgrade**: The polling-first approach works but could be enhanced with WebSocket streams for margin calls and settlement status for lower-latency updates.
- **ParticipantDetail page**: The spec includes a `/dashboard/participants/:id` detail view with document viewer and KYC timeline — this route is not yet implemented.
