## Summary
Drained the file-level repo hygiene backlog left after the manual branch cleanup
(119 stale branches already deleted). Filled in the Terraform stub modules that
`main.tf` wires in (vpc, security-groups, elasticache — the last reconnected to
its already-declared `redis_*` variables), removed the two truly orphan empty
modules (iam, s3), deleted the V2–V5 zero-byte Flyway migration stubs, and
documented `corporate-actions` as a library module (no entrypoint by design).

## Key Decisions
1. **vpc + security-groups: implemented, not removed.** Both are referenced by
   `main.tf`, and `eks`/`rds`/`msk` consume `module.vpc.{vpc_id,
   private_app_subnet_ids, private_data_subnet_ids}` — so a `terraform apply`
   would fail with empty modules. Wrote a real three-tier VPC (public /
   private_app / private_data across 3 AZs, one NAT gateway + private route
   table per AZ) exposing exactly the outputs the downstream modules read, plus
   shared ALB/app security groups. Matched the existing module style (header
   block, `local.identifier`, `common_tags = merge(var.tags, {Module=..,
   TaskID="INFRA-1"})`).
2. **elasticache: implemented and wired in (not removed).** The root
   `variables.tf` already declares `redis_node_type / redis_num_shards /
   redis_replicas_per_shard` and `dev.tfvars.example` sets them — clear intent.
   Removing the module would have orphaned declared+exemplified variables, so I
   finished it (cluster-mode Redis, EKS-node-only ingress, platform-CMK at-rest,
   TLS in-transit) and added the `module "elasticache"` block to `main.tf`
   consuming the vpc data subnets + eks node SG, mirroring the rds/msk pattern.
3. **iam + s3: removed.** Unlike elasticache, these had zero supporting context —
   no variables, no tfvars entries, and not referenced anywhere in
   `main.tf`/deploy/CI. They were pure orphan empty dirs, so deletion is the
   correct hygiene action (avoids over-engineering speculative modules).
4. **V2–V5 migrations: removed.** Zero-byte stubs; real DDL is V1 then V6+. No
   runner/config/doc references them (Flyway tolerates the version gap and the
   current DB already works with these as no-ops). Deletion is behaviour-
   preserving and removes dead stubs.
5. **corporate-actions: documented as a library, not given an entrypoint.** It
   is `package corporateactions` with no external importer and is absent from
   every service list (compose/k8s/CI) — so there was nothing to "drop." Adding
   a `main.go`/`Dockerfile` would be speculative over-engineering. Added a
   `README.md` stating it is a library (like `src/shared`), why it has no
   entrypoint, and that it will be wired into a binary in Phase 0.8. Confirmed
   `go build ./...` and `go test ./...` pass unchanged.

## Blockers Found
- `terraform` is **not installed** in this environment, so `terraform fmt` /
  `validate` / `plan` could not be run. The HCL was written to match the
  existing rds/msk/eks modules' style and the exact input/output names the
  composition requires; a `terraform validate` in CI (R014) should confirm.

## Suggested Follow-ups
- **R014 (CI gate):** add `terraform fmt -check -recursive` + `terraform
  validate` for `infrastructure/terraform/` so module wiring regressions
  (missing outputs, type mismatches) fail automatically — they currently can't
  be caught locally without the binary.
- **mse-equities / Phase 0.8:** when corporate-actions is wired to a binary, add
  the consuming service's `cmd/.../main.go` + `Dockerfile` and register it in
  the service lists; update `src/corporate-actions/README.md` to point at the
  consumer.
- The VPC layout assumes `vpc_cidr` is at least a /20 (uses `cidrsubnet(.., 4,
  ..)`); the default /16 is fine. If a smaller CIDR is ever set, revisit the
  `newbits` math.
