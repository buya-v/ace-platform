# Handoff — T029: Daily Settlement

**Status:** DONE
**Agent:** Coder
**Phase:** 2

---

## Summary

Built a daily mark-to-market settlement engine for commodity futures as a standalone Go module (`settlement-engine`). The engine executes end-of-day settlement cycles: fetching settlement prices, calculating variation margin (daily P&L) for all positions, generating net settlement instructions per participant, and submitting payments via a pluggable payment gateway. All 35 tests pass including concurrency tests.

## Key Decisions

1. **Separate Go module (`settlement-engine`)** — follows the clearing-engine/margin-engine/matching-engine pattern with its own `go.mod`, zero external dependencies, and identical Decimal type for cross-service compatibility.
2. **Variation margin as core P&L** — daily P&L is calculated as `(currentSettlementPrice - previousSettlementPrice) * netQuantity` for existing positions, or `(currentSettlementPrice - avgEntryPrice) * netQuantity` for new positions. This is the standard futures mark-to-market approach.
3. **Net settlement per participant** — P&L across all instruments is netted per participant before generating payment instructions. Each participant receives exactly one instruction (PAY_IN or PAY_OUT), reducing payment volume.
4. **Zero-sum validation** — total pay-in always equals total pay-out, verified in tests. This is a fundamental property of mark-to-market settlement.
5. **Payment gateway as interface** — `payment.Gateway` is an interface allowing the engine to be wired to any payment/banking rail. Dev uses `InMemoryGateway` with configurable failure injection.
6. **Settlement price store with previous-day tracking** — the valuation store automatically links each day's settlement price to the previous day's price, enabling seamless daily P&L calculation.
7. **Server on port 50054/8084** — avoids collision with matching-engine (50051/8081), clearing-engine (50052/8082), and margin-engine (50053/8083).
8. **Cycle lifecycle** — PENDING → VALUING → CALCULATED → SETTLING → COMPLETED/FAILED, with error capture at each stage.

## Suggested Follow-ups

- **Margin engine integration** — before processing settlements, check margin adequacy via the margin-engine's `CalculateMargin` to ensure participants have sufficient collateral (as suggested in T028 handoff).
- **Clearing engine integration** — consume positions from clearing-engine's `position.Manager.GetAll()` / `GetByInstrument()` to feed into settlement calculations. The `Position.NetQuantity` and `Position.AvgEntryPrice` fields map directly.
- **Market data integration** — wire settlement price feeds from the market-data-service to `Engine.SetSettlementPrice()` for automated end-of-day price ingestion.
- **Real payment gateway** — implement a production `payment.Gateway` backed by banking/payment rails (e.g., RTGS, ACH) instead of the in-memory stub.
- **gRPC API** — define settlement-specific proto (RunSettlementCycle, GetCycle, GetSettlementInstructions RPCs) and wire to the server layer.
- **Settlement scheduling** — add a scheduler/cron to trigger daily settlement cycles at market close.
- **Audit trail** — persist settlement cycles, P&L records, and payment results to a database for regulatory reporting.
- **Physical delivery settlement** — extend for commodity delivery scenarios (warehouse receipts, delivery instructions) beyond cash-settled variation margin.
