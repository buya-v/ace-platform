## Summary

Created 5 GitHub Actions workflows covering the full CI/CD lifecycle: build/test/lint (ci.yml), security scanning (security.yml), container image build & push to ECR (docker-build.yml), Terraform plan/apply with workspace selection (terraform.yml), and deployment to dev/staging/prod EKS clusters with promotion chain (deploy.yml). Includes branch protection configuration and 69 passing tests.

## Key Decisions

1. **Service-aware change detection** — CI and Docker workflows detect which services changed and only build/test those, using a matrix strategy for parallelism.
2. **OIDC for AWS auth** — All AWS credential steps use `role-to-assume` (OIDC federation), never static access keys. Requires `AWS_CI_ROLE_ARN` and `AWS_DEPLOY_ROLE_ARN` secrets.
3. **Sequential Terraform apply** — Apply runs dev → staging → prod sequentially, each gated by a GitHub Environment approval. Plan runs on all 3 environments in parallel on PRs with results posted as PR comments.
4. **Terraform workspace selection** — Uses `terraform workspace select <env>` matching the T002 workspace strategy (dev/staging/prod), with `-var-file="environments/<env>.tfvars"`.
5. **Deploy promotion chain** — On main branch: auto-deploy to dev, then promote to staging (with approval), then prod (with approval). Each uses a GitHub Environment gate.
6. **Automatic rollback** — Deploy job runs `kubectl rollout undo` if rollout status fails within timeout.
7. **Security-in-depth** — CodeQL, Trivy (filesystem + IaC config + built images), govulncheck, and Gitleaks secret detection. Weekly scheduled security scan.
8. **Branch protection as code** — `.github/branch-protection.json` defines rules for main (CI Gate + Security Gate required, 1 review, no force push) and develop (CI Gate required, 1 review).
