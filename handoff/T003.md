## Summary

Implemented the full EKS Terraform module with 5 managed node groups (system, exchange-core, app-general, monitoring, data-pipeline), OIDC provider for IRSA, VPC CNI with prefix delegation, KMS-encrypted secrets, and Karpenter autoscaler infrastructure (IRSA role, SQS interruption queue, instance profile). Created Kubernetes manifests for Istio mTLS enforcement (STRICT mode mesh-wide + per-namespace), Karpenter NodePools with EC2NodeClass, and NodeLocal DNSCache DaemonSet. Helm values provided for Istio (base, istiod, gateway) and Karpenter.

## Key Decisions

1. **5 node groups with taints** — exchange-core, monitoring, and data-pipeline node groups use taints to ensure workload isolation; system and app-general are untainted for general scheduling.
2. **VPC CNI prefix delegation** — Enabled `ENABLE_PREFIX_DELEGATION` and `ENABLE_POD_ENI` for higher pod density per node, critical for a commodity exchange platform.
3. **KMS encryption for K8s secrets** — Envelope encryption with auto-rotating KMS key, as required for financial platform compliance.
4. **Istio STRICT mTLS + TLS 1.3 minimum** — Mesh-wide PeerAuthentication with STRICT mode, plus per-namespace policies for ace-exchange and ace-services. Outbound traffic restricted to REGISTRY_ONLY.
5. **Karpenter spot interruption handling** — SQS queue + EventBridge rules for graceful spot instance draining. Exchange-core NodePool uses on-demand only; app-general and data-pipeline allow spot.
6. **NodeLocal DNSCache on link-local 169.254.20.10** — Standard Kubernetes NodeLocal DNSCache with system-node-critical priority, tolerates all taints to run on every node.
7. **Kustomize overlays** — Dev reduces Karpenter limits; prod increases them. Staging uses base defaults.
8. **OIDC provider from T002 handoff** — Used the EKS module's OIDC provider for Karpenter IRSA, as T002 specified.

## Suggested Follow-ups

- T006 (CI/CD) should use the `cluster_name` and `cluster_endpoint` outputs from the EKS module for kubectl/helm configuration
- T005 (Auth Service) should use `oidc_provider_arn` output for creating service account IRSA bindings
- Application services deploying to `ace-exchange` or `ace-services` namespaces will automatically get Istio sidecar injection (label: `istio-injection: enabled`)
- Karpenter Helm install requires passing Terraform outputs: `cluster_name`, `cluster_endpoint`, `karpenter_interruption_queue_name`, `karpenter_controller_role_arn`
- The `EC2NodeClass` in `karpenter-nodepools.yaml` has placeholder variables (`${CLUSTER_NAME}`, `${KARPENTER_INSTANCE_PROFILE}`) that must be templated at deploy time
- Consider adding NetworkPolicies alongside Istio AuthorizationPolicies for defense-in-depth
