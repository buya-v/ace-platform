# Handoff — R007: Kafka fail-fast wiring

**Status:** DONE
**Agent:** Coder
**Phase:** R-series (deferred backlog drain)

---

## Summary
Stopped the silent in-process Kafka fallback across all 9 services. Each
`internal/kafka/wiring.go` `NewProducerFromEnv` / `NewConsumerFromEnv` now FAILS
FAST (`log.Fatal`) when `KAFKA_BROKERS` is unset outside of unit tests, instead
of silently returning a `ChannelProducer`/`ChannelConsumer` that cannot cross
process boundaries. The in-process channel adapter is now reachable from
production code only under `testing.Testing()`, via explicit constructors
(`newInProcessProducer` / `newInProcessConsumer`, and matching-engine's existing
`NewTradeProducer`) — never as an env-absence default.

## Key Decisions
1. **Test-mode signal = `testing.Testing()` (stdlib), not an env var.** This is
   the strongest "strictly unit tests" guard: it is true only inside a `go test`
   binary and can never be set accidentally by an operator in production. Using
   an env var (e.g. `KAFKA_ALLOW_INPROCESS`) was rejected because it would
   re-introduce a foot-gun that could silently drop cross-service events in a
   real deployment.
2. **`log.Fatal` over returning an error.** The `*FromEnv` constructors return
   bare `Producer`/`Consumer` (no error) and are startup-time wiring; the task
   explicitly permits `log.Fatal`. Keeping the signature avoids churning every
   (future) caller. Note: `*FromEnv` currently has no `cmd/main.go` callers yet —
   only tests — so this is forward-looking hardening for when binaries wire it.
3. **Explicit in-process constructors.** Extracted `newInProcessProducer()` /
   `newInProcessConsumer(groupID, dlqProducer)` (unexported) so the channel
   adapter has a clearly-named, test-only entry point rather than being inlined
   in the env-absence branch. matching-engine reuses its existing exported
   `NewTradeProducer` for this role.
4. **Testable predicate `kafkaBrokersConfigured()`.** Centralised the broker
   presence check (`strings.TrimSpace(os.Getenv("KAFKA_BROKERS")) != ""`),
   replacing the duplicated `brokers != "" && len(strings.TrimSpace(brokers))>0`.
   Behaviour is identical (whitespace-only → not configured) and it is directly
   unit-tested with an env matrix.
5. **Existing tests stay green unmodified.** The pre-existing
   `TestNewProducerFromEnv_NoBrokers` / `_EmptyBrokers` assert a `*ChannelProducer`
   when brokers are unset; under `testing.Testing()` the new code still returns
   the in-process adapter, so those tests pass without edits. Only the fatal path
   (production, no brokers) is new behaviour — and it is unreachable from any test
   binary by construction, so it cannot be exercised in-process (documented, not
   tested, per the standard treatment of `log.Fatal`).

## Scope
- **In scope (9 services):** matching-engine, clearing-engine, margin-engine,
  settlement-engine, market-data-service, warehouse-service, auth-service,
  compliance-service (producer + consumer where present); gateway (consumer only);
  matching/warehouse (producer only).
- **Out of scope:** securities-service `internal/kafka` has no `*FromEnv`
  env-absence fallback (its `Producer` is injected; `nil` is a no-op), so there
  is nothing to fail-fast on there.

## Suggested Follow-ups
1. **R008/R009 (engine concurrency):** when these wire `*FromEnv` into the actual
   `cmd/*/main.go` startup, rely on this fail-fast — set `KAFKA_BROKERS` in all
   deploy manifests (docker-compose, K8s) so engines abort loudly on misconfig.
2. **Deploy manifests:** confirm `KAFKA_BROKERS` is present in every service's
   env in `deploy/` and `infrastructure/`; with R007, an omission now crashes the
   pod at startup (intended) rather than silently dropping events.
3. **Real wire adapter:** the cross-service e2e gap (channel stubs don't bridge
   processes) still requires a real Kafka broker behind the `Producer`/`Consumer`
   interfaces for full-stack e2e (see the "channel-based Kafka stubs" pattern).
   R007 makes the misconfiguration impossible to hit silently; it does not itself
   provide the wire-protocol adapter.
