## Key decisions / gotchas discovered (useful for future live-verification tasks)

1. **Match downstream hops by what each event actually carries, not a single
   universal token.** `trades.executed`/`clearing.novated` carry the participant
   IDs; `settlement.completed` re-keys correlation to `cycle-<tradeID>` (contains
   the trade_id); **`margin.call-issued` carries NEITHER the trade_id NOR the
   instrument** — it is keyed by the margin-engine `CallID` (`mc-N`) and its only
   stable correlator is `participant_id`. The seeded test therefore matches the
   trade by the **unique per-run participant ID** (present in trades/novated/
   margin payloads) and matches settlement by the server-assigned `trade_id`.
   A first cut that matched margin by `trade_id` produced a false "HOP 3 BROKE"
   while the call had genuinely fired — the economic flow was complete; the test's
   match token was wrong.

2. **Read from `FirstOffset`, never `LastOffset`, for a fresh-stack consumer.**
   A brand-new consumer group reading "latest" races the producer: if the group's
   first rebalance completes after the event is produced, the reader seeks past it
   and silently skips it. On a freshly brought-up stack the only traffic is the
   test's, so reading from the start is race-free as long as matching uses a
   **unique per-run token** (the run-stamped participant ID), which never
   false-matches a prior run's residual events.

3. **The broker's advertised listener is `kafka:9092`** (unreachable from the
   host). The produce path (gateway → matching, in-network) is unaffected, but a
   host-side Kafka *consumer* cannot fetch partitions. Run the Go test suites from
   **inside the compose network** (ephemeral `golang:1.22` container on the
   project network, host module cache mounted). This is the documented way to run
   `tests/kafka-e2e`/`tests/e2e` against this stack.

4. **R028 D2 verified at the runtime:** `margin-engine` logs `Seeded SPAN risk
   parameters for 1 instrument(s) from built-in demo default`, and a real trade
   produces `MARGIN CALL: participant=P-R029-BUY-… deficit=5025000` → the margin
   call IS issued for the seeded instrument (zero-collateral participant ⇒
   deficit ⇒ call). This is why R029 can make `margin.call-issued` a REQUIRED hop
   whereas the synthetic transport test keeps it best-effort.

## How to reproduce

```bash
# from the worktree root
cat > .env.r029 <<'EOF'
POSTGRES_PORT=15432
KAFKA_PORT=9092
ZOOKEEPER_PORT=12181
INSTRUMENTS=WHT-HRW-2026M07-UB
EOF
docker compose --env-file .env.r029 build matching-engine clearing-engine margin-engine settlement-engine auth-service compliance-service platform-service gateway
docker compose --env-file .env.r029 down -v
docker compose --env-file .env.r029 up -d postgres zookeeper kafka \
  matching-engine clearing-engine margin-engine settlement-engine \
  auth-service compliance-service platform-service gateway
# ensure DLQ + cross-service topics exist
for t in trades.executed clearing.novated margin.call-issued settlement.completed; do
  docker exec garudax-kafka kafka-topics --bootstrap-server localhost:9092 --create --if-not-exists --topic "ace-commodities.$t"      --partitions 3 --replication-factor 1
  docker exec garudax-kafka kafka-topics --bootstrap-server localhost:9092 --create --if-not-exists --topic "ace-commodities.dlq.$t"  --partitions 3 --replication-factor 1
done
# run the suites from inside the network
NET=$(basename "$PWD")_garudax-network
docker run --rm --network "$NET" --user "$(id -u):$(id -g)" \
  -v "$PWD":/work -v ~/go/pkg/mod:/go/pkg/mod \
  -e GOPATH=/go -e GOMODCACHE=/go/pkg/mod -e GOCACHE=/tmp/gocache -e GOFLAGS=-mod=mod -e HOME=/tmp \
  -e KAFKA_BROKERS=kafka:9092 -e GATEWAY_URL=http://gateway:8080 -e E2E_BASE_URL=http://gateway:8080 \
  -w /work/tests/kafka-e2e golang:1.22 go test -v -timeout 180s ./...
docker run ... -w /work/tests/e2e golang:1.22 go test -v -timeout 240s ./...
```
