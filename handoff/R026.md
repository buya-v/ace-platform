# Handoff — R026: Fix Dockerfile build contexts so images build from current source

**Status:** DONE
**Agent:** Coder
**Phase:** R-series (deploy-pipeline remediation)

---

## Summary
Fixed the P1 deploy-tier defect from R015's fresh full-stack run: fresh service
images could not be built from the committed Dockerfiles because each service
used a narrow `./src/<svc>` build context, while after R002 (`shared/pkg/types/decimal`)
and R010 (`shared/pkg/tenant`) the consuming modules use relative
`replace ... => ../shared/...` directives that point **outside** that context
(`replacement directory ../shared/... does not exist`). Widened the build context
to `./src` for every service whose `go.mod` consumes a shared module and rewrote
those Dockerfiles to copy both the service dir and `shared/`, preserving the
relative `../shared` layout. **Verified with real `docker compose build` for all
12 Go services** (10 widened + 2 unchanged narrow) — all PASS, no `/tmp` workarounds.

## Key Decisions
1. **Widened context only where required.** The 10 services with a `replace ... => ../shared`
   directive AND a Dockerfile were switched to `context: ./src`,
   `dockerfile: <svc>/Dockerfile`: matching/clearing/margin/settlement engines,
   gateway, compliance-service, market-data-service, warehouse-service,
   securities-service, fix-gateway. `auth-service` and `platform-service` have
   no external/shared deps, so they keep their narrow `./src/<svc>` context
   (per task allowance — zero-dep services need no widening). `corporate-actions`
   consumes `decimal` but is a library with **no Dockerfile** (R017), so nothing
   to change there.
2. **Dockerfile pattern:** `WORKDIR /build` → `COPY shared ./shared` →
   `COPY <svc> ./<svc>` → `WORKDIR /build/<svc>` → build to `/build/bin/<svc>`.
   From `/build/<svc>`, the go.mod's `../shared/pkg/types/decimal` and
   `../shared/pkg/tenant` resolve to `/build/shared/pkg/...` ✓. Output goes to
   `/build/bin/<svc>` (not `/build/<svc>`, which is now a directory).
3. **Kept the existing build constraints:** `CGO_ENABLED=0 GOOS=linux` static
   builds, non-root `appuser` runtime on `alpine:3.19`, original EXPOSE ports
   per service preserved. Added `ENV GOTOOLCHAIN=auto` to the builder stage so
   the `go 1.25.x` module directive can fetch the toolchain if the base image's
   patch release lags (matches the documented R-series build behavior).
4. **`src/.dockerignore` added.** Because the wide context is now the whole
   `src/` tree, this excludes frontend artifacts (`node_modules`, `dist`,
   `coverage`, `*.tsbuildinfo`, etc.) so the context stays lean. It only applies
   to the `context: ./src` (wide) builds; narrow-context services are unaffected.
   (Observed context transfer for gateway: ~1 MB.)
5. **CI `docker-build.yml` made context-aware.** Its single hardcoded
   `context: src/${{ matrix.service }}` could not build the widened services.
   The `detect-services` step now emits one JSON object per service
   `{service, context, dockerfile}` — wide `context: src` for services whose
   `go.mod` has a `replace ... => ../shared` directive, narrow `context: src/<svc>`
   otherwise (so frontend/narrow services are untouched). The matrix switched
   from `service:` to `include:`, and the build step uses
   `context: ${{ matrix.context }}` + `file: ${{ matrix.dockerfile }}` (root-relative
   dockerfile, which is how `docker/build-push-action` resolves an explicit `file:`).
   This workflow remains `workflow_dispatch`-only / guarded (a deploy step).

## Verification (real builds, Docker 29.3.1)
`docker compose config` parses OK. `docker compose build <svc>` PASS for all 12
Go services from a clean checkout:
- Widened (10): gateway (decimal+tenant), matching-engine, clearing-engine,
  margin-engine, settlement-engine, compliance-service, market-data-service,
  warehouse-service, securities-service, fix-gateway.
- Narrow, unchanged (2): auth-service, platform-service.
The gateway build exercised the hardest case (both `../shared/pkg/types/decimal`
and `../shared/pkg/tenant` replace targets) and compiled, confirming the relative
replace layout is preserved inside the context. Test images were pruned after.

## Blockers Found
None. Docker and network were available in the worker env, so this was verified
by actual image builds rather than the static-only fallback the task allowed for.
