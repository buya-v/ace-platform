## Key Decisions
1. **`internal/eventbus` package per engine, not inline in main.go.** The
   decode â†’ engine-call â†’ publish bridge lives in a testable package so it can be
   exercised against the in-process channel `Producer`/`Consumer` without a
   broker. This also keeps the `kafka` package free of a dependency on
   `internal/engine`/`internal/types` (eventbus is the composition root that
   depends on both).
2. **Activation gated on `eventbus.Enabled()` (= `KAFKA_BROKERS` set).** When
   brokers are configured the real adapter is wired; when unset (unit/dev) Kafka
   wiring is skipped entirely and the engine still serves gRPC. This does NOT
   regress R007: `NewProducerFromEnv`/`NewConsumerFromEnv` still `log.Fatal` if
   ever called without brokers outside tests â€” eventbus simply does not call them
   when `Enabled()` is false. Added `kafka.Enabled()` (wraps the existing
   `kafkaBrokersConfigured`) to each engine's kafka package.
3. **Single-handler composition done in main.go for margin & settlement.** The
   engines support exactly one `MarginCallHandler`/`CycleHandler`
   (last-write-wins). Rather than the Runtime claiming the handler exclusively
   (which would clobber the existing persistence/logging handler), the Runtime
   exposes `PublishMarginCall`/`PublishCycle` and main.go composes
   persist+log+publish. Clearing/matching have no such conflict (clearing
   publishes directly from its consumer handler return path; matching composes
   inside the existing trade handler).
4. **Snapshot-after-unlock preserved (R008).** All publish calls happen from
   handlers the engines already fire *outside* their locks, so no new
   lock-ordering risk. `go test -race` passes on all four engines'
   eventbus+engine+kafka packages.
5. **Idempotency + DLQ + retry are inherited, not reimplemented.** The real
   `KafkaConsumer` already tracks `processedIDs` (idempotent consume), retries per
   `ConsumerConfig`, and routes to a DLQ topic via the producer â€” preserved
   unchanged. Bridges return `nil` (drop) for permanently-bad records
   (un-parseable price) and `err` (retry) for transient engine/publish failures.
6. **Deterministic downstream keys for idempotency.** Settlement uses
   `cycle-<tradeID>` as the cycle ID so reprocessing the same novation is
   naturally idempotent.
7. **Integration test makes clearing.novated + settlement.completed REQUIRED,
   margin.call-issued BEST-EFFORT.** A novation and a settlement cycle always
   fire; a margin call only fires when the running margin-engine's params/
   collateral make the position deficient (deployment-dependent). Requiring the
   margin event would conflate "no call issued" with "event did not propagate."
