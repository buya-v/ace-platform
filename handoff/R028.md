## Key Decisions

1. **D1 root cause was an unset field, not a missing transport.** `orderbook`
   already copies `order.ParticipantID` onto the trade, and `eventbus.PublishTrade`
   already forwards it â€” but `server.SubmitOrder` built `types.Order` with only
   `AccountID` set, leaving `ParticipantID` empty, so every matched trade carried
   empty `Buyer/SellerParticipantID` and clearing novation rejected it
   (`novation: both buyer and seller participant IDs are required`). Fixed by
   populating `order.ParticipantID` in `SubmitOrder`. The novation guard was **not**
   weakened.

2. **Participant identity is resolved from the gateway-set `X-Participant-Id`
   header.** The gateway forwards the JWT participant claim as the
   `x-participant-id` metadata â†’ `X-Participant-Id` header (see gateway
   `handler.SubmitOrder` + proxy httpclient). `handleSubmitOrder` now reads it into
   a `participantID`, threads it through `SubmitOrderRequest.ParticipantID`, and
   `SubmitOrder` falls back `ParticipantID â†’ AccountID` when no distinct participant
   is supplied (in the current ace-commodities flow the participant *is* the
   account). Added an optional `participant_id` JSON body field for direct callers.

3. **Guaranteed order IDs.** `orderbook.SubmitOrder` now assigns
   `order.OrderID = ob.idGen.NewID()` when the caller omits it, so the trade's
   `Buy/SellOrderID` are never blank and resting orders remain cancellable. The
   auction path inherits this because auction fills operate on orders already
   placed on the book via `OrderBook.SubmitOrder`.

4. **D2 seeding lives in a dedicated `internal/seed` package, not in business
   logic.** SPAN parameters are config/seed data; the engine calc operates on
   whatever `params.Store` holds. `seed.FromEnv()` loads a JSON file when
   `MARGIN_RISK_PARAMS_FILE` is set (production / custom instruments, fail-fast on a
   bad file), otherwise applies a built-in demo default for `WHT-HRW-2026M07-UB`.
   This mirrors the existing collateral-source convention (in-memory default + env
   for the real source). Demo values reuse the proven-safe wheat numbers from the
   params unit tests (PriceScanRange 300, VolScanRange 50, SpotPrice 450,
   ContractSize 5000) so margin yields a positive, non-overflowing requirement.
   `cmd/margin-engine/main.go` applies the seed right after `params.NewStore()`.

5. **D3 topics added to the existing bootstrap, not a new mechanism.**
   `infrastructure/docker/kafka/create-topics.sh` (the only enumerated topic list â€”
   the MSK terraform module sets `auto.create.topics.enable=false` but does not
   enumerate topics) now creates the four `ace-commodities.*` cross-service
   application topics and their `ace-commodities.dlq.*` counterparts. DLQ names
   match `topicWithoutPrefix()` (`<tenant>.dlq.<topic-without-prefix>`,
   e.g. `ace-commodities.dlq.trades.executed`) referenced by the matching-engine
   kafka consumer. DLQ topics get 90-day retention so failures survive for
   investigation; all `create_topic` calls are `--if-not-exists` (idempotent).

## How a real trade now carries participant IDs end-to-end

1. Gateway `SubmitOrder` reads `claims.ParticipantID` from the JWT and forwards it
   as metadata `x-participant-id` â†’ HTTP header `X-Participant-Id`.
2. matching `handleSubmitOrder` resolves `participantID` (header, falling back to
   body `account_id`) and passes it on `SubmitOrderRequest`.
3. matching `SubmitOrder` sets `order.ParticipantID`; `orderbook.SubmitOrder`
   assigns an `OrderID` if missing.
4. On a cross, `orderbook`/`auction` copy `incoming/resting.ParticipantID` and
   `OrderID` onto `types.Trade` (`Buyer/SellerParticipantID`, `Buy/SellOrderID`).
5. `eventbus.PublishTrade` emits `ace-commodities.trades.executed` with the
   populated IDs.
6. clearing `onTradeExecuted` maps `p.Buyer/SellerParticipantID` onto the trade and
   `Novate()` succeeds (no empty-ID rejection) â†’ `ace-commodities.clearing.novated`.
7. margin consumes `clearing.novated`; with seeded params it computes margin â†’
   `ace-commodities.margin.call-issued` (best-effort). settlement consumes
   `clearing.novated` â†’ `ace-commodities.settlement.completed`.
