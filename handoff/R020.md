# Handoff — R020: gateway fees + pre-trade risk → decimal

**Status:** DONE
**Agent:** Coder
**Phase:** R0-extension (money-float follow-ups from R006 audit)

---

## Summary
Converted the gateway's trading-fee calculation and pre-trade risk
order-value/quantity/price-band checks from `float64` to the shared fixed-point
`github.com/garudax-platform/decimal` type, closing two of the user-visible
money-path residuals R006 flagged (fees charged, orders blocked). Wired the
shared decimal module into `src/gateway` via the validated relative `replace`
directive. Gateway reporting/observability/middleware/tickets/bot floats
(metrics, SLA, rate-limit, percentages) were deliberately left as `float64` per
scope.

## Key Decisions
1. **Domain/computation = decimal; transport DTOs stay float64.** `FeeRule`
   (RateBPS/MinFee/MaxFee/PerContractFee), `FeeResult`, `FeeTransaction.Amount`,
   `OrderRequest` (Price/Quantity), `OrderLimits` (MaxOrderQty/MaxOrderValue) and
   `PositionLimits` (MaxLong/MaxShort/MaxGross) now carry `decimal.Decimal`. The
   pure JSON-request DTOs `CreateFeeRuleRequest` and `FeeRuleUpdate` keep
   `float64`/`*float64` and are converted at the handler boundary. This keeps the
   HTTP wire contract identical (clients still send/receive JSON numbers — the
   decimal type's Marshal/UnmarshalJSON emit/accept plain numbers) while the
   money math is exact.
2. **DB scan/exec is the float boundary.** `database/sql` cannot scan into or
   bind `decimal.Decimal` (it implements no `sql.Scanner`/`driver.Valuer`), so
   the PgStore paths scan numeric columns into `float64` temporaries and convert
   with a local `decFromFloat` helper, and bind via `.Float64()`. This matches
   how securities-service treats non-raw numeric columns and keeps the
   zero-network build. (These DB paths are currently dormant — the gateway runs
   with a nil DB / in-memory store — but remain correct and compile-clean.)
3. **Multiply-before-divide for the rate fee.** `rateFee =
   tradeValue.MulDecimal(RateBPS).DivInt64(10000)` rather than dividing the rate
   by 10000 first, so a fractional basis-point rate (e.g. 2.5 bps) is not lost to
   the 4-dp scale. Added `TestCalculateFee_FractionalBasisPoints` proving
   2.5 bps on 1,000,000 = 250 exactly.
4. **Price-band check avoids decimal/decimal division.** The shared type exposes
   no Decimal/Decimal divide, so the band decision is rearranged to the exact
   fixed-point comparison `|price - last| * 100 > bandPct * last` (valid since
   `last > 0`). `PriceBandPct` stays `float64` (it is a percentage, not money,
   and was not in the audit's enumerated scope); it is lifted to decimal only for
   that one multiply. The percentage shown in the error message is the sole
   remaining float and is display-only.
5. **Untrusted-input multiplies use `Try*` and fail closed.** Order
   price/quantity come from clients, so `CheckOrderSize` notional and
   `CheckPriceBand` use `TryMulDecimal`/`TryMulInt64`; an overflow (only
   reachable at astronomically large inputs) is treated as
   ORDER_VALUE_EXCEEDED / PRICE_BAND_EXCEEDED rather than panicking. The fee
   calculator (dormant library code, bounded internal inputs) uses the
   convenience panic-on-overflow methods for readability.
6. **`DefaultOrderLimits` sentinel.** `math.MaxFloat64` is not representable in
   `Decimal` (range ~±9.22e14); replaced with `decimal.DecimalFromRaw(math.MaxInt64)`
   (~9.22e14) as the effectively-unlimited value.

## Suggested Follow-ups
- **Volume30D / PriceBandPct left float64 intentionally.** `ParticipantTier.Volume30D`
  is a 30-day traded-volume statistic (never used in money arithmetic here) and
  `PriceBandPct` is a percentage; both are outside the R006-enumerated money
  fields. If a later task wants full money-typing of the fees domain, convert
  `Volume30D` as well.
- **R007–R011 remain** the other R006 residuals (corporate-actions,
  compliance MCSD/FRC, fix-gateway price tags, gateway tenant enforcement).
- If/when the gateway gets a real Postgres driver, consider giving the shared
  `decimal` type `sql.Scanner`/`driver.Valuer` so the `decFromFloat` DB shims can
  be dropped across all consumers.
