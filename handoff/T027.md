**Status:** DONE
**Agent:** Coder

## Summary

Built the clearing engine for trade novation, position management, and multilateral netting. The engine consumes trades from the matching engine (T008), novates them into CCP-intermediated clearing obligations (buyer↔CCP, seller↔CCP), updates net positions with VWAP and realized P&L tracking, and supports multilateral netting to reduce gross obligations to net settlement amounts. All 37 tests pass including concurrency tests.

## Key Decisions

1. **Separate Go module (`clearing-engine`)** — follows the matching-engine pattern with its own `go.mod`, zero external dependencies, and identical Decimal type for cross-service compatibility.
2. **Trade novation as core primitive** — every bilateral trade is split into exactly two CCP-intermediated obligations. The CCP constant (`ACE-CCP`) is defined but the CCP's own position is not tracked (that's a T028/T029 concern for margin and settlement).
3. **VWAP position tracking with realized P&L** — positions track volume-weighted average entry price. When closing/reducing positions, realized P&L is calculated. Position flips (long→short) correctly close the old position and open a new one at the trade price.
4. **Idempotent trade processing** — the engine tracks processed trade IDs and rejects duplicates, critical for at-least-once delivery from Kafka.
5. **Append-only obligation store** — follows the same pattern as T008's TradeStore interface. In-memory implementation for dev/testing; production will use PostgreSQL.
6. **Mutex-based concurrency** — single mutex on the engine (same pattern as matching-engine) ensures correctness. Position manager has its own RWMutex for read-heavy queries.
7. **Server on port 50052/8082** — avoids collision with matching-engine (50051/8081). Health endpoints follow the same `/healthz`/`/readyz` pattern.

## Suggested Follow-ups

- **T028 (Margin/Risk)** should consume positions from `position.Manager.Get()` and obligations from `store.ObligationStore` to calculate margin requirements. The `Position.NetQuantity` and `Position.AvgEntryPrice` fields are the key inputs.
- **T029 (Settlement)** should consume netting results from `netting.Service.Net()` to generate settlement instructions. The `NettingResult.NetQuantity` and `NettingResult.NetValue` represent final settlement obligations.
- **Kafka integration** — wire the engine to consume from the `exchange.trades.{instrument_id}` Kafka topic (mentioned in T008 handoff). The `Engine.ClearTrade()` method is the entry point.
- **gRPC API** — define clearing-specific proto (ClearTrade, GetPosition, GetNettingResults RPCs) and wire to the server layer, same pattern as matching-engine.
- **CCP position tracking** — the CCP itself should maintain a flat book (sum of all participant positions = 0). Add CCP position validation as an invariant check.
- **Obligation status transitions** — currently obligations are created as NOVATED; add status machine for NOVATED→NETTED→SETTLED flow.
