# T801 Handoff — Guided prompts for incomplete CRUD commands + intent disambiguation

## Summary

Added a guided prompt layer to `src/gateway/internal/bot/executor.go` that intercepts incomplete CRUD commands (e.g., "create new instrument", "create instrument" without parameters) and returns a helpful format guide instead of falling through to the keyword-based list handlers that would incorrectly return a list.

## Changes Made

### `src/gateway/internal/bot/executor.go`

1. **Guided prompts block** — inserted AFTER all regex CRUD handlers and BEFORE the keyword list handlers (between the batch screen handler and the system health handler at line ~716). The block fires when the message contains a create/new/add/register verb but no complete parameter set (since the regex handlers already consumed fully-formed commands above).

   Covers 7 incomplete-command patterns:
   - `create/new instrument` → format guide with example
   - `create/new commodity` → format guide with example
   - `create/add fee/schedule/rule` → fee management command menu
   - `create/add ticket/bug/issue/report` → ticket creation format guide
   - `add/new receipt` → issue receipt format guide
   - `register/new facility/warehouse` → register facility format guide
   - `new/add order` → buy/sell order format guide

2. **Guard on keyword list handler** — changed:
   ```go
   if containsAny(lower, "instrument", "commodity", "contract") {
   ```
   to:
   ```go
   if containsAny(lower, "instrument", "commodity", "contract") && !containsAny(lower, "create", "new", "add") {
   ```
   This prevents "create new instrument" from matching the list handler after passing through the guided prompt block.

3. **Improved default reply** — replaced the generic "I can help with system health, alerts, margin status, and tickets. What would you like to know?" with a more actionable message listing 6 common commands and pointing to `help` for the full list.

### `src/gateway/internal/bot/handler_test.go`

- Added `"strings"` import
- Updated `TestChat_FallbackMode_DefaultKeyword` to use `strings.Contains(reply, "I didn't understand that")` instead of an exact match on the old default reply text

## Placement Logic

The correct order in `Execute()` is:
1. Profile / whoami handler
2. Regex CRUD handlers (buy/sell orders, halt, resume, approve/reject KYC, create commodity, create instrument, update instrument, create fee schedule, set tier, add fee rule, issue receipt, pledge receipt, screen participant, batch screen)
3. **[NEW] Guided prompts block** — catches incomplete create/new/add commands
4. Keyword list handlers (system health, margin, settlement, alerts, participants, instruments, warehouse, tickets, fees, positions, help)
5. Default fallback

## Verification

- `go build ./...` — clean build, no errors
- `go test ./internal/bot/` — all tests pass (was 1 failure in `TestChat_FallbackMode_DefaultKeyword` due to changed reply text; fixed by updating assertion to `strings.Contains`)

## Decisions Made

- Placement after batch screen (not after update instrument) ensures that the guided prompt cannot intercept fully-formed commands like "create fee schedule Standard 2027" or "add fee rule trading farmer 10bps" which are handled by their own regex handlers higher in the chain.
- The instrument/commodity/contract keyword list guard uses `!containsAny(lower, "create", "new", "add")` (without "register") because "register" does not appear in those keyword list paths.
- The default reply uses emojis inline (per task spec) to make the suggestions more scannable.
