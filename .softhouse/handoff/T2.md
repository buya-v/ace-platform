## Summary

## Part A — Announcement handlers (`handlers_announcement.go`)

- `handleCreateAnnouncement`: validates title+body required; audience defaults to `PUBLIC`; returns 201
- Route: `GET|POST /api/v1/securities/announcements`

## Part B — Audit trail handler (`handlers_audit.go`)

- Route: `GET /api/v1/securities/audit-trail`

## Part C — Force-logout / Suspend / Reinstate (in `handlers_participant.go`)

- Added `actorFromRequest()` helper (reads `X-Actor-ID` header, defaults to `"system"`)
- Updated `handleParticipant` dispatcher to detect `/force-logout`, `/suspend`, `/reinstate` suffixes
- `forceLogoutOrSuspend()` shared logic:
  1. Get participant (404 if missing)
  2. `UpdateStatus → PARTICIPANT_SUSPENDED`
  3. List+cancel all PENDING orders for the participant
  4. Log `AuditEntry{EntityType:"PARTICIPANT", Action:"SUSPEND"}` (best-effort)
  5. Return `{participant_id, cancelled_orders, status}`
- `handleForceLogout` and `handleSuspendParticipant` both delegate to `forceLogoutOrSuspend`
- `handleReinstateParticipant`: sets `PARTICIPANT_ACTIVE`, logs `{Action:"REINSTATE"}`, returns updated participant

## Part D — Audit logging (best-effort, non-failing)

- `handlers_instrument.go` `handleUpdateInstrumentStatus`: logs `{EntityType:"INSTRUMENT", Action:"UPDATE", Detail: newStatus}`

## Decisions

- `announcementStore`/`auditStore` inserted before `dayManager` in `New()` to keep all stores grouped before engine args
- Audit calls are nil-guarded and errors are discarded — operations never fail due to audit failure
- `AuditStore.List()` uses lexicographic string comparison on RFC3339 timestamps (correct: RFC3339 sorts lexicographically)
- `forceLogoutOrSuspend()` shared to avoid duplication — semantics of force-logout and suspend are identical
- `extractParticipantID()` added to parse `participants/{id}/sub-resource` paths
