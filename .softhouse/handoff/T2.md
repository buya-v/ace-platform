## Deliverables

### `infrastructure/db/migrations/V26__securities_instruments.sql`

Creates the `securities` schema and all instrument reference data tables:

- **`securities.instruments`** — Master instrument table (equities, bonds, ETFs). 35 columns covering identifiers (ISIN/CUSIP/SEDOL/ticker/MIC), asset classification, issuer info, equity-specific fields (shares_outstanding, market_cap), bond-specific fields (par_value, coupon_rate, coupon_frequency, day_count_convention, maturity_date, issue_date, next_coupon_date), ETF-specific fields (nav_per_share, fund_manager), and trading controls (price_band_pct, max_order_qty, max_order_value, short_sell_allowed, margin_eligible). 7 indexes.
- **`securities.short_sell_restricted_list`** — SSR restricted list entries. Composite PK on (instrument_id, restricted_from).
- **`securities.locates`** — Short-sell locate records. 3 indexes on participant_id, instrument_id, status.
- **`securities.position_limits`** — Per-participant per-instrument position limits. Composite PK.
- **`securities.large_trader_thresholds`** — Large trader reporting thresholds. PK on instrument_id.
- **`securities.ssr_triggers`** — SSR trigger tracking. Composite PK on (instrument_id, trigger_date).

GRANT: `garudax_exchange_svc` gets full DML on all tables in schema; `garudax_clearing_svc` gets SELECT on instruments and position_limits.

### `infrastructure/db/migrations/V27__securities_trading.sql`

Depends on V26 (foreign keys to `securities.instruments`). Creates:

- **`securities.orders`** — Securities order book. Mirrors exchange.orders for commodities. Includes is_short_sell, locate_id, settlement_date. 6 indexes.
- **`securities.trades`** — Append-only trade ledger. Includes accrued_interest and settlement_value for bond settlement. Protected with `CREATE RULE no_update/no_delete`. 5 indexes.
- **`securities.execution_reports`** — FIX-style execution report log. Append-only (same rule pattern). 3 indexes.
- **`securities.positions`** — Net position per participant per instrument. Includes short_qty tracking. Composite PK.

GRANT: `garudax_exchange_svc` gets INSERT on orders/trades/execution_reports, UPDATE on specific order columns (filled_qty, remaining_qty, status, updated_at); `garudax_clearing_svc` gets full UPDATE on positions.

### `infrastructure/db/migrations/V28__securities_settlement.sql`

Depends on V26 and V27. Creates:

- **`securities.settlement_obligations`** — T+2 settlement obligation per trade side. 7-state status machine (PENDING → AFFIRMED → NETTED → INSTRUCTED → SETTLING → SETTLED | FAILED). Tracks affirm/net/instruct/settle/fail timestamps individually. 6 indexes.
- **`securities.netting_results`** — Bilateral netting output per run per participant per instrument per settlement date. 3 indexes.
- **`securities.csd_accounts`** — CSD custody accounts (PROPRIETARY, CLIENT_SEGREGATED, COLLATERAL). 2 indexes.
- **`securities.csd_balances`** — Holdings per account per instrument. Composite PK. Enforced balance integrity via CHECK constraints: `available_qty >= 0`, `total_qty >= 0`, and `available_qty = total_qty - pledged_qty - pending_out_qty`.
- **`securities.csd_transfers`** — FoP and DvP transfer instructions. Partial index on settlement_obligation_id. 5 indexes.
- **`securities.settlement_penalties`** — CSDR-style fail penalties (ACCRUED → INVOICED → PAID | WAIVED). 3 indexes.
- **`securities.buy_ins`** — Buy-in proceedings after T+4 fail. 5-state machine. 3 indexes.
- **`securities.corporate_actions`** — Corporate action master (8 action types). Polymorphic fields for dividend/split/rights/merger variants. 5 indexes.
- **`securities.corporate_action_entitlements`** — Per-participant entitlements per corporate action. 3 indexes.

GRANT: `garudax_clearing_svc` gets full DML on settlement/netting/CSD/penalty/buy-in/entitlement tables; SELECT only on corporate_actions and csd_accounts.

# T2 Handoff — W3C Traceparent Header Propagation (prior content preserved below)



## Summary

Implemented W3C Trace Context (traceparent) propagation across the GarudaX platform with zero external dependencies. All 31 new shared tests pass and all existing gateway tests continue to pass.

## Deliverables

### New files

**`src/shared/internal/observability/tracing.go`**
- `TraceContext` struct: `TraceID` (32 hex chars), `SpanID` (16 hex chars), `ParentSpanID`, `Sampled bool`
- `ParseTraceparent(header string) (TraceContext, error)` — strict W3C validation (version check, lengths, lowercase-hex-only, no-all-zero IDs, reserved `ff` version rejected)
- `FormatTraceparent(tc TraceContext) string` — produces `00-<trace_id>-<span_id>-<flags>`
- `NewTraceContext()` — generates root span (new trace_id + span_id, Sampled=true)
- `NewSpan(tc TraceContext)` — creates child span (same trace_id, new span_id, ParentSpanID set to incoming span_id)
- `TraceFromContext(ctx) TraceContext` — extracts TraceContext from context
- `InjectTraceparent(req *http.Request, tc TraceContext)` — sets traceparent header on outgoing requests
- `TraceMiddleware(logger *slog.Logger) func(http.Handler) http.Handler` — HTTP middleware with full W3C propagation; also stores trace_id via `WithTraceID` for `LoggerFromContext` compatibility

**`src/shared/internal/observability/tracing_test.go`**
- 31 tests covering: parse valid/invalid headers (empty, wrong field count, reserved version, non-hex, all-zero IDs, wrong lengths), ID generation (length, hex format, uniqueness for 100 samples), NewTraceContext/NewSpan semantics, FormatTraceparent round-trip, context store/retrieve, InjectTraceparent no-op on empty, TraceMiddleware (no header, valid header, invalid header fallback, sampled flag propagation, trace_id in context)

### Modified files

**`src/gateway/internal/observability/middleware.go`**
- Updated the existing `TracingMiddleware` to add W3C traceparent support alongside legacy `X-Request-ID`/`X-Trace-ID` headers
- Priority order for trace_id: (1) incoming `traceparent` header, (2) incoming `X-Request-ID`, (3) newly generated
- Sets `traceparent` response header (W3C format) and `X-Request-ID`/`X-Trace-ID` (backward compat)
- Logs `trace_id`, `span_id`, `parent_span_id` on every request_started and request_completed log entry
- Added `parseTraceparent()`, `generateSpanID()`, `isTraceparentHex()`, `isAllZeroString()` helpers (no external deps)
- All 5 pre-existing gateway observability tests continue to pass

## Decisions Made

1. **ParseTraceparent semantic model**: The incoming `parent-id` field is stored as `tc.SpanID` (not `tc.ParentSpanID`). This is because the incoming `parent-id` represents the remote caller's span ID — it is the span we descend from. `NewSpan(tc)` then sets `child.ParentSpanID = tc.SpanID`. This avoids an extra assignment and keeps the semantic chain clear: ParseTraceparent gives you the remote span, NewSpan creates your local span descending from it.

2. **Gateway self-contained**: The gateway keeps its own `internal/observability` package and does not import the shared module. The W3C parsing logic is ~40 lines. This preserves the zero-dep pattern — adding a module dependency would require `go.mod` changes.

3. **Backward compatibility preserved**: The gateway's `TracingMiddleware` still emits `X-Request-ID` and `X-Trace-ID` response headers and still reads `X-Request-ID` from incoming requests. All existing tests pass without modification.

4. **Middleware position unchanged**: `TracingMiddleware` was already wired in `main.go` at the correct position (after RequestID, before Auth). No changes to `main.go` were needed.
