## Handlers Implemented

### `handleListInstruments` — GET /api/v1/securities/instruments

- Reads `asset_class`, `trading_status`, `exchange_code` query params and passes them as `store.InstrumentFilters` to `instrumentStore.List()`.
- Applies **in-memory search** on the returned slice: case-insensitive substring match of `search` param against both `ticker` and `name` fields.
- Applies **limit** (default 50) and **offset** (default 0) pagination on the filtered results.
- Returns `{"data": [...], "total": <unfiltered_count>, "limit": N, "offset": N}`.
- Returns an empty JSON array (never `null`) when no results match.

### `handleGetInstrument` — GET /api/v1/securities/instruments/{id}

- Extracts ID using `extractLastSegment()` (last non-empty path segment).
- Returns 404 with `ErrorResponse` when `store.ErrNotFound`.
- Skips the request if the last segment is `"status"` (that path is handled by `handleUpdateInstrumentStatus`).

### `handleCreateInstrument` — POST /api/v1/securities/instruments

- Validates: `ticker`, `name`, `asset_class` required; `lot_size > 0`; `tick_size > 0`.
- Generates a **UUID v4** via `newUUID()` which calls `crypto/rand.Read` and sets the version 4 / variant bits per RFC 4122.
- Sets `CreatedAt` and `UpdatedAt` to `time.Now().UTC().Format(time.RFC3339)`.
- Defaults `TradingStatus` to `ACTIVE` if not provided by the caller.
- Returns 201 with the created instrument; 409 on duplicate ID.

### `handleUpdateInstrument` — PATCH /api/v1/securities/instruments/{id}

- Decodes JSON body into `store.InstrumentUpdate` (zero-value = no change semantics).
- Calls `instrumentStore.Update(id, partial)`.
- Fetches the updated record and returns 200; returns 404 if not found.

### `handleUpdateInstrumentStatus` — PUT /api/v1/securities/instruments/{id}/status

- Detects the `/status` sub-resource in `handleInstrument`'s dispatcher via `strings.HasSuffix`.
- Extracts `{id}` as the second-to-last path segment.
- Validates `status` field against `validTradingStatuses` map (ACTIVE, HALTED, SUSPENDED, DELISTED).
- Fetches the instrument first to capture `previous_status`.
- Calls `instrumentStore.UpdateStatus(id, req.Status)`.
- Returns 200 with `instrumentStatusResponse{instrument_id, previous_status, current_status, reason, changed_at}`.

## Route Wiring

`server.go` routes are unchanged in their URL patterns:

```
GET  /api/v1/securities/instruments        → handleInstruments → handleListInstruments
POST /api/v1/securities/instruments        → handleInstruments → handleCreateInstrument
GET  /api/v1/securities/instruments/{id}   → handleInstrument  → handleGetInstrument
PATCH /api/v1/securities/instruments/{id}  → handleInstrument  → handleUpdateInstrument
PUT  /api/v1/securities/instruments/{id}/status → handleInstrument → handleUpdateInstrumentStatus
```

The `/status` sub-route is detected inside `handleInstrument` using `strings.HasSuffix` before the method switch, so the existing `"/api/v1/securities/instruments/"` catch-all registration handles both `/{id}` and `/{id}/status`.

## Architecture Decisions

1. **Separate file, no new imports in server.go beyond `strings`.** All instrument handler logic lives in `handlers_instrument.go`. The `server.go` dispatcher only routes — it does not contain business logic. This separation keeps `server.go` readable even as more resource handlers are added.

2. **`newUUID()` uses `crypto/rand`.** The T1 handoff noted UUID generation as a suggested follow-up. Implemented as a pure stdlib function with no external dependencies, consistent with the zero-dep pattern for all other services.

3. **In-memory search applies after store filtering.** The store's `List` method filters by `asset_class`, `trading_status`, and `exchange_code`. The text search (`search` param) is applied in-memory on the returned slice. This keeps the store interface clean and avoids pushing text-search logic into the store layer.

4. **`extractLastSegment` is an exported-internal helper.** It is used by both `handleGetInstrument` and `handleUpdateInstrument` to extract the `{id}` from the path, replacing the brittle index-based slice approach (`r.URL.Path[len("/api/v1/.../"):` ]).

5. **`/status` detection in dispatcher, not a separate route registration.** The stdlib `ServeMux` does not support parameterized routes. Adding `"/api/v1/securities/instruments/{id}/status"` as a separate pattern would cause conflicts with the `"instruments/"` catch-all. Detecting the `/status` suffix in the dispatcher is the correct pattern for stdlib routing.

## Suggested Follow-ups

1. **Handler tests** — Add `internal/server/handlers_instrument_test.go` using `net/http/httptest`. Cover: list with each filter combo, list with search, list with limit/offset, create validation errors, create success, get 404, get success, patch not-found, patch success, status invalid enum, status 404, status success.

2. **`UpdatedAt` on PATCH** — The current `handleUpdateInstrument` does not update `UpdatedAt` on the stored record because `InstrumentUpdate` has no timestamp field and `InMemoryInstrumentStore.Update` does not modify `UpdatedAt`. A follow-up should either: (a) add `UpdatedAt string` to `InstrumentUpdate`, or (b) have the store implementation always set `inst.UpdatedAt = time.Now()` on any successful update.

3. **Pagination cursor** — The OpenAPI spec references a `cursor` parameter and `next_cursor` in the response. The current implementation uses offset-based pagination. If cursor-based pagination is required, add a `next_cursor` field to the response derived from the last returned item's ID (base64-encoded).

4. **ISIN uniqueness check** — The OpenAPI spec says ISIN must be unique. The in-memory store only checks ID uniqueness on `Create`. A follow-up should add an ISIN index to `InMemoryInstrumentStore` and return 409 when a duplicate ISIN is detected.

5. **`UpdatedAt` on status change** — `handleUpdateInstrumentStatus` does not update `UpdatedAt` on the instrument. This is a minor gap — status change is a meaningful mutation and should be reflected in the timestamp.
