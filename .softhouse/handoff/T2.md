# T2 — T+2 Settlement Engine

## Summary

Implemented the T+2 settlement engine for the securities-service, covering the full lifecycle from trade execution to settlement obligation creation and cycle processing (PENDING -> AFFIRMED -> NETTED -> SETTLED).

## Changes

### Part A — Types (`internal/types/types.go`)
- Added `SettlementStatus` type with 7 constants: `SETTLE_PENDING`, `SETTLE_AFFIRMED`, `SETTLE_NETTED`, `SETTLE_INSTRUCTED`, `SETTLE_SETTLING`, `SETTLE_SETTLED`, `SETTLE_FAILED`
- Added `SettlementObligation` struct with all required fields and JSON tags
- Added `SettlementResult` struct for cycle summaries

### Part B — Store (`internal/store/store.go`)
- Added `SettlementStore` interface with 5 methods: `Create`, `Get`, `ListByDate`, `ListByStatus`, `UpdateStatus`
- Added `InMemorySettlementStore` implementation using `sync.RWMutex` + `map[string]*types.SettlementObligation`

### Part C — Settlement Engine (`internal/settlement/engine.go`) — NEW FILE
- `SettlementEngine` struct with `orderStore` and `settlementStore` dependencies
- `CreateObligationsFromTrades`: looks up buy/sell orders to resolve participant IDs, creates SETTLE_PENDING obligations with `NetAmount = Price * Quantity`
- `ProcessSettlementCycle`: runs the full cycle for a date — affirm PENDING, net AFFIRMED by (instrument, buyer, seller), settle NETTED
- 6 tests covering: basic obligation creation, invalid order lookup, full cycle, multiple trades with same parties (netting), empty date, idempotent re-run

### Part D — Matching Engine Integration (`internal/engine/engine.go`)
- Added `settlementEngine *settlement.SettlementEngine` field to `MatchingEngine`
- Added `settlementEngine` parameter to `NewMatchingEngine` (nil-safe)
- After trade creation in `MatchOrder`, calls `settlementEngine.CreateObligationsFromTrades(trades)` if engine is not nil (non-fatal on error)

### Part E — HTTP Endpoints (`internal/server/`)
- `handlers_settlement.go` — NEW FILE with two handlers:
  - `GET /api/v1/securities/settlements?date=YYYY-MM-DD` or `?status=SETTLE_*` — list obligations
  - `POST /api/v1/securities/settlements/cycle` with body `{"date": "YYYY-MM-DD"}` — trigger settlement cycle
- `server.go` — added `settlementStore` and `settlementEngine` fields to `Server` struct, updated `New()` constructor, registered settlement routes

### Part F — main.go
- Creates `InMemorySettlementStore` and `SettlementEngine`
- Passes both to `MatchingEngine` and `Server`

## Decisions
- Settlement engine uses `orderStore.Get()` to resolve participant IDs from trade's buy/sell order IDs (no schema change to SecurityTrade)
- Settlement obligation creation from matching engine is non-fatal (logged but doesn't fail matching)
- Netting groups by (instrument, buyer, seller) — obligations in same group advance to NETTED together
- Simplified settlement: NETTED -> SETTLED without real CSD interaction (as specified)

- 6 new settlement tests pass
- Build: `go build ./...` clean
