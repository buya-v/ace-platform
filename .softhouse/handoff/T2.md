## Decisions Made

### Column mapping divergences (DB schema vs types.go)
- `securities.instruments` PK is `instrument_id` (not `id`). All WHERE/INSERT/UPDATE use `instrument_id`.
- `securities.instruments.shares_outstanding` maps to `Instrument.OutstandingShares`.
- `securities.orders.tif` maps to `SecurityOrder.TimeInForce`.
- `securities.orders.filled_qty` maps to `SecurityOrder.FilledQuantity`.
- `securities.orders` has no `avg_fill_price` column — `AvgFillPrice` is always read as `0.0` from the DB store.
- `securities.orders.status` uses `NEW` where the service domain uses `PENDING`. Translation functions `dbStatusToPending` / `pendingToDBStatus` handle the mapping transparently.

### Submit — account_id required by DB schema
`securities.orders.account_id NOT NULL` has no equivalent in `SecurityOrder`. We use `participant_id` for both `participant_id` and `account_id` columns on INSERT as a pragmatic default. A future migration or a new `AccountID` field in the type can fix this properly.

### settlement_date required by DB schema
`securities.orders.settlement_date NOT NULL` has no equivalent in `SecurityOrder`. We default to `CURRENT_DATE + INTERVAL '2 days'` (T+2) on INSERT.

### List — no pagination fields in InstrumentFilters/OrderFilters
The interface does not carry `limit`/`offset`. Both `PgInstrumentStore.List` and `PgOrderStore.List` return all matching rows ordered deterministically (`instrument_id` ASC, `created_at` DESC). Pagination is applied in-memory by the server handlers (as is already done for the in-memory stores).

### Cancel — distinguishing "not found" vs "wrong state"
When 0 rows are affected by the UPDATE (status IN ('NEW','PARTIALLY_FILLED')), the store does a secondary SELECT EXISTS to distinguish ErrNotFound from a wrong-state error.

### DATE and TIMESTAMPTZ formatting
`listing_date` (DATE) is formatted as `YYYY-MM-DD` via `TO_CHAR`. `created_at`/`updated_at` (TIMESTAMPTZ) are formatted as RFC3339 UTC strings matching what the in-memory store produces.

### No pgx-specific types used
All scan targets are stdlib types (`string`, `int`, `float64`, `int64`). This keeps the store free of pgx-specific scanner dependencies while still using the pgx driver.
