## Test Infrastructure Pattern

All handler tests follow the pattern established in `handlers_instrument_test.go`:

- Package `server` (internal, white-box)
- `newXxxTestServer(t) (*httptest.Server, *store.InMemoryXxxStore)` — returns server + direct store access for seeding
- Reuses shared helpers: `doJSON`, `assertStatus`, `decodeBody`, `testTenant`
- `TenantMiddleware` with `testTenant = "ace-commodities"` wraps all handlers
- `t.Cleanup(ts.Close)` handles server teardown

## Decisions Made

1. **Separate test server constructors per feature** — rather than extending `newTestServerWithStores`, created `newPendingChangeTestServer`, `newReferencePriceTestServer`, `newPositionTestServer`. Each is self-contained and clearly signals which stores are under test.

2. **Direct store seeding for GetReferencePrice staleness tests** — seeded via `rpStore.Set()` with a fabricated past timestamp to avoid time.Sleep in tests. The stale detection path would be untestable via HTTP POST alone (SetAt is set server-side to `time.Now()`).

3. **Position store seeded via `Update()` not `GetOrCreate()`** — `Update()` upserts, which is the correct path for pre-populating existing positions in tests. `GetOrCreate()` would also work but Update is semantically correct for seeding non-zero positions.

4. **Four-eyes test uses exact submitter/reviewer match** — the handler checks `req.ReviewerID != "" && req.ReviewerID == change.SubmittedBy`. Test correctly sets both to `"maker-1"` to trigger the 403.

5. **ApproveNonPending also tests Reject path** — a single test function covers both the Approve and Reject error on non-PENDING_APPROVAL, keeping related error paths co-located.
