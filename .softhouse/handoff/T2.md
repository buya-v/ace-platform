## Decisions Made

1. **`TestSurveillance_Layering_Real` uses the engine's real `checkLayering`** — not a LARGE_TRADE threshold proxy. It submits 6 PENDING BUY orders from `layer-participant`, then has `oppositeParticipant` submit the incoming SELL that crosses one of them. The `checkLayering` call fires for `buyOrder.ParticipantID` (= `layer-participant`) which has 6 resting PENDING buys on that instrument, exceeding the `> 5` threshold.

2. **`TestSurveillance_Concentration` satisfies the `tradeCount >= 5` guard** — the engine's `checkConcentration` silently returns if fewer than 5 trades exist for the instrument today. The test seeds 4 background trades (10 shares each, distinct participants) then 1 dominant trade (100 shares), totalling 5 trades. The dominant buyer holds 100/(4×10+100) ≈ 71% of daily volume → alert fires.

3. **`newInvestigationAndSurveillanceTestServer` passes both stores to `New()`** — the handler's `handleCreateInvestigation` calls `s.surveillanceStore.UpdateAlertStatus(...)` only when both `req.AlertID != ""` AND `s.surveillanceStore != nil`. Using the investigation-only server (which passes `nil` for survStore) would silently skip the status update, so a dedicated helper was needed.

4. **Dashboard tests reuse `seedAlert`** from `handlers_surveillance_test.go` — the helper is already declared in `package server` scope, no re-declaration needed.

5. **`TestDashboard_NotConfigured` uses `newTestServer`** — that helper passes `nil` for the surveillance store, which triggers the `503 NOT_CONFIGURED` branch in `handleSurveillanceDashboard`.
