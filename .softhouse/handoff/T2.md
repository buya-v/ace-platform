# T2 Handoff — Securities-Service Tenant-Awareness

**Status:** DONE
**Build:** `go build ./...` — PASS
**Tests:** `go test ./...` — PASS (all packages)

---

## Summary

Made `securities-service` fully tenant-aware by threading `tenantID` through the middleware → handler → engine → Kafka publish chain. Topics are now dynamically scoped per tenant instead of using hardcoded `ace-commodities.*` constants.

---

## Changes Made

### 1. New: `internal/middleware/tenant.go`
Copied and adapted the `TenantMiddleware` pattern from `src/gateway/internal/middleware/tenant.go`.

- `TenantID` named string type with `String()` method
- `TenantMiddleware(validTenants []string) func(http.Handler) http.Handler` — validates `X-GarudaX-Tenant` header, skips `/healthz`, `/readyz`, `/metrics`
- `ValidTenantsFromEnv()` — reads `VALID_TENANTS` env var (comma-separated), defaults to `"ace-commodities,mse-equities"`
- `WithTenant(ctx, id)`, `TenantFromContext(ctx)`, `MustTenant(ctx)` — context helpers

### 2. Updated: `internal/kafka/topics.go`
Changed 3 constants to functions:
```go
func TopicTradeExecuted(tenantID string) string  { return tenantID + ".securities.trade.executed" }
func TopicOrderCreated(tenantID string) string   { return tenantID + ".securities.order.created" }
func TopicOrderCancelled(tenantID string) string { return tenantID + ".securities.order.cancelled" }
```

### 3. Updated: `internal/kafka/wiring.go`
`PublishTradeExecuted`, `PublishOrderCreated`, `PublishOrderCancelled` now take `tenantID string` as second parameter.

### 4. Updated: `internal/engine/engine.go`
`MatchOrder` signature: `func (e *MatchingEngine) MatchOrder(tenantID string, order *types.SecurityOrder)`. Passes `tenantID` to `PublishTradeExecuted`.

### 5. Updated: `internal/server/handlers_order.go`
- `handleSubmitOrder`: extracts `tenantID` from context via `middleware.TenantFromContext`; passes to `PublishOrderCreated` and `engine.MatchOrder`; includes `"tenant_id"` in the 201 response
- `handleCancelOrder`: extracts `tenantID` from context; passes to `PublishOrderCancelled`

### 6. Updated: `internal/server/server.go`
`StartAPIServer` now wraps the mux with `middleware.TenantMiddleware(middleware.ValidTenantsFromEnv())`.

### 7. Updated: `cmd/securities-service/main.go`
Registers per-tenant Kafka topics for all tenants from `ValidTenantsFromEnv()` at startup, replacing the hardcoded single-tenant registration.

## Design Decisions

1. **TenantMiddleware before routing** — same pattern as gateway; health paths (`/healthz`, `/readyz`, `/metrics`) bypass enforcement via exact-path match, so the health server (on port 9089) is unaffected.

2. **`MustTenant` not used in handlers** — handlers use `TenantFromContext` + explicit 401 check rather than panic-on-missing, to give cleaner error messages in the rare case middleware is misconfigured.

3. **`handleCancelOrder` also validates tenant** — even though cancel doesn't need it for topic routing today, it's consistent policy; future multi-tenant ownership checks will need it.

4. **Main.go registers topics for all tenants at startup** — the in-memory `ChannelProducer` requires explicit topic registration. A real wire-protocol Kafka producer would not need this step.

## Suggested Follow-ups

- **T3 (Instrument handlers tenant-awareness)**: `handleCreateInstrument`, `handleListInstruments`, `handleUpdateInstrumentStatus` should also extract and include `tenant_id` in responses for consistency.
- **T4 (Tenant-scoped store filtering)**: Order and instrument stores currently hold all tenants' data in a single in-memory map. For production isolation, stores need a `tenantID` filter on all queries.
- **T5 (Settlement tenant routing)**: `handleSettlementCycle` and `settlement.SettlementEngine` need the same tenantID threading for Kafka events.
- **T6 (Middleware tests for server integration)**: Add tests that send requests without tenant header and assert the 401 response from the full HTTP stack.
