# T2 Handoff: Platform Schema Migrations (V29 + V30)

**Task:** Create `V29__platform_schemas.sql` and `V30__ace_schema_renames.sql` — the SQL foundation for GarudaX multi-tenancy
**Status:** COMPLETE
**Artifacts:**

- `/infrastructure/db/migrations/V30__ace_schema_renames.sql`

---

## Summary

Produced two Flyway-style SQL migrations implementing Phase 0.6 (ace-commodities retrofit) of the GarudaX multi-tenant architecture, as specified in `GarudaX_Strategy_Directive.md §3.2 / §5` and `docs/platform-architecture.md §2, §3, §11, §12`.

---

## What V29 Does

Creates the `platform` schema with two tables:

**`platform.tenants`** — tenant registry (§2.1 DDL)
- Full column set from the spec: `tenant_id` (slug PK), `display_name`, `description`, `status`, `is_flagship`, `governance_tier`, `asset_classes` (TEXT[]), `default_settlement_cycle`, `primary_currency`, `timezone`, `regulatory_body`, `kms_cmk_arn`, `onboarding_metadata` (JSONB), `config_version`, timestamps, lifecycle timestamps (`activated_at`, `suspended_at`, `decommissioned_at`)
- Unique partial index on `is_flagship WHERE is_flagship = TRUE` — enforces the one-flagship invariant
- Seeds both known tenants: `ace-commodities` (ACTIVE, STANDARD) and `mse-equities` (ONBOARDING, FLAGSHIP)

**`platform.audit`** — platform-level audit log (§3.2 DDL)
- Hash-chained append-only log: `prev_hash` + `entry_hash` columns; hash population is the application's responsibility
- `CREATE OR REPLACE RULE` blocks UPDATE and DELETE permanently
- Separate from each tenant's `*_compliance.audit_events` chain — this table covers tenant lifecycle and cross-tenant admin events only

**Service role grants:**
- Creates `garudax_platform_svc` (the future control-plane service's DB role)
- Grants SELECT on `platform.tenants` to all existing service roles so they can validate tenant context in middleware without a round-trip to the control plane

## What V30 Does

Renames all 8 single-tenant schemas to the `ace_*` namespace (§5, Appendix A.1):

| Before | After |
|---|---|
| `reference` | `ace_reference` |
| `participants` | `ace_participants` |
| `exchange` | `ace_exchange` |
| `clearing` | `ace_clearing` |
| `compliance` | `ace_compliance` |
| `warehouse` | `ace_warehouse` |
| `market_data` | `ace_market_data` |
| `securities` | `ace_securities` |

`auth` is intentionally **not** renamed — it is platform-level per §3.4.
`platform` was created in V29 and is also **not** renamed.

Re-grants schema USAGE and table-level permissions to all existing service roles under the new schema names. Role *names* are unchanged in this migration; IRSA role renames are a Terraform-level operation (separate Phase 0.6 step).

## Key Decisions Made

1. **Full spec DDL, not a simplified stub.** V29 uses the full DDL from `platform-architecture.md §2.1` (adds `asset_classes`, `default_settlement_cycle`, `primary_currency`, `timezone`, `regulatory_body`, `kms_cmk_arn`, `config_version`, lifecycle timestamps). A simplified subset would not satisfy the downstream control-plane service.

2. **`CREATE OR REPLACE RULE` for append-only audit.** Matches the exact pattern specified in §3.2. The RULE approach is simpler than a trigger and is idempotent (safe to re-apply if migration is retried).

3. **`gen_random_uuid()::text` for `audit_id`.** The spec notes UUID v7 as preferred but defers to the application. Using `gen_random_uuid()` (v4) with a cast to TEXT keeps the column type consistent with other `id VARCHAR(64)` primary keys in the codebase (e.g., V26 `securities.instruments`). Application layer should supply v7 UUIDs for time-ordered queries.

4. **Grants re-issued in V30** under new schema names. Without this, existing service roles would lose USAGE on their schemas after the rename. The grants use `IF EXISTS` guards so V30 is safe to run on a fresh DB (e.g., CI) where roles may not yet exist.

5. **`ON CONFLICT (tenant_id) DO NOTHING`** for seed data. Safe to run on a DB that already has partial seed data.

6. **Locking note in V30 header.** `ALTER SCHEMA RENAME` takes an exclusive lock. The comment documents this so operators can plan the deployment window. The lock is brief (milliseconds) but worth noting for a live commodity exchange.

## Migration Style Conformance

Both files follow the style established in V1, V25, V26:
- Plain SQL with `DO $$ BEGIN ... END $$` role-guard blocks from V1
- `CREATE ... IF NOT EXISTS` everywhere for idempotency
- `CREATE INDEX IF NOT EXISTS` with descriptive names following `idx_{schema}_{table}_{column(s)}` convention
- Header comment block explaining version, purpose, and source document references
- Inline comments on columns matching the V26 style

## Blockers / Risks

- **Application code not updated here.** V30 renames schemas at the DB level only. All Go services must update their SQL queries from `exchange.*` to `ace_exchange.*` etc. in the same deployment. This is a code-level change outside the scope of this migration task.
- **`tickets` schema.** V25 creates a `tickets` schema that is not domain-scoped to any tenant. If this schema holds production data for `ace-commodities`, it should be renamed to `ace_tickets` in a follow-up (or kept platform-level with a comment). Not included in V30 to avoid scope creep.
- **TimescaleDB hypertables.** If `ace_market_data` contains hypertables (V8), Postgres `ALTER SCHEMA RENAME` also renames the hypertable's schema. TimescaleDB metadata tables store the schema name and may need a secondary update via `SELECT alter_table_set_schema(...)`. Coordinate with the DBA before applying V30 to production.

## Suggested Follow-ups

- **T3 (Code Update):** Update all Go services to use `ace_{domain}` schema names. Each service reads `GARUDAX_TENANT_ID` from env and constructs schema name as `fmt.Sprintf("%s_%s", tenantID, domain)`. This is the "Step 2: Code Updates" from platform-architecture.md §11.
- **T4 (Control Plane Service):** Build `src/control-plane/` Go service that owns `platform.tenants` and `platform.audit`. Reads T2's V29 schema as its data layer.
- **T5 (Tenant Middleware):** Implement `TenantFromContext(ctx)` and the `X-GarudaX-Tenant` signed header pattern (platform-architecture.md §4). All services must adopt this before MSE onboarding.
- **tickets schema:** Decide whether `tickets` is platform-level or per-tenant. If per-tenant, add a V31 to rename it (e.g., `ALTER SCHEMA tickets RENAME TO ace_tickets`) and add `tenant_id` column.
- **TimescaleDB check:** Before applying V30 to production, confirm whether `market_data` contains active hypertables and whether a TimescaleDB-specific rename step is needed alongside the schema rename.
