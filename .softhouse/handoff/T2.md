## Provisioner Design

### Schema names (8 per tenant)
Hyphens in tenant ID are replaced with underscores for valid SQL identifiers:

```
{sanitised_id}_reference
{sanitised_id}_participants
{sanitised_id}_exchange
{sanitised_id}_clearing
{sanitised_id}_compliance
{sanitised_id}_warehouse
{sanitised_id}_market_data
{sanitised_id}_securities
```

Example: tenant `mse-equities` → `mse_equities_reference`, `mse_equities_participants`, etc.

### Kafka topic prefixes (8 per tenant)
Tenant ID used as-is (hyphens are valid in Kafka topic names):

```
{tenant_id}.trades
{tenant_id}.clearing
{tenant_id}.margin
{tenant_id}.settlement
{tenant_id}.compliance
{tenant_id}.market-data
{tenant_id}.warehouse
{tenant_id}.auth
```

### Dry-run mode
When `db == nil` (current MVP): schemas and topic lists are computed and returned in `ProvisionResult` but no SQL is executed. When a real `*sql.DB` is provided, each schema is created with `CREATE SCHEMA IF NOT EXISTS` inside a single transaction; rollback on any error.

## Tenant Middleware Bypass

The `/platform/` prefix bypass is handled in `TenantMiddleware` via a `tenantBypassPrefixes` slice checked at request time. The bypass runs before the header-presence check, so platform API calls receive no `TENANT_REQUIRED` error.

Auth middleware still applies — platform API callers need a valid JWT. The bypass is tenant-scope only, not auth-scope.

## Decisions Made

1. **`httputil.ReverseProxy` for platform routes** — Platform-service is a pure REST service and its path structure (`/platform/v1/tenants`) should be forwarded as-is. Using `httputil.ReverseProxy` avoids having to add entries to the `rpcToHTTP` method map in `httpclient.go`, which is designed for RPC-style service proxying. The reverse proxy transparently forwards the request path, method, body, and response.

2. **`NewWithProvisioner` constructor** — Added alongside the existing `New` (which uses `provisioning.New(nil)`) so callers with a real DB can inject a configured provisioner without changing `New`'s signature.

3. **Provisioning failure is non-fatal for MVP** — If `ProvisionTenant` returns an error (only possible when db!=nil and SQL fails), the handler returns HTTP 201 with a `provisioning.status="FAILED"` payload. The tenant record was already committed. This avoids a transactional rollback concern between the in-memory store and DB provisioning.

4. **`tenantBypassPrefixes` slice, not map** — The bypass list is small (currently one entry) and checked sequentially. A slice with `strings.HasPrefix` is simpler than a radix tree or map. If the list grows significantly, a sorted prefix array with binary search would be appropriate.

5. **Schema names use underscore-sanitised ID, topic prefixes use raw ID** — SQL identifiers cannot contain hyphens; Kafka topic names can. The `sanitiseID` helper applies only to schema construction.
