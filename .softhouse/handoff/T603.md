## Design Decisions

### 1. Graceful skip pattern
Each test skips via `test.skip()` in `beforeEach` if `isPortalAvailable()` returns false. This follows the established ACE Platform UAT pattern — CI passes even without a running admin portal.

### 2. Independent login per test
Every test navigates to `/login` and logs in fresh. This avoids shared state between tests and is consistent with how the existing `admin-verification.spec.ts` tests work.

### 3. SPA sidebar navigation
All navigation uses `navigateTo()` from `helpers.ts`, which clicks sidebar links rather than using `page.goto()`. This preserves the in-memory React auth token — a hard constraint from the admin-ui architecture (`Sidebar.tsx` uses React Router's `NavLink`).

### 4. Dual validation layer
Each test runs both:
- The shared `runPageCheck()` from `page-checks.ts` (reusable validators)
- Test-specific assertions for page-unique elements (e.g. KPI cards for Dashboard Home, halt/resume buttons for Circuit Breakers)

### 5. Soft assertions throughout
All content assertions use `expect.soft()` so a single missing element doesn't abort the test. Hard assertions are only used for the skip guard.

### 6. Bot Chat test (test 17)
Uses `openBotChat()` and `sendBotMessage()` from `helpers.ts`. The test handles the case where the bot button is absent (soft assertion, early return with screenshot) so it doesn't fail if the bot feature is not deployed.

### 7. Exact sidebar labels
Navigation labels match the exact strings from `Sidebar.tsx`:
- Operations section: `System Health`, `Order Book`, `Positions`, `Risk Overview`, `Margin Calls`, `Settlement`, `Circuit Breakers`, `Warehouse`, `Market Phase`, `Surveillance`, `Fee Management`, `Reports`, `Tickets`
- Compliance section: `Participants`, `Compliance Alerts`, `Audit Log`
- Root link: `Overview` (the `/dashboard` NavLink)

None. All helper functions (`loginAsAdmin`, `navigateTo`, `waitForData`, `captureScreenshot`, `assertNoErrors`, `openBotChat`, `sendBotMessage`, `isPortalAvailable`) exist in `helpers.ts` and work correctly for these tests. The `PAGE_CHECKS` map in `page-checks.ts` has validators for 14 of the 18 pages; the remaining 4 (Risk Overview, Dashboard Home, Bot Chat, and the extended Circuit Breakers/Reports checks) are implemented inline.
