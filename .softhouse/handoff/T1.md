# T1 Handoff (Securities Architecture) — Securities Architecture Specification

## Summary

Created a complete architecture specification for the GarudaX Securities module (Phase 7) covering equities, bonds, and ETFs. The spec is designed to be directly consumable by implementation agents with zero ambiguity.

## Deliverables

- **`docs/securities-architecture.md`** — 12-section architecture specification (~1,100 lines)

## Key Decisions Made

1. **Separate `securities` schema** — New PostgreSQL schema with its own tables rather than extending `exchange` or `clearing` schemas. This keeps securities and commodity data isolated and independently queryable.

2. **Three migration files** (V26, V27, V28) — Split by concern: V26 = instrument reference data + trading controls, V27 = orders/trades/positions, V28 = settlement/CSD/corporate actions. Follows the existing pattern of V11/V12/V14.

3. **Reuse existing Kafka envelope** — Securities events use the `ace-event-envelope-v1` schema from T051. New topic domain is `ace.securities.*` following the `ace.{domain}.{event-type}` convention.

4. **Same CLOB matching algorithm** — Securities orders go through the identical price-time priority matching as commodities. The matching engine is extended with validation (lot size, tick size, short-sell) but the core algorithm is unchanged.

5. **CCP novation reused** — The clearing engine's existing CCP novation logic applies to securities. The key difference is that securities obligations carry `settlement_date` (T+2) rather than settling in the current daily cycle.

6. **Settlement state machine with 7 states** — PENDING -> AFFIRMED -> NETTED -> INSTRUCTED -> SETTLING -> SETTLED | FAILED. Each transition has explicit conditions and triggers.

7. **CSDR-style fail management** — Penalty interest (0.25-0.50 bps/day by asset class) plus buy-in procedures after T+4, following EU CSDR regulation as the model.

8. **API routes under `/api/v1/securities/`** — Separate namespace from commodity endpoints. Same auth roles (trader, clearing_admin, exchange_admin).

9. **Existing services extended, not replaced** — Matching engine, clearing engine, settlement engine, and margin engine are extended with securities-specific logic. Gateway, auth, compliance, market-data are reused without modification (only new route registrations in gateway).
