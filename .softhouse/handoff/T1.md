## Summary

### Part A — New Alert Type Constants (types.go)

Appended 8 new `AlertType` constants to the existing 4:
- `AlertTypeFrontRunning` = `"FRONT_RUNNING"`
- `AlertTypeSpoofing` = `"SPOOFING"`
- `AlertTypeLayering` = `"LAYERING"`
- `AlertTypeInsiderTrading` = `"INSIDER_TRADING"`
- `AlertTypeMarketManipulation` = `"MARKET_MANIPULATION"`
- `AlertTypeConcentration` = `"CONCENTRATION"`
- `AlertTypeUnusualActivity` = `"UNUSUAL_ACTIVITY"`
- `AlertTypeCrossMarket` = `"CROSS_MARKET"`

Also added `AlertStatusInvestigating = "INVESTIGATING"` between OPEN and RESOLVED.

### Part B — Enhanced checkSurveillance (engine.go)

Refactored `checkSurveillance` into orchestrator + 5 focused sub-checks:

| Check | Alert Type | Trigger Condition |
|---|---|---|
| `checkSpoofing` | `SPOOFING` | Participant cancelled >3 orders in last 60 seconds |
| `checkLayering` | `LAYERING` | Participant has >5 PENDING orders on same side for same instrument |
| `checkInsiderTrading` | `INSIDER_TRADING` | Instrument has ANNOUNCED or PROCESSING corporate action |
| `checkMarketManipulation` | `MARKET_MANIPULATION` | Trade price deviates >5% from explicit `MARKET_MANIPULATION` threshold |
| `checkConcentration` | `CONCENTRATION` | Participant >20% of daily volume (requires ≥5 trades to suppress false positives) |

Added `SetCorporateActionStore(store.CorporateActionStore)` setter to `MatchingEngine`.

**Key design decisions:**
- `checkMarketManipulation` uses only explicit `AlertTypeMarketManipulation` thresholds — not PRICE_SPIKE — to avoid conflicting with existing threshold tests.
- `checkConcentration` requires `tradeCount >= 5` before firing to prevent alerts at market open when one participant is necessarily 100% of minimal volume.
- All heuristic checks are gated behind `e.orderStore != nil` and silently fail.

### Part C — Alert-to-Investigation Linking

- Added `AlertID string` field to `types.Investigation` (`json:"alert_id,omitempty"`).
- Updated `handleCreateInvestigation` to accept `alert_id` in request body.
- When `alert_id` is provided and `surveillanceStore` is wired, the alert transitions to `AlertStatusInvestigating` after investigation creation.
- Added `GetAlert(id string)` and `UpdateAlertStatus(id string, status AlertStatus)` to `SurveillanceStore` interface and `InMemorySurveillanceStore` implementation.

### Part D — Surveillance Dashboard Handler

New file: `src/securities-service/internal/server/handlers_surveillance_dashboard.go`

`GET /api/v1/securities/surveillance/dashboard` returns:
```json
{
  "total_alerts": {"open": N, "investigating": N, "resolved": N},
  "alerts_by_type": {"LARGE_TRADE": N, "SPOOFING": N, ...},
  "top_firms": [{"firm_id": "X", "alert_count": N}],
  "top_instruments": [{"instrument_id": "X", "alert_count": N}],
  "recent_alerts": [...last 10 by CreatedAt desc...]
}
```

Aggregation is done in-handler from `ListAlerts({})`. Top lists limited to 10, sorted descending.

Note: `top_firms` currently uses `InstrumentID` as a firm proxy — full implementation requires participant→firm join.

- **gateway handler.go**: Added `SecuritiesSurveillanceDashboard` proxy method.
