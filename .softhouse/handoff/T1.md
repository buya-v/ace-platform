## Architecture Decisions

1. **HTTP-only, no gRPC.** The securities-service is a supporting service (reference data + order gateway), not a real-time trading pipeline engine. HTTP fits the CRUD pattern better and matches auth-service / compliance-service. Port 8085 (API) + 9085 (health) follows the `service_api = 808x, health = 908x` convention.

2. **Port allocation.** The existing platform already uses 8085 for auth-service in the gateway's backend map (`cfg.AuthServiceAddr` → port 8085). The securities-service is a *new* service that the gateway does not yet know about. Port 8085 is safe for securities-service since auth-service runs on its own pod; the gateway should be updated to map `"securities-service"` → `:8085` and auth-service should either keep 8085 internally or move. This is a follow-up concern.

   **Recommended resolution:** use port 8089 for securities-service API and 9089 for health to avoid collision. The scaffold uses `DefaultConfig()` which can be overridden via `API_PORT` / `HEALTH_PORT` env vars without a code change.

3. **Atomic readyFlag.** Uses `atomic.Int32` (same as auth-service `server.go`) rather than the gateway's package-level `int32` pattern. Both work; `atomic.Int32` is safer for embedding.

4. **In-memory stores.** Consistent with every other service at scaffold time. `InstrumentStore` and `OrderStore` interfaces are defined so a PostgreSQL implementation can be dropped in later without changing callers.

5. **Partial update semantics.** `InstrumentUpdate` uses zero-value-means-no-change — standard Go idiom for patch operations without requiring pointers for every field.

6. **No router library.** Uses stdlib `net/http.ServeMux`. The gateway uses a custom router; individual services use ServeMux. Pattern matches auth-service.

## Types Implemented (from securities-architecture.md)

- `AssetClass`: EQUITY, BOND, ETF
- `TradingStatus`: ACTIVE, HALTED, SUSPENDED, DELISTED
- `OrderType`: LIMIT, MARKET, STOP, STOP_LIMIT
- `OrderSide`: BUY, SELL, SHORT_SELL
- `OrderStatus`: PENDING, PARTIALLY_FILLED, FILLED, CANCELLED, REJECTED, EXPIRED
- `TimeInForce`: GTC, IOC, FOK, DAY, GTD
- `Instrument` struct: all fields with json snake_case tags
- `BondDetails` struct: InstrumentID, MaturityDate, CouponRate, CouponFrequency, ParValue, DayCountConvention
- `SecurityOrder` struct: all fields with json snake_case tags
- `ErrorResponse` / `ErrorDetail`: standard error envelope

Note: The architecture spec's `SecurityInstrument` has many additional fields (IssuerName, Sector, MarketCap, bond/ETF-specific fields, trading controls). The `Instrument` struct in types.go implements the core fields required by the task spec. Additional fields from the full spec can be added in the next task without breaking the store interface.
