## Summary

Produced the exhaustive GarudaX multi-tenant platform architecture specification covering all 12 required sections. The document is designed to be directly actionable by implementation agents -- every section includes exact DDL, pseudo-code, naming conventions, and configuration structures.

## Key Decisions Made

1. **Schema rename strategy:** `ALTER SCHEMA ... RENAME TO` in V29-V30 migrations (Postgres supports this natively, zero-downtime for reads, brief lock for DDL).

2. **Tenant ID format:** Lowercase slug with hyphens (`ace-commodities`, `mse-equities`), validated by regex `^[a-z][a-z0-9-]{2,62}[a-z0-9]$`. Immutable after creation.

3. **Defense-in-depth tenant_id column:** Every table in every tenant schema gets a `tenant_id NOT NULL` column with a `CHECK` constraint. This is redundant with schema-level isolation but prevents bugs where a service accidentally connects to the wrong schema.

4. **HMAC-signed tenant header:** `X-GarudaX-Tenant: {tenant_id}:{timestamp_ms}:{hmac-sha256}` -- signed to prevent header injection, timestamped to prevent replay. 5-minute freshness window.

5. **JWT tenant_roles map:** Replaced flat `roles` array with `tenant_roles: {"ace-commodities": ["admin","trader"], "mse-equities": ["viewer"]}`. Platform roles in separate `platform_roles` array.

6. **Kafka topic convention:** `{tenant_id}.{domain}.{event-type}` (e.g., `mse-equities.securities.trade-executed`). Consumer groups: `{tenant_id}-{service}-consumer`.

7. **Platform schemas:** `platform.tenants`, `platform.audit`, `platform.tenant_config`, `platform.platform_config` -- all platform-level, not tenant-scoped.

8. **MSE-wins precedent:** Default settlement cycle is T+2 (MSE preference), default market session includes auctions (MSE preference). ACE overrides to T+0 and continuous-only via tenant config.

9. **Migration phasing:** 0.5 (this spec) -> 0.6 (ACE retrofit: schema renames, topic renames, IRSA renames) -> 0.7 (control plane service) -> 0.8 (MSE tenant build). Each phase is a softhouse run.
