## Key Decisions Made

1. **`TenantID` is a named string type** ‚Äî prevents accidental mixing of raw `string` tenant IDs with other string parameters. Implements `fmt.Stringer` for ergonomic logging.

2. **Unexported `contextKey` struct** ‚Äî uses a zero-size struct type (not a string constant) as the context key. This guarantees no external package can construct or collide with the key. Matches the pattern used in `internal/observability/logger.go`.

3. **`MustTenant` panics intentionally** ‚Äî consistent with the platform architecture spec ¬ß4.2 pseudo-code: "A request without a tenant context is a bug." The panic message names the required middleware so the developer knows exactly what to add.

4. **`TenantMiddleware` takes `[]string` ‚Üí builds `map[string]bool`** ‚Äî construction-time O(n) cost amortised over all requests; per-request lookup is O(1). No sync needed because the map is read-only after construction.

5. **Health bypass uses exact path matching** ‚Äî `/healthz`, `/readyz`, `/metrics` bypass tenant enforcement. These three are fixed constants (not a prefix match) to avoid accidental bypass of paths like `/metricsadmin`.

6. **Error codes follow task spec exactly** ‚Äî `TENANT_REQUIRED` (401) for missing header, `UNKNOWN_TENANT` (403) for unregistered tenant. The architecture doc (¬ß4.1) uses different codes (`MISSING_TENANT_CONTEXT`, `TENANT_NOT_FOUND`) but the task spec takes precedence; downstream code should use the constants from this package.

7. **Simplified middleware (no HMAC)** ‚Äî the task specification calls for header/whitelist validation only. Full HMAC + timestamp verification (as shown in platform-architecture ¬ß4.2) is a gateway-layer concern; internal services receive a pre-validated tenant_id. A comment in `middleware.go` documents this design boundary.

8. **`TenantMetricLabels` returns `""` when no tenant** ‚Äî callers on bypass paths (/healthz etc.) can unconditionally call `TenantMetricLabels` without branching; the empty string is a valid Prometheus label value for platform-level metrics.

## Suggested Follow-ups

- **All domain services** should call `tenant.MustTenant(ctx)` at the top of every handler function ‚Äî before any database query ‚Äî to surface missing tenant bugs immediately.
- **Observability integration** ‚Äî `TenantLogger` should be called in `TracingMiddleware` (observability package) after tenant context is available so every log line automatically carries `tenant_id`. Sequence: TenantMiddleware ‚Üí TracingMiddleware ‚Üí handler.
