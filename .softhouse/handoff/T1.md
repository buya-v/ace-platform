## What Was Done

### Part A — New types in `types.go`

- `ShortSellRestricted bool` added to `Instrument` struct
- `LocateID string` added to `SecurityOrder` struct
- New struct `LocateRequest` (id, instrument_id, borrower/lender firm, quantity, status, timestamps)
- New struct `RequestForQuote` (id, instrument_id, requestor_firm, quantity, side, status, response_quote_id, tenant_id, timestamps)
- New struct `GiveUpRequest` (id, trade_id, from/to firm, status, reason, timestamps)

Statuses: Locate — PENDING | APPROVED | REJECTED | EXPIRED | USED; RFQ — OPEN | RESPONDED | EXPIRED | CANCELLED; GiveUp — PENDING | ACCEPTED | REJECTED.

### Part B — New stores in `store.go`

- `LocateStore` interface + `InMemoryLocateStore`: Create, Get, List(firmID), Approve(id, lenderFirmID), Use(id)
- `RFQStore` interface + `InMemoryRFQStore`: Create, Get, List(instrumentID, status), Respond(id, quoteID), Cancel(id)
- `GiveUpStore` interface + `InMemoryGiveUpStore`: Create, Get, List(firmID), Accept(id), Reject(id, reason)

All use sequential int IDs (`nextID` counter).

### Part C — `handlers_locate.go` (new file)

- `POST /api/v1/securities/locates` — request a locate (validates instrument, firm, quantity)
- `GET /api/v1/securities/locates` — list locates (optional firm_id query param)
- `POST /api/v1/securities/locates/{id}/approve` — approve a locate (sets lender_firm_id)
- Returns 503 if locateStore is nil

### Part D — `handlers_rfq.go` (new file)

- `POST /api/v1/securities/rfq` — submit RFQ (validates side: BUY or SELL)
- `GET /api/v1/securities/rfq` — list RFQs (optional instrument_id/status filters)
- `POST /api/v1/securities/rfq/{id}/respond` — respond to RFQ (quote_id required)
- `POST /api/v1/securities/rfq/{id}/cancel` — cancel RFQ
- Returns 503 if rfqStore is nil

### Part E — `handlers_give_up.go` (new file)

- `POST /api/v1/securities/trades/{id}/give-up` — initiate give-up for a trade
- `GET /api/v1/securities/give-ups` — list give-ups (optional firm_id filter)
- `POST /api/v1/securities/give-ups/{id}/accept` — accept give-up
- `POST /api/v1/securities/give-ups/{id}/reject` — reject give-up (reason required)
- Returns 503 if giveUpStore is nil

### Part F — Short-sell validation in `handlers_order.go`

Replaced the `SHORT_SELL_DISABLED` block with real validation:
1. If `inst.ShortSellRestricted` → 422 `SHORT_SELL_RESTRICTED`
2. If `order.LocateID == ""` → 422 `LOCATE_REQUIRED`
3. If locate store available: validates locate is APPROVED, marks it USED; invalid locate → 422 `INVALID_LOCATE`

### Part G — Wiring

**`server.go`**: Added `locateStore`, `rfqStore`, `giveUpStore` fields; updated `New()` from 28 to 31 params; added 6 new routes; added `give-up` case to `handleTrade` dispatcher.

**`cmd/securities-service/main.go`**: Passed `NewInMemoryLocateStore()`, `NewInMemoryRFQStore()`, `NewInMemoryGiveUpStore()` to `server.New()`.

**`src/gateway/cmd/gateway/main.go`**: Registered 11 new routes for P4a endpoints.

## Decisions Made

- Sequential int IDs for new stores (matches `LocateRequest.ID int` type), unlike existing UUID-based stores
- `hasSuffix` and `extractPenultimateSegment` helpers in handlers_locate.go for sub-resource routing
- `Use()` on locate is best-effort when locateStore is available but the order proceeds regardless if store is nil
- Give-up handler for `POST /trades/{id}/give-up` uses string trade ID directly (trade store uses string IDs)
