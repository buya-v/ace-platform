# T118 — OpenTelemetry Tracing and Structured Logging

## Status: DONE

## Summary

Added distributed tracing and structured logging to the GarudaX platform using only Go stdlib packages (log/slog, sync/atomic, net/http). Created a shared observability package and integrated it into the gateway and matching-engine services.

## What Was Done

### 1. Created `src/shared/internal/observability/` package (template)

- **`logger.go`**: Structured JSON logger wrapper using `log/slog`
  - `NewLogger(serviceName)` — creates a JSON slog logger with service_name field
  - `NewLoggerWithLevel(serviceName, level)` — configurable log level
  - Context helpers: `WithTraceID`, `WithRequestID`, `TraceIDFromContext`, `RequestIDFromContext`
  - `LoggerFromContext(ctx, base)` — enriches logger with trace_id and request_id from context

- **`middleware.go`**: HTTP tracing middleware
  - Extracts or generates trace_id from X-Request-ID header
  - Stores trace_id and request_id in context
  - Logs request_started and request_completed events with duration, status, path
  - Sets X-Request-ID and X-Trace-ID response headers
  - Log level: INFO for 2xx, WARN for 4xx, ERROR for 5xx

- **`metrics.go`**: Prometheus-compatible metrics (no external deps)
  - `NewMetrics(serviceName)` — thread-safe request counter and latency histogram
  - `MetricsMiddleware()` — records request count by method+status and latency
  - `MetricsHandler()` — serves Prometheus text format at /metrics
  - `MetricsJSON()` — serves JSON metrics at /metrics.json
  - Uses `sync/atomic` + `sync.Mutex` instead of `expvar` to avoid global registry conflicts in tests

### 2. Updated Gateway (`src/gateway/`)

- **`cmd/gateway/main.go`**: Replaced all `log.Printf`/`log.Fatalf` with slog structured logging
  - Added `observability.TracingMiddleware` to middleware chain
  - Added `metrics.MetricsMiddleware()` to middleware chain
  - Added /metrics and /metrics.json endpoints on health port
  - Added /metrics to public paths (no auth required)

- **`internal/middleware/logging.go`**: Replaced manual JSON marshaling with `slog.LogAttrs`
  - Removed `logEntry` struct and `encoding/json` import
  - Uses `slog.LevelInfo`/`LevelWarn`/`LevelError` based on status code
  - Includes request_id and user_id as structured fields

### 3. Updated Matching Engine (`src/matching-engine/`)

- **`cmd/matching-engine/main.go`**: Replaced all `log.Printf`/`log.Fatalf` with slog
  - Trade handler logs structured `trade_matched` events with trade_id, instrument_id, quantity, price, aggressor_side
  - Exec report handler logs structured `execution_report` events
  - Instrument registration logs `instrument_registered` events
  - Startup/shutdown messages use structured slog

### 4. Template-Duplicated to Services

Following the zero-dep pattern, the observability package is copied into each service's `internal/observability/` directory (same approach as the kafka and types packages).

## Decisions Made

1. **Used sync/atomic counter map instead of expvar**: `expvar.NewMap` panics on duplicate name registration, making it unsafe in tests. The custom `counterMap` using `sync.RWMutex` + `atomic.Int64` is test-friendly and equally performant.

2. **Trace ID == Request ID initially**: The middleware uses X-Request-ID as the trace_id. In a multi-service deployment, each service preserves the trace_id across hops. The gateway already forwards x-request-id in metadata to backend services.

3. **Metrics on health port**: The /metrics endpoint is served on the health port (separate from the main API port) following Kubernetes conventions for Prometheus scraping.

4. **slog replaces log.Printf as drop-in**: All `log.Printf("msg %s", val)` calls are replaced with `logger.Info("msg", slog.String("key", val))`. Fatal calls replaced with `logger.Error(...)` + `os.Exit(1)`.
