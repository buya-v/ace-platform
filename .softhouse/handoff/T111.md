# T111 — Fee Management Service

## Status: DONE

## Summary

Created a fee management package (`src/gateway/internal/fees/`) in the gateway service with:

1. **Database migration** (`infrastructure/db/migrations/V18__fee_tables.sql`):
   - `fees.fee_schedules` — fee schedule lifecycle with effective dates
   - `fees.fee_rules` — configurable rules by fee type, tier, instrument pattern with rate_bps, min/max/per-contract
   - `fees.fee_transactions` — recorded fee charges per trade
   - `fees.participant_tiers` — participant tier tracking with 30-day volume
   - Default schedule with 5 rules: farmer(10bps), hedger(15bps), speculator(25bps), market_maker(5bps), clearing(5bps)

2. **Fee calculator** (`calculator.go`):
   - Pure function `CalculateFee()` — no DB dependency, fully testable
   - Rule matching with priority: exact tier+instrument > exact tier+wildcard instrument > wildcard tier+exact instrument > wildcard tier+wildcard instrument
   - Instrument pattern matching with prefix wildcards (`WHT-*`)
   - Min/max fee enforcement and per-contract fee addition
   - `CalculateAllFees()` for computing all applicable fee types at once

3. **Store** (`store.go`):
   - `Store` interface for testability (mock in tests, PgStore for production)
   - PostgreSQL queries for schedules, rules, transactions, tiers
   - Dynamic query building for date-range filtered fee transactions

4. **HTTP handlers** (`handlers.go`):
   - `GET /api/v1/fees/schedule` — list active schedules with rules (authenticated)
   - `GET /api/v1/fees/my-fees?from=&to=` — participant's own fee transactions
   - `GET /api/v1/admin/fees` — all schedules (admin)
   - `POST /api/v1/admin/fees/rules` — create fee rule with validation (admin)

5. **Tests** (`calculator_test.go`): 47 tests, 65.9% coverage
   - Calculator: farmer/hedger/speculator/market_maker rates, min/max fee, per-contract, edge cases
   - Pattern matching: wildcard, prefix, exact
   - Rule priority: exact over wildcard at both tier and instrument levels
   - Handlers: success/error/auth/validation for all 4 endpoints
   - Utility: roundTo4, itoa, collectFeeTypes, isValidFeeType

## Decisions Made

- Fee calculation is a **pure function** — takes rules as input, no DB calls. This makes it trivial to test.
- Used the same Store interface pattern as `refdata` package for consistency.
- `middleware.ClaimsFromContext()` used for auth — same pattern as existing gateway handlers.
- Defaults `instrument_pattern` and `participant_tier` to `"*"` when omitted in rule creation.
- Used `itoa()` helper instead of importing `strconv` to keep zero-dep pattern within the package.
