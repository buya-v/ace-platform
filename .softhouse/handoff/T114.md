# T114 — Participant Tiering and Membership Management

## Status: DONE

## Summary
Extended the compliance-service with a participant tiering and membership lifecycle package. Created the SQL migration for the `participants` schema and built a complete `membership` Go package with service, store, and domain types.

## Artifacts Created

### Migration
- `infrastructure/db/migrations/V21__participant_management.sql` — Creates `participants.members` and `participants.membership_history` tables with indexes.

### Go Package: `src/compliance-service/internal/membership/`
- `member.go` — Domain types: `Member`, `HistoryEntry`, `Tier` (5 values), `Status` (4 values), `EntityType`, `NetWorthCategory`, `TierConfig` with fee/position/order-size limits, `ListFilter`, validation, transition rules.
- `store.go` — `Store` interface and `MemoryStore` in-memory implementation (deep-copy safe).
- `service.go` — `Service` with operations: `CreateMember`, `Activate`, `ChangeTier`, `Suspend`, `Reinstate`, `Terminate`, `GetMember`, `ListMembers`, `GetMemberHistory`, `GetTierConfig`.
- `service_test.go` — 55 tests, 91.4% coverage. Covers all operations, invalid transitions, error cases, full lifecycle, tier configs, store isolation.

## Decisions Made
1. **Zero external dependencies** — follows the established zero-dep Go module pattern.
2. **Activate operation added** — task spec had PENDING->ACTIVE on KYC approval but no explicit Activate method; added it as the KYC approval trigger.
3. **Terminate from SUSPENDED blocked** — requires reinstatement first (SUSPENDED->ACTIVE->TERMINATED), preventing accidental termination during investigation.
4. **TierConfig with operational parameters** — fee rates (bps), position limits, order size limits per tier, enabling downstream services to query limits.
5. **MemoryStore uses deep copies** — prevents mutation leaks between caller and store.
