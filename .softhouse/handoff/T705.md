## Decisions made

- **UAT-prefixed IDs**: All test data uses `uat-` prefix (`uat-test-commodity`,
  `uat-test-participant`, `uat-test-holder`, `UAT-TEST-2027M01`) to avoid
  conflicts with production data and make cleanup straightforward.

- **Commodity test (test 1) checks id in response**: The T701 implementation
  auto-generates IDs with `COMM-<hex>` prefix when none is supplied, but also
  accepts a client-provided `id`. The test accepts `body.id`, `body.ID`, or
  `body.data.id` to handle both the direct-object and wrapped-object response
  shapes used elsewhere in the gateway.

- **Update instrument (test 3) skips on 404**: The in-memory fallback store
  from T701 initialises empty, so `UAT-TEST-2027M01` may not exist if test 2
  was skipped or the server restarted. A 404 skip is correct behaviour here
  rather than a test failure.

- **Fees tier test (test 5) skips on 404**: Same logic â€” participant must exist
  in the store before a tier assignment is possible.

- **Screening and warehouse (tests 6â€“7) only assert status < 500**: These
  endpoints may legitimately return 4xx (validation errors, unknown
  participants) depending on backend state; the UAT contract only guarantees
  the service is reachable and does not crash.

- **Bot CRUD test (test 8) checks reply text loosely**: The bot's LLM response
  is non-deterministic. The assertion checks that the reply string (under
  `data.reply` or `reply`) contains either `"commodity"` or `"created"`,
  which is the minimum bar for a relevant response to a commodity-creation
  command. `expect.soft()` means this won't fail the suite on a chatty reply.

- **Separate `test.describe` block with its own `beforeAll`**: Kept isolated
  from the existing `API Contract UAT` suite so the two can be run
  independently or together without shared state.
