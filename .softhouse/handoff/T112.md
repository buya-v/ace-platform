# T112 — Pre-trade Risk Checks

## Summary

Added pre-trade risk validation to the gateway before forwarding orders to the matching engine. Implemented order size limits, notional value limits, and price band checks as pure functions with a fail-open design when the risk database is unavailable.

## Deliverables

### SQL Migration
- `infrastructure/db/migrations/V19__risk_parameters.sql` — Creates `risk` schema with `position_limits` and `order_limits` tables, seeds default limits for 6 commodity instruments.

### Risk Package (`src/gateway/internal/risk/`)
- `pretrade.go` — `PreTradeChecker` struct with `CheckOrder()`, plus pure functions: `CheckOrderSize()`, `CheckPriceBand()`, `CheckParticipantStatus()`, `ParseOrderRequest()`. All checks return `*RiskError` on failure or nil on pass.
- `store.go` — `Store` interface with `PgStore` PostgreSQL implementation. Methods: `GetOrderLimits`, `ListOrderLimits`, `UpsertOrderLimits`. Includes `DefaultOrderLimits()` for fail-open defaults.
- `pretrade_test.go` — 30 tests covering all check functions, parser edge cases, fail-open behavior (nil store, DB errors, no limits configured), and integration scenarios.

### Gateway Handler Changes (`src/gateway/internal/handler/`)
- `handler.go` — Added `NewWithRisk()` constructor. Modified `SubmitOrder()` to run pre-trade risk checks before forwarding. Added `ListOrderLimits()` and `UpdateOrderLimits()` admin handlers.
- `routes.go` — Added `GET /api/v1/admin/risk/order-limits` and `PUT /api/v1/admin/risk/order-limits/{instrument_id}` routes.

## Design Decisions

1. **Fail-open**: When the risk DB is unavailable, orders are allowed through. This is intentional for a commodity exchange where blocking trading is worse than missing a risk check.
2. **Pure functions**: `CheckOrderSize` and `CheckPriceBand` are pure functions that take data and return errors — easy to test, no side effects.
3. **Last price = 0**: Price band checks are skipped when `lastPrice <= 0` (first trade of the day scenario). The current wiring passes 0 for last price — a future task should wire in actual last trade data from market-data-service.
4. **Parse flexibility**: `ParseOrderRequest` handles quantity as both JSON number and string, matching the existing gateway pattern.

## Test Results

- Risk package: 30 tests, 70.5% coverage (uncovered: PgStore methods requiring real DB)
- All existing gateway tests: PASS (0 regressions across handler, middleware, auth, router, kafka, refdata, observability)
