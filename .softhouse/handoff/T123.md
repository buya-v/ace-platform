# T123 — Settlement Engine DVP Settlement and Multi-Instrument Support

## Status: DONE

## Summary

Enhanced the settlement engine with delivery-vs-payment (DVP) coordination and multi-instrument settlement cycle support. The DVP coordinator ensures atomic settlement for physical delivery instruments (validate delivery -> lock payment -> confirm delivery -> release payment), with full rollback on any step failure. The engine now supports running settlement cycles across ALL active instruments simultaneously, aggregating net settlement amounts per participant across instruments.

## Files Created

- `src/settlement-engine/internal/dvp/coordinator.go` — DVPCoordinator with CoordinateSettlement method
- `src/settlement-engine/internal/dvp/coordinator_test.go` — 9 tests covering all DVP paths

## Files Modified

- `src/settlement-engine/internal/types/settlement.go` — Added InstrumentType, InstrumentConfig, DeliveryReceipt, DVPResult, MultiInstrumentResult, InstrumentSettlementResult types
- `src/settlement-engine/internal/engine/engine.go` — Added RunMultiInstrumentCycle, RegisterInstrument, SetDVPCoordinator; preserved RunSettlementCycle for backward compatibility
- `src/settlement-engine/internal/engine/engine_test.go` — Added 6 multi-instrument/DVP integration tests

## Decisions Made

1. **Backward compatibility preserved**: The original `RunSettlementCycle` method is untouched. New functionality is in `RunMultiInstrumentCycle`.
2. **DVP as a coordinator, not external system**: DVP coordination uses `DeliveryValidator` and `PaymentLocker` interfaces. The coordinator handles correct ordering and rollback logic. Real implementations plug in behind the interfaces.
3. **Instruments default to cash-settled**: Unregistered instruments are treated as cash-settled, so existing code paths work without changes.
4. **Per-instrument error isolation**: If one instrument's P&L calculation fails (e.g., missing price), other instruments still process. The cycle is marked FAILED but partial results are available.
5. **Zero external dependencies**: All DVP code uses only stdlib. The `DeliveryValidator` and `PaymentLocker` are interface-based (same pattern as `payment.Gateway`).
