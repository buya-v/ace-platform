## Summary

### Surveillance Package (`src/compliance-service/internal/surveillance/`)
- `rules.go` ‚Äî Type definitions: `Rule`, `Alert`, `Trade`, `OrderCancel`, `VolumeSnapshot`, `PositionSnapshot`, severity/status enums, and parameter structs for each rule type.
- `detector.go` ‚Äî `TradeDetector` with:
  - `ProcessTrade(trade)` ‚Äî checks trade against all enabled rules
  - `checkWashTrading` ‚Äî detects self-trades and cross-trade wash patterns within configurable time window
  - `checkSpoofing` ‚Äî detects large order cancellations near trade execution (requires `RecordOrderCancel` to feed cancel events)
  - `CheckConcentration` ‚Äî detects position concentration exceeding threshold percentage with tiered severity
  - `CheckUnusualVolume` ‚Äî detects volume exceeding N standard deviations above rolling average
  - `PurgeOldData` ‚Äî memory management for in-process trade/cancel history
  - `MemoryAlertSink` ‚Äî in-memory `AlertSink` implementation for testing
  - `DefaultRules()` ‚Äî returns the 4 default rules matching the V20 migration seed data
  - Exported `ZScore`, `StdDev`, `Mean` math helpers
- `detector_test.go` ‚Äî 33 tests covering all detection rules, edge cases, disabled rules, bad params, multi-rule interaction, and math helpers. 94.2% statement coverage.

## Decisions Made

1. **Concentration and unusual volume checks are not triggered by ProcessTrade** ‚Äî they require aggregate position/volume data not available from individual trades. They are exposed as separate methods (`CheckConcentration`, `CheckUnusualVolume`) callable with snapshot data from an external aggregation layer.
2. **Spoofing detection requires order cancel events** ‚Äî implemented `RecordOrderCancel` to feed cancellation data. Marked as a real interface but depends on matching engine emitting cancel events via Kafka.
3. **Self-trade deduplication** ‚Äî when buyer == seller (direct self-trade), the trade is recorded once per participant in the internal history, preventing double-counting in the wash trading window check.
4. **AlertSink interface** ‚Äî decouples alert emission from detection. Implementations can persist to DB, forward to Kafka, or collect in memory. `MemoryAlertSink` provided for testing.
5. **Zero external dependencies** ‚Äî follows the established zero-dep Go module pattern. All detection rules are pure functions operating on trade data.

## Suggested Follow-ups

- **Spoofing data source**: Need matching engine to emit `order.cancelled` events to populate the cancel history via `RecordOrderCancel`.
- **Concentration aggregation**: Need a periodic job or API that computes `PositionSnapshot` from clearing-engine position data and calls `CheckConcentration`.
- **Volume aggregation**: Need a periodic job that computes `VolumeSnapshot` from market-data-service daily volume data and calls `CheckUnusualVolume`.
- **Alert persistence**: Wire `AlertSink` to the `compliance.surveillance_alerts` table via a PostgreSQL implementation.
