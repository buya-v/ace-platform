# T117 — Distributed session store (Redis) for auth and rate limiting

## Status: DONE

## Summary

Replaced in-memory session storage and rate limiter state with Redis-backed
implementations that gracefully fall back when Redis is unavailable.

### Changes

**Auth Service — Redis Session Store**
- Created `src/auth-service/internal/store/redis_session.go`
  - `RedisSessionStore` wraps an inner store (InMemoryStore or PostgresStore)
  - Session operations use Redis: `session:{id}` keys with SETEX and TTL matching refresh token expiry
  - Per-user session tracking via Redis SET `user_sessions:{user_id}` for efficient `RevokeUserSessions`
  - All non-session methods (users, API keys, PKCE) delegate to the inner store
  - `NewRedisSessionStore(inner, redisURL, sessionTTL)` with connectivity check
  - `NewRedisSessionStoreFromClient(inner, rdb, sessionTTL)` for testing
- Created `src/auth-service/internal/store/redis_session_test.go`
  - 10 tests covering all Redis session operations plus delegation verification
  - Tests skip gracefully when Redis is unavailable
- Updated `src/auth-service/internal/config/config.go` — added `RedisURL` field
- Updated `src/auth-service/cmd/auth-service/main.go` — layers Redis session store when `REDIS_URL` is set, with warning fallback on connection failure
- Updated `go.mod` — added `github.com/redis/go-redis/v9 v9.18.0`

**Gateway — Redis Rate Limiter**
- Created `src/gateway/internal/middleware/redis_ratelimit.go`
  - `RedisRateLimiter` implements sliding-window rate limiting via Redis INCR + EXPIRE
  - Key pattern: `ratelimit:{group}:{client_ip}:{window_bucket}`
  - Falls back to in-memory `RateLimitGroup` if Redis INCR fails per-request
  - `RedisRateLimit()` middleware function mirrors the existing `RateLimit()` API
- Created `src/gateway/internal/middleware/redis_ratelimit_test.go`
  - 4 tests covering allow/deny, per-key isolation, middleware integration, and headers
  - Tests skip gracefully when Redis is unavailable
- Updated `src/gateway/internal/config/config.go` — added `RedisURL` field
- Updated `src/gateway/cmd/gateway/main.go` — uses Redis rate limiter when `REDIS_URL` is set, with warning fallback
- Updated `go.mod` — added `github.com/redis/go-redis/v9 v9.18.0`

### Decisions Made
- Session data is JSON-serialized into Redis (simple, human-readable, no schema migration needed)
- Revocation deletes keys immediately (no soft-delete/mark-as-revoked pattern — deleted = revoked)
- User-sessions tracking uses a Redis SET per user for O(1) member lookup
- Rate limiter uses time-bucketed keys (`unix_timestamp / window_seconds`) so keys auto-expire
- Rate limit per window = `Rate * windowSeconds` (maps token-bucket semantics to fixed-window counting)
- Window defaults to 1 minute if not specified

### Test Results
- All existing auth-service tests pass (43 tests)
- All existing gateway tests pass
- 10 new Redis session tests pass (with Redis) / skip (without Redis)
- 4 new Redis rate limiter tests pass (with Redis) / skip (without Redis)
- 1 unit test for `ParseRedisURL` (always runs, no Redis needed)

### Configuration
- Set `REDIS_URL=redis://localhost:6379/0` to enable Redis for both services
- Redis is already defined in `docker-compose.yml` as `garudax-redis` on port 6379
- Both services gracefully fall back to in-memory if Redis is unreachable at startup
