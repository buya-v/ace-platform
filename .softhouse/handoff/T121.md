# T121 ŌĆö Clearing Engine Default Management Fund and Waterfall

## Status: COMPLETE

## Summary

Implemented CCP default management waterfall per IOSCO/PFMI Principle 4 (Credit Risk) and Principle 7 (Liquidity Risk). The waterfall is a pure calculation engine with no external dependencies, following the zero-dep Go module pattern established by the platform.

## Deliverables

### 1. Migration: `infrastructure/db/migrations/V23__default_fund.sql`
- `clearing.default_fund_contributions` ŌĆö participant contributions (PK: participant_id)
- `clearing.default_events` ŌĆö default event log with JSONB waterfall tracking
- `clearing.ccp_capital` ŌĆö CCP's own capital reserves (skin-in-the-game, additional)

### 2. Package: `src/clearing-engine/internal/defaultmgmt/`
- **`fund.go`** ŌĆö `DefaultFundManager`: thread-safe in-memory store for participant contributions, CCP skin-in-the-game, and CCP additional capital
- **`waterfall.go`** ŌĆö `DefaultWaterfall`: pure-function five-layer waterfall executor
  - Layer 1: Defaulter's margin/collateral
  - Layer 2: Defaulter's default fund contribution
  - Layer 3: CCP skin-in-the-game capital
  - Layer 4: Non-defaulting members' contributions (pro-rata by contribution size)
  - Layer 5: CCP additional capital
  - Returns `WaterfallResult` with amount absorbed per layer and pro-rata details
- **`fund_test.go`** ŌĆö 9 tests for fund manager operations
- **`waterfall_test.go`** ŌĆö 13 tests covering all waterfall scenarios

## Decisions Made
- Waterfall is a pure calculation with no DB side effects ŌĆö DB persistence is deferred to when the store layer needs it
- Pro-rata allocation uses integer arithmetic (raw Decimal values) to avoid floating-point precision issues; last participant gets the remainder to handle rounding
- Sorted participant IDs for deterministic pro-rata output
- Early-exit optimization: waterfall stops processing layers once loss is fully covered
- Fund manager uses RWMutex for concurrent read safety
