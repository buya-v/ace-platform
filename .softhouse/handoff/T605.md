## Design Decisions

### Graceful skip pattern
- `beforeAll` probes `/healthz` with 8s timeout; if unreachable, token stays `null`
- Each test checks `if (!token) { test.skip(); return; }` or checks for `result.status === 0`
- No test will FAIL purely because the gateway is down — all fail-safely with SKIP

### `checkEndpoint` reuse
- All 30+ endpoint tests delegate to `checkEndpoint()` from `api-checks.ts`
- Only tests with field-level contracts (health, margin/calls/stats, book, bot/chat) perform additional assertions beyond status < 500

### `expect.soft()` throughout
- Every assertion uses `expect.soft()` so a failing endpoint does not abort the run
- Final test report shows all failures at once, not just the first

### Token handling
- `getToken()` from `api-checks.ts` handles both `access_token` and `AccessToken` field names
- Token obtained once in `beforeAll`; all 30+ auth tests share the same token

### Instrument constant
- `INSTRUMENT = 'WHT-HRW-2026M07-UB'` declared at top — easy to override for other environments

## Suggested Follow-ups

- **T606 (Reviewer):** Review this spec for coverage gaps and correctness
- **POST endpoints:** Destructive endpoints (halt/resume instrument, mass cancel, trigger margin calc) are intentionally omitted from contract tests to avoid side-effects. These should be tested in a dedicated mutation test spec with proper cleanup.
- **Response body assertions:** Currently most endpoints only assert status < 500. Once the gateway response shapes are stabilised, add `expectedFields` to each check for richer contract validation.
- **`instruments/list` vs `instruments`:** The spec tests both path variants. If one returns 404, investigate whether the gateway routes use `/list` suffix or bare path.
