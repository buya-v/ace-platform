# T115 — Settlement Statements and Regulatory Reporting

## Status: DONE

## Summary

Added reporting capabilities for settlement statements, market summaries, and large trader positions to the gateway service.

## Deliverables

### 1. Database Migration
- `infrastructure/db/migrations/V22__reporting_tables.sql`
- Creates `reporting` schema with three tables:
  - `reporting.daily_statements` — per-participant daily settlement statements with JSONB columns for positions, margin, P&L, and fees
  - `reporting.market_summaries` — per-instrument OHLCV + settlement price + open interest
  - `reporting.large_trader_positions` — reportable positions with % of open interest
- All tables have appropriate unique constraints for upsert behavior

### 2. Reporting Package (`src/gateway/internal/reporting/`)

**store.go** — PostgreSQL data access layer:
- `Store` interface with 7 methods (CRUD for all three report types + trade listing)
- `PgStore` implementation with upsert queries (ON CONFLICT DO UPDATE)
- Follows the same pattern as `internal/fees/store.go`

**generator.go** — Pure report generation functions:
- `GenerateSettlementStatement()` — computes P&L (long/short aware), aggregates fees, calculates net amount
- `GenerateMarketSummary()` — computes OHLCV from chronological trade list
- `GenerateLargeTraderReport()` — identifies positions exceeding threshold % of open interest
- All functions are pure (no side effects), deterministic, and round to 4 decimal places

**handlers.go** — HTTP handlers with 4 routes:
- `GET /api/v1/reports/settlement-statement?date=YYYY-MM-DD` — authenticated participant's own statement
- `GET /api/v1/reports/trade-summary?from=&to=` — authenticated participant's own trades
- `GET /api/v1/admin/reports/market-summary?date=` — admin: all instrument summaries
- `GET /api/v1/admin/reports/large-traders?date=` — admin: reportable positions
- Date validation, auth checks, proper error responses following gateway conventions

**generator_test.go** — 44 tests, 67.8% coverage:
- 7 settlement statement tests (long, short, multiple positions, no positions, all fee types, JSON validation)
- 4 market summary tests (basic, no trades, single trade, decimal precision)
- 7 large trader report tests (above/below threshold, net short, zero/missing OI, multiple instruments, ID format)
- 2 utility tests (date validation, rounding)
- 24 handler tests (success, missing params, invalid dates, not found, no auth, store errors, empty results, route registration, fallback to Sub)

### 3. Gateway Wiring
- Reporting handlers registered in `cmd/gateway/main.go` via `reportingHandlers.RegisterRoutes(rt)`
- All existing gateway tests continue to pass (0 regressions)

## Decisions Made
- Report generation functions are pure and separated from HTTP/DB concerns (easy to test, easy to reuse)
- Settlement statement stores positions/margin/pnl/fees as JSONB for schema flexibility
- Large trader report uses abs(net position) / open interest for threshold comparison
- `PgStore` uses upsert (ON CONFLICT DO UPDATE) so reports can be regenerated safely
- Handlers fall back to `claims.Sub` when `claims.ParticipantID` is empty (consistent with fees package)
