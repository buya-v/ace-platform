# T208 — Docker Compose + Dockerfiles for Bot Services

## Status: DONE

## Summary

Added Docker infrastructure for the admin-bot services to the platform's Docker Compose stack.

## Changes Made

1. **Created `src/admin-bot/mcp-server/Dockerfile`** — node:22-slim based, compiles TypeScript and runs `dist/index.js`. This image is for building the MCP server; at runtime it is mounted as a read-only volume into the orchestrator container (stdio-based, spawned as child process).

2. **Created `src/admin-bot/orchestrator/Dockerfile`** — node:22-slim based, compiles TypeScript, exposes port 3100, runs `dist/index.js`. This is the HTTP server container.

3. **Updated `docker-compose.yml`**:
   - Added `admin-bot-orchestrator` service under a new "Bot Services" section after Frontend UIs
   - Service depends on `gateway` (condition: service_started)
   - Mounts `./src/admin-bot/mcp-server:/mcp:ro` so orchestrator can spawn MCP server as child process
   - Exposes port 3100
   - Environment: PORT, GARUDAX_GATEWAY_URL, BOT credentials, OPENAI_API_KEY, MCP_SERVER_PATH, BOT_DAILY_REPORT_HOUR
   - Added `BOT_ORCHESTRATOR_URL` env var to gateway service

4. **Updated `.env.example`**:
   - Added `BOT_ADMIN_EMAIL`, `BOT_ADMIN_PASSWORD`, `OPENAI_API_KEY` variables under new "Admin Bot" section

## Design Decisions

- **MCP server is NOT a separate container** — it's stdio-based and spawned by the orchestrator as a child process. The MCP server source is volume-mounted at `/mcp` in the orchestrator container.
- **Used `service_started` instead of `service_healthy`** for gateway dependency — the orchestrator can handle gateway unavailability gracefully and retry.
- **Added `restart: unless-stopped`** to the orchestrator for resilience.
