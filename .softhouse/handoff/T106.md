# T106 — Settlement Engine PostgreSQL Persistence

## Status: DONE

## Summary

Replaced settlement-engine in-memory stores with PostgreSQL-backed implementations while maintaining full backward compatibility via a `valuation.PriceStore` interface with in-memory fallback.

## Changes Made

### 1. Migration: `infrastructure/db/migrations/V14__settlement_engine_tables.sql`
- Created `settlement` schema with three tables:
  - `settlement.cycles` — settlement cycle state (PK: `id`, indexed on `status`, `settle_date`)
  - `settlement.instructions` — payment instructions (PK: `id`, FK to cycles, indexed on `cycle_id`, `participant_id`, `status`)
  - `settlement.prices` — settlement prices with composite PK (`instrument_id`, `price_date`), indexed on both columns

### 2. New File: `src/settlement-engine/internal/store/postgres.go`
- `PostgresCycleStore` — SaveCycle (upsert), GetCycle, GetAllCycles
- `PostgresInstructionStore` — SaveInstruction (upsert), GetByCycleID, GetByParticipantID, GetByStatus
- `PostgresPriceStore` — SetSettlementPrice, GetSettlementPrice, HasPreviousPrice
- `OpenDB()` — connection helper using `pgx/v5/stdlib` driver
- Status string parsers: `parseCycleStatus`, `parsePayDirection`, `parseInstructionStatus`
- All stores use parameterized SQL ($1, $2, ...) for injection safety

### 3. New File: `src/settlement-engine/internal/store/postgres_test.go`
- 10 test functions covering: decimal serialization, store construction, status round-trips, field mapping for cycles/instructions/prices

### 4. Interface: `valuation.PriceStore`
- Added `PriceStore` interface to `internal/valuation/valuation.go`
- Both `valuation.Store` (in-memory) and `store.PostgresPriceStore` implement it
- `SetSettlementPrice(instrumentID, date, price)` — no error return (matches existing API)
- `GetSettlementPrice(instrumentID, date)` — returns `(SettlementPrice, error)`

### 5. Engine + PnL refactored to use interface
- `engine.Engine.priceStore` changed from `*valuation.Store` to `valuation.PriceStore`
- `pnl.Calculator.priceStore` changed from `*valuation.Store` to `valuation.PriceStore`
- `engine.NewEngine()` and `pnl.NewCalculator()` accept `valuation.PriceStore`
- `engine.GetPriceStore()` returns `valuation.PriceStore`

### 6. Updated `cmd/settlement-engine/main.go`
- When `DB_HOST` env var is set: connects to PostgreSQL, uses `PostgresPriceStore`
- Otherwise: uses `valuation.NewStore()` (in-memory, same as before)
- Supports env vars: `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`, `DB_SSLMODE`

### 7. Updated `go.mod`
- Added `github.com/jackc/pgx/v5 v5.7.4` dependency (same version as clearing-engine)

## Decisions Made
- PostgresPriceStore.SetSettlementPrice does not return error (matches interface contract); DB errors are logged to stdout. This keeps the interface compatible with the in-memory store and all existing callers.
- CycleStore and InstructionStore are created but not yet wired into the engine (the engine currently stores cycles in a `map[string]*SettlementCycle` in memory). Wiring these requires deeper engine refactoring and is left for a follow-up task.
- Status values stored as strings in SQL (e.g., "COMPLETED", "PAY_IN") for readability, with round-trip parsers.

## Test Results
- All existing tests pass (0 failures)
- New store tests pass (10 tests)
- Build compiles cleanly

## Suggested Follow-ups
- Wire `PostgresCycleStore` and `PostgresInstructionStore` into the engine (requires engine refactoring to persist cycles/instructions after each step)
- Add integration tests that run against a real PostgreSQL instance
- Consider adding a `CycleStore` interface to the engine to fully decouple cycle persistence
