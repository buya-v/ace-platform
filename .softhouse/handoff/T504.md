## Segment Classification Logic

`parseMessageSegments(reply: string): Segment[]` splits on `\n` and classifies each line:

| Prefix/Pattern | Segment type |
|---|---|
| `✅ ...` | success (green left border) |
| `❌ ...` | error (red left border) |
| `⚠️ ...` or `⚠ ...` | warning (yellow left border) |
| `• ...` or `- text` | bullet (indented, marker) |
| `**text**` or `# text` | heading (bold, 600 weight) |
| `Key: Value` | kv (key in --text-muted, value in --text-primary) |
| anything else | text (default color) |

Empty lines are preserved as `text` segments with empty `text` field to maintain layout spacing.

## Decisions Made

1. **Segment rendering**: `kv` segments use a flex row with separate `<span>` for key and value to allow independent color styling via `kvKey`/`kvValue` CSS classes.
2. **Action navigation**: Link actions (`type === 'link'`) set `window.location.pathname = action.payload`, enabling SPA-compatible navigation without a full page reload. Non-link actions are ignored in the card (can be extended).
3. **Emoji stripping for ⚠️**: Used a regex `replace(/^⚠️?\s*/, '')` to handle both the compound `⚠️` (U+26A0 + U+FE0F) and the bare `⚠` form reliably across environments.
4. **BotChatPanel refactor**: User messages still render as plain text; only bot messages use `BotMessageCard`. User-role `msg.actions` still render as the existing `actionChip` chips (they use `sendMessage(action.label)` behavior, which is appropriate for quick-reply actions vs navigation).
5. **normalizeAction**: Maps `{url}` or `{target}` raw gateway fields to the canonical `payload` field, making the UI agnostic to which field the API uses.
