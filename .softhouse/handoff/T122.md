# T122 — Margin Engine SPAN-like Margining Model with Scenarios

## Status: DONE

## Summary

Replaced the margin engine's implicit scenario generation with a SPAN-like model that supports pre-computed risk arrays and inter-commodity spread credits. The existing percentage-based calculation is preserved as a fallback when risk arrays are not configured for an instrument.

## What was built

### 1. SQL Migration — `infrastructure/db/migrations/V24__margin_scenarios.sql`
- `margin.risk_arrays` table: stores pre-computed P&L impacts per instrument per scenario (16 standard scenarios)
- `margin.spread_credits` table: defines margin relief for offsetting positions in correlated instruments
- Seed data: 16 wheat (WHT-HRW-2026M07-UB) scenarios covering price shifts of +/-3%, +/-6%, +/-10%, +/-15% with vol shifts of +/-25%

### 2. SPAN Scanner Package — `src/margin-engine/internal/span/`
- **`scanner.go`** — `SPANScanner` type:
  - `LoadRiskArrays()`: loads pre-computed risk array entries indexed by instrument+scenario
  - `HasRiskArray()`: checks if an instrument has risk arrays (determines SPAN vs fallback)
  - `ScanPortfolio()`: evaluates all positions simultaneously across all scenarios, computing portfolio-level P&L per scenario, returns worst-case loss
  - `ScanSinglePosition()`: convenience wrapper for single-position scanning
  - Pure function: positions x scenarios matrix multiplication
- **`spread.go`** — `SpreadCreditor` type:
  - `LoadCredits()`: loads spread credit definitions
  - `ApplySpreadCredits()`: calculates margin reduction for offsetting positions
  - Handles standard spreads (long A / short B) and reverse spreads
  - Credit is proportional to the overlapping quantity (min of abs values)

### 3. Calculator Integration — `src/margin-engine/internal/margin/calculator.go`
- Added `SetSPANScanner()` and `SetSpreadCreditor()` methods
- `Calculate()` now checks for SPAN risk arrays first, falls back to existing scanner
- `CalculatePortfolio()` applies spread credits when configured and >1 instrument present
- Fully backward compatible: without calling SetSPANScanner, behavior is identical to before

### 4. Tests
- `span/scanner_test.go`: 19 tests covering scanner and spread creditor
  - Single/multi-contract scanning, portfolio-level scanning
  - Offset verification (portfolio risk < sum of individual risks)
  - Missing instrument fallback, risk array loading/replacement
  - Spread credits: offsetting, same-direction, reverse, partial overlap, missing legs
  - Utility function tests (sortInts, minAbs)
- `margin/calculator_test.go`: 6 new integration tests added to existing suite
  - SPAN scanner integration, fallback for missing risk arrays
  - Backward compatibility (empty SPAN matches original)
  - Spread credit reduction verification
  - Single-instrument no-credit verification

- All existing tests continue to pass unchanged

## Decisions Made
1. **SPAN as optional overlay**: Rather than replacing the existing scanner, the SPAN scanner is an opt-in alternative. This allows gradual migration per instrument.
2. **Risk arrays indexed by instrument+scenario**: O(1) lookup during scanning, matching the SQL schema's composite primary key.
3. **Portfolio-level scanning**: `ScanPortfolio` evaluates all positions simultaneously (the correct SPAN approach) rather than summing individual scans. This naturally captures diversification benefits.
4. **Spread credit formula**: `(longMargin + shortMargin) * min(|longQty|, |shortQty|) * creditPct * 0.01`. Per-instrument margin is the scan risk for that instrument (not per-contract, total).
5. **Zero external dependencies**: The span package uses only the existing types package, consistent with the zero-dep pattern.
