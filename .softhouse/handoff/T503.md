# T503 Handoff ‚Äî Bot gateway: forward user JWT for audited action execution

## Status: DONE

## Summary

Verified and enhanced the bot gateway to ensure every write action is attributed to the authenticated user.

### What was already working
- `handler.go` Chat method: token extraction at lines 97-100 was already correct ‚Äî Bearer token extracted from Authorization header and passed to `executor.Execute(userToken)`.
- `executor.go`: ALL `doRequest` calls already passed `userToken`. The `doRequest` helper at line 609-611 correctly sets `Authorization: Bearer <token>` when token is non-empty. No missing calls were found.

### What was added

Added two new methods to `ActionExecutor` in `executor.go`:

1. **`fetchUserEmail(token string) string`** ‚Äî calls `GET /api/v1/auth/me` with the user's token, parses the response for the user's email. Handles both `{"data":{"email":"..."}}` and flat `{"email":"..."}` response shapes. Returns empty string on failure (graceful degradation).

2. **`withAttribution(reply, token string) string`** ‚Äî fetches the user's email and appends `\n\nExecuted by: <email>` to the reply. If the email cannot be fetched (server error, empty token), returns the reply unchanged.

Applied `withAttribution` to all 17 write operations:
- Order placement (buy/sell), modify order, cancel order
- Bust trade
- Disable participant
- Set circuit breaker
- Suspend trader, reinstate trader, file SAR, resolve alert
- Halt instrument, resume instrument
- Approve KYC, reject KYC
- Mass cancel
- Set risk/order limit
- Trigger settlement cycle

Read-only operations (profile, orders list, market data, health, netting, positions, etc.) intentionally left without attribution ‚Äî they are read-only and don't require audit attribution.

### Tests added (6 new tests in `handler_test.go`)
- `TestFetchUserEmail_EmptyToken` ‚Äî no request made when token empty
- `TestFetchUserEmail_ServerReturnsEmail` ‚Äî nested `data.email` field parsed correctly, Authorization header forwarded
- `TestFetchUserEmail_FlatEmailField` ‚Äî flat `email` field parsed as fallback
- `TestFetchUserEmail_ServerError` ‚Äî returns empty string gracefully on 5xx
- `TestWithAttribution_AppendsEmail` ‚Äî appends `\n\nExecuted by: <email>` when email resolves
- `TestWithAttribution_NoEmailFallsThrough` ‚Äî returns reply unchanged when email fetch fails

## Decisions Made
- Attribution is best-effort: if `/auth/me` is unavailable or returns an error, the write operation still succeeds and the reply is returned unchanged. This avoids blocking admin actions due to a secondary lookup failure.
- Both `{"data":{"email":""}}` (nested) and `{"email":""}` (flat) response shapes are handled for compatibility with the auth service's response format.
- Attribution is applied only on successful write operations (HTTP 2xx). Failed operations return error messages without attribution since no action was executed.
