# T109 ‚Äî Warehouse Service PostgreSQL Persistence

## Status: DONE

## Summary

Wired PostgreSQL persistence to the warehouse service by:

1. **Extracted `DataStore` interface** (`internal/store/iface.go`) ‚Äî defines all 17 store operations (Facility CRUD, Inspection CRUD, Receipt lifecycle, Delivery lifecycle, Inventory queries, Receipt events). Both `Store` (in-memory) and `PostgresStore` implement this interface with compile-time checks.

2. **Created `PostgresStore`** (`internal/store/postgres.go`) ‚Äî full PostgreSQL implementation using `database/sql` + `github.com/jackc/pgx/v5/stdlib` driver. All operations map to the V9 warehouse schema:
   - Facilities: CRUD with `facility_commodities` join table for approved commodity IDs
   - Inspections: create/get/update with all quality metric columns (NUMERIC ‚Üí Decimal)
   - Receipts: transactional create (lot uniqueness check, facility status check, capacity check), transfer, cancel, pledge, release ‚Äî all within transactions with `FOR UPDATE` row locking
   - Deliveries: transactional create (marks receipt as DELIVERY_PENDING), complete (success ‚Üí DELIVERED with inventory withdrawal, failure ‚Üí reverts to ACTIVE)
   - Inventory: reads from `warehouse.current_inventory` view (double-entry aggregation)
   - Facility capacity: reads from `warehouse.facility_utilization` view with fallback
   - Receipt events: ordered audit trail from `warehouse.receipt_events`

3. **Updated service layer** (`internal/service/service.go`) ‚Äî changed `store` field from `*store.Store` to `store.DataStore` interface. The `New()` constructor now accepts `store.DataStore`.

4. **Updated main.go** ‚Äî checks `DB_HOST` env var. When set, opens PostgreSQL connection with configurable `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`, `DB_SSLMODE`, `DB_SCHEMA`. Falls back to in-memory store when `DB_HOST` is not set.

5. **Added pgx dependency** ‚Äî `github.com/jackc/pgx/v5` added to `go.mod`.

## Decisions Made

- Used `database/sql` with pgx stdlib adapter (not pgx native) for standard library compatibility
- Receipt numbers come from DB-generated `receipt_number` column (via `DEFAULT` in schema) rather than the in-memory `EWR-` prefix pattern
- Decimal values stored as `NUMERIC(18,4)` strings, parsed back through `types.ParseDecimal`
- All multi-step operations (receipt create, transfer, cancel, pledge, release, delivery create/complete) wrapped in transactions with `FOR UPDATE` row locking
- Metadata in receipt events stored as JSONB (cancel reason wrapped in `{"reason":"..."}`)
- `GetFacilityCapacity` reads from `warehouse.facility_utilization` view with fallback to direct query if view returns no rows
