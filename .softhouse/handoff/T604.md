## Design Decisions

1. **`expect.soft()` throughout** — all assertions are soft so one failure does not abort
   the test; all 12 assertions per test are always checked.

2. **`isPortalAvailable` skip pattern** — `beforeEach` calls `isPortalAvailable(page)`
   and skips the entire describe block with a descriptive reason if the portal is
   unreachable. Follows the established UAT graceful-skip pattern.

3. **Fresh bot per test** — `beforeEach` calls `loginAsAdmin` then `openBotChat` so every
   test starts with a clean chat session. This avoids conversation history from prior
   messages polluting the response pattern matching.

4. **Screenshots for evidence** — tests 1, 2, 4, 8, 10, 12 call `captureScreenshot` after
   the bot reply, saving timestamped PNGs to `uat/screenshots/`.

5. **Permissive regex patterns** — bot reply patterns are broad (e.g.,
   `/margin|active|calls|shortfall|no.*active/i`) to accept both populated and
   empty-state responses. The test validates the bot responded coherently, not that
   specific data values are present.

6. **No navigation before bot open** — the bot widget is available on all dashboard pages,
   so no `navigateTo()` call is needed. `loginAsAdmin` lands on `/dashboard` which is
   sufficient.
