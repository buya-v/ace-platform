## Architecture Decisions

### 1. Internal test package (`package server`, not `package server_test`)

Handler tests are in `package server` (same package as production code). This is required because `server.registerRoutes` and all handler methods are unexported. External test packages (`package server_test`) cannot call unexported methods, and the Go tool does not make symbols from `_test.go` files in `package server` available to `package server_test`.

The correct pattern: internal `_test.go` files + direct access to `srv.registerRoutes(mux)`.

### 2. `httptest.Server` with production route table

`newTestServer(t)` creates fresh in-memory stores, constructs a `*Server`, calls `srv.registerRoutes(mux)` (the production path), and wraps it in `httptest.NewServer`. This exercises the full HTTP dispatch chain (mux → dispatcher → handler → store).

### 3. Pre-seeded store for terminal-status scenarios

`TestCancelOrder_FilledOrderRejected` cannot be set up via the HTTP API alone (no endpoint creates FILLED orders). `newServerWithFilledOrder(t)` seeds an `InMemoryOrderStore` directly before constructing the test server — the correct approach for boundary state testing.

### 4. Atomic ticker counter for uniqueness

`orderTickerSeq` (`atomic.Int64`) generates unique ticker symbols across tests to prevent ID conflicts when tests run in parallel under `-race`.

### 5. `handleCreateInstrument` uses 400, not 422

The instrument handler returns `http.StatusBadRequest` (400) for validation errors, not 422 as stated in the task spec. Tests match the actual implementation. This is documented as a follow-up.
