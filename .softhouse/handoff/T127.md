# T127 — Load Test: Matching Engine Throughput and Latency Benchmarks

## Status: DONE

## Summary

Created a comprehensive load test suite in `tests/load/` with Go benchmarks covering the matching engine and clearing engine. The suite uses self-contained type/logic copies (following the zero-dep pattern) since Go's `internal` package restriction prevents cross-module imports.

## Files Created

- `tests/load/go.mod` — standalone Go module
- `tests/load/engine/types.go` — duplicated Decimal, Order, Trade, ClearingObligation types
- `tests/load/engine/orderbook.go` — duplicated OrderBook with price-time priority CLOB
- `tests/load/engine/clearing.go` — duplicated NovationService and NettingService
- `tests/load/matching_bench_test.go` — 8 matching engine benchmarks
- `tests/load/clearing_bench_test.go` — 5 clearing engine benchmarks
- `tests/load/report_test.go` — summary report test with performance target verification

## Benchmarks Implemented

### Matching Engine (matching_bench_test.go)

| Benchmark | Description | Result |
|---|---|---|
| BenchmarkSingleInstrumentThroughput | Alternating buy/sell at same price | ~271K ops/sec |
| BenchmarkMatchLatency (depth 1-1000) | Per-match latency with percentiles | p99: 7us (depth 1) to 3.4ms (depth 1000) |
| BenchmarkDeepBookStress | 10K resting orders, single market sweep | ~44ms for 10K trades |
| BenchmarkBurstOrders | 1000 rapid orders with duplicate/consistency checks | ~2.1ms per 1000 orders |
| BenchmarkCancelStorm (100-5000) | Insert then cancel all orders | ~2.6us/cancel (100) to ~2.6ms/cancel (5000) |
| BenchmarkConcurrentInstruments | 5 instruments in parallel goroutines | ~2.7ms for 1000 trades across 5 books |
| BenchmarkMultiPriceLevelMatching | Sweep across 10-200 price levels | 13us (10 levels) to 245us (200 levels) |
| BenchmarkOrderPlacementNoMatch | Pure insertion (no matching) | ~1.4us/order |

### Clearing Engine (clearing_bench_test.go)

| Benchmark | Description | Result |
|---|---|---|
| BenchmarkNovation | Single trade novation throughput | ~754K ops/sec |
| BenchmarkNovationBatch (100-2000) | Batch novation | 127us (100) to 2.9ms (2000) |
| BenchmarkNetting (100-10K) | Multilateral netting calculation | 13us (100) to 460us (10K) |
| BenchmarkNettingEfficiency | Verify 100% netting efficiency | ~88us for 2000 obligations |
| BenchmarkFullPipeline (100-2000) | Match + novate + net end-to-end | ~4.4ms per 1000 trades |

## Performance Target Verification

- **>1000 orders/sec per instrument**: PASS — measured ~271,000 ops/sec (271x target)
- **<10ms p99 match latency**: PASS — p99 = 6.9us at depth 1, 113us at depth 10, 463us at depth 100, 3.4ms at depth 1000
- **Zero lost or duplicate orders**: PASS — BenchmarkBurstOrders verifies no duplicate trade/order IDs

## Decisions Made

1. **Self-contained module**: Used type duplication instead of `replace` directives because Go's `internal` package restriction prevents any external module from importing `internal/orderbook`. This follows the established zero-dep pattern (Decimal type duplication across all 9 services).

2. **Book reset in BenchmarkOrderPlacementNoMatch**: Reset the book every 1000 orders to avoid O(n^2) degradation from unbounded book growth during benchmarking.

3. **Percentile reporting**: BenchmarkMatchLatency reports p50/p95/p99 via `b.ReportMetric()` for machine-readable output plus `b.Logf()` for human-readable summaries.

## How to Run

```bash
# Full benchmark suite (takes ~2 minutes)
cd tests/load && go test -bench=. -benchtime=2s -benchmem -timeout 300s

# Summary report with performance targets
cd tests/load && go test -run TestBenchmarkSummaryReport -v -timeout 120s

# Single benchmark
cd tests/load && go test -bench=BenchmarkSingleInstrumentThroughput -benchtime=5s -benchmem
```
