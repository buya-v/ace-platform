## Known Issues / Non-Blocking

1. **`covdata` tool warning in securities-service** — `go: no such tool "covdata"` appears for the
   `cmd/securities-service` package when running with `-cover`. This is a Go 1.22 toolchain issue
   where `go test -cover` on a main package emits a covdata merge step that requires a separate
   binary. All 5 package-level tests pass without issue. Not introduced by Phase 0.6 changes.

2. **DLQ prefix hardcoded per service** — Each service's `consumer.go` has `ace-commodities.dlq.*`
   hardcoded. Correct for the current single-tenant ace-commodities deployment but will need
   parameterisation before mse-equities consumers are wired (see T3 handoff Follow-ups).

3. **Metric key prefix is string-based, not Prometheus label** — Observability uses
   `{tenant_id}_GET_200` string prefix rather than a proper multi-dimensional label. Acceptable
   stopgap; should be replaced with Prometheus label dimensions when the metrics layer is upgraded.

## Suggested Follow-ups

- **Phase 0.7** — Dynamic tenant registry: replace static `["ace-commodities", "mse-equities"]`
  whitelist with a `TenantStore` interface backed by `platform.tenants` table query.
- **DLQ parameterisation** — Accept `tenant_id` in `ConsumerConfig` so DLQ prefix is derived at
  runtime, not hardcoded. Required before any second tenant's Kafka consumers are deployed.
- **Kafka consumer tenant injection** — Add `tenant.WithTenant(ctx, ...)` in
  `ChannelConsumer.processRecord` so downstream handlers have tenant context matching the topic
  prefix, enabling tenant-aware DB routing in handlers.
- **V30 migration deployment gate** — SQL schema is `ace_*` in code (T2). Ensure V30 Flyway
  migration runs before deploying updated service images in any environment.
- **`participants.` schema** — Not found in any Go source file. If a participants service is added
  (Phase 0.7 control plane), it must use `ace_participants.*` from the outset.
