## Decisions Made

1. **Whitebox test package for server** — `handlers_test.go` declares `package server` (not
   `package server_test`) to access `registerRoutes`, `handleTenants`, `handleTenant`, `healthz`,
   and `readyz`. These are unexported methods that cannot be reached from a blackbox test without
   modifying the production code. This is the standard Go pattern for whitebox testing.

2. **Status validation is the handler's responsibility, not the store's** — `TestTenantStore_UpdateStatus_InvalidStatus`
   confirms the store accepts any string for status. The handler enforces the `validStatuses` map.
   This matches the production code — `UpdateStatus` in `store.go` has no validation guard.

3. **Minimal fake SQL driver (zero deps)** — Implemented `database/sql/driver` interfaces directly
   in the test file to exercise `createSchemas`. No external SQLite or mock library required,
   preserving the zero-dependency module constraint. The fake driver always succeeds (Exec returns
   `fakeResult{}`), which is sufficient to cover the happy-path `createSchemas` loop.

4. **HTTP status codes match implementation, not task spec** — The task description referred to
   "422" for validation failures. The actual handler returns `http.StatusBadRequest` (400) for
   `VALIDATION_ERROR` and `INVALID_STATUS`. Tests assert 400 to match the production implementation.

5. **`httptest.NewServer` + `registerRoutes`** — `newHandlerTestServer` builds a real `*http.ServeMux`
   via `srv.registerRoutes(mux)` and wraps it in `httptest.NewServer`. This exercises the full
   HTTP dispatch path including the path-prefix routing for `/platform/v1/tenants/` and the
   `/status` sub-resource suffix detection.
