## Pattern Analysis

All configuration follows established codebase conventions:

**1. Service Naming**
- `securities-service` suffix aligns with supporting services (auth-service, compliance-service, market-data-service, warehouse-service)
- Distinct from core trading pipeline `-engine` services (matching-engine, clearing-engine, margin-engine, settlement-engine)

**2. Port Allocation (gRPC / Health HTTP)**
- Follows `5005x` for gRPC, `808x` for health HTTP convention
- Port assignment chain:
  - matching-engine: 50051 / 8081
  - clearing-engine: 50052 / 8082
  - margin-engine: 50053 / 8083
  - settlement-engine: 50054 / 8084
  - auth-service: 50055 / 8085 ← (8085 is taken)
  - compliance-service: 50056 / 8086
  - market-data-service: 50057 / 8087
  - warehouse-service: 50058 / 8088

- HTTP API: 8085:8085 (shared with auth-service, differentiated by hostname/virtual-host in production gateway)

- Common environment (`*common-env`): KAFKA_BROKERS, BIND_ADDRESS

## Decisions Made

1. **Port 9085 for health endpoint:** Auth-service uses 8085 for HTTP API. Since both services will run in the same Docker Compose stack, the health endpoint is on 9085 (metrics port convention). In production, the gateway (reverse proxy) would route based on request path or SNI, not just port.

2. **DB credentials pattern:** Used explicit `DB_*` environment variables to match auth-service (T40), not the stripped pattern of some other services. This provides clear service-level database isolation and is required for the implementation.

3. **GRPC_PORT 50059:** Next available gRPC port following the strict sequence. Will enable service-to-service gRPC calls from gateway or other engines once wired up.

4. **Restart: unless-stopped:** Matches the admin-bot-orchestrator pattern (the only other non-engine service with explicit restart policy). Provides resilience during development/testing.
