# T204 — Bot Chat API — Gateway Proxy to Orchestrator

## Status: DONE

## Summary
Added bot chat endpoints to the Go gateway that proxy to the orchestrator service, with a built-in fallback mode for MVP use without an orchestrator.

## Files Created
- `src/gateway/internal/bot/handler.go` — BotHandlers struct, POST /api/v1/bot/chat and GET /api/v1/bot/suggestions endpoints
- `src/gateway/internal/bot/suggestions.go` — Page-specific suggestion data (dashboard, surveillance, margin, settlement, tickets, participants)
- `src/gateway/internal/bot/bridge.go` — Orchestrator proxy bridge (HTTP POST to BOT_ORCHESTRATOR_URL/chat, 30s timeout)
- `src/gateway/internal/bot/fallback.go` — Built-in keyword responses for MVP mode (health, alerts, margin, tickets, settlement, trading, KYC, help)
- `src/gateway/internal/bot/handler_test.go` — 32 tests, 96.7% coverage

## Files Modified
- `src/gateway/cmd/gateway/main.go` — Wired bot handlers with BOT_ORCHESTRATOR_URL env var

## Decisions
- Both endpoints require admin or exchange_admin role (enforced in handler, not middleware, following tickets pattern)
- Orchestrator bridge returns error on failure, handler falls back to keyword responses automatically
- Chat response shape: `{ "data": { "reply": string, "actions": [], "suggestions": [] } }` — matches gateway convention
- BOT_ORCHESTRATOR_URL empty = fallback mode, no orchestrator needed for MVP

## Test Results
- 32 tests pass (including subtests): auth checks, role checks, all keyword responses, orchestrator proxy success, orchestrator error fallback, orchestrator unreachable fallback, all page suggestions
- All existing gateway tests still pass (16 packages)
- 96.7% statement coverage on bot package
