## Decisions Made

1. **Zero external dependencies** — maintained stdlib-only pattern (no prometheus/client_golang).
2. **`# HELP` and `# TYPE` always emitted** — even when no data recorded yet, so Prometheus scraper sees consistent metadata.
3. **Error counter key matches request counter key** — both use `METHOD_STATUS` so they can be joined in PromQL.
4. **`metricsRecorder.Write()` added** — the existing `metricsRecorder` only overrode `WriteHeader`. Added `Write()` to capture response body size. This is consistent with the `statusRecorder` in middleware.go which already does this for tracing.
5. **`MetricsServer` uses `net/http` mux** — serves both `/metrics` (text) and `/metrics.json` on the same port. `Start()` launches a goroutine; `Shutdown(ctx)` is graceful.
6. **Size histogram buckets** — 256B to 1MB range covers typical API responses. Same cumulative bucket logic as latency histogram.
7. **Content-Type** — changed from `text/plain; charset=utf-8` to `text/plain; version=0.0.4; charset=utf-8` to match Prometheus exposition format spec.

**gateway** is the only service with full metrics wiring. It uses its own local copy of the observability package (not `src/shared`), which is consistent with the zero-dependency-per-service pattern.

**matching-engine** has a `MetricsMiddleware()` definition in its own observability package but does not call it in `main.go`. The health server in matching-engine does not mount `/metrics`.

**All other services** (clearing, margin, settlement, auth, compliance, market-data, warehouse) have no observability/metrics code at all.

## Suggested Follow-ups

- **Wire MetricsMiddleware in matching-engine**: `srv.StartHealthServer()` in `internal/server/server.go` — add `m.MetricsHandler()` to the health mux (same pattern as gateway).
- **Add MetricsMiddleware to clearing/margin/settlement engines**: Each has a `StartHealthServer()` — mount `/metrics` on it. Use the shared library or duplicate the pattern.
- **Consolidate shared vs per-service observability**: gateway and matching-engine have local copies of the same metrics code. Consider whether to migrate to `src/shared` (adds a module dependency) or keep copies (current pattern).
- **Consider a `MetricsServer` in services that don't have HTTP at all**: gRPC-only services (auth, compliance) could use `MetricsServer` to expose `/metrics` on a separate port (e.g., 9090) without an HTTP server for business traffic.
