## Structural Verification

### Port Collision: 8089 (not 8085)?

**RESOLVED. No collision.**

- `securities-service` uses `API_PORT: 8089`, exposed as `"8089:8089"` in `docker-compose.yml`
- Port 8085 belongs to `auth-service` — a different service entirely
- `SecuritiesServiceAddr` in gateway config defaults to `localhost:50059` (gRPC), and `main.go` derives the HTTP address as `securities-service:8089`

### DB Store Present?

**YES.** `src/securities-service/internal/store/pg_store.go` exists. It provides the `PgInstrumentStore` and `PgOrderStore` implementations backed by PostgreSQL. The in-memory `store.go` is used for tests; `pg_store.go` is the production path.

### Gateway Config: `SecuritiesServiceAddr` present?

**YES.** `src/gateway/internal/config/config.go` line 39:

```go
SecuritiesServiceAddr: envStr("SECURITIES_SERVICE_ADDR", "localhost:50059"),
```

And `docker-compose.yml` passes `SECURITIES_SERVICE_ADDR: securities-service:50059` to the gateway container.

### Gateway Routes Wired?

**YES.** `src/gateway/internal/handler/routes.go` registers 9 securities routes:

```
GET    /api/v1/securities/instruments        → ListSecuritiesInstruments
POST   /api/v1/securities/instruments        → CreateSecuritiesInstrument
GET    /api/v1/securities/instruments/{id}   → GetSecuritiesInstrument
PATCH  /api/v1/securities/instruments/{id}   → UpdateSecuritiesInstrument
PUT    /api/v1/securities/instruments/{id}   → UpdateSecuritiesInstrument
GET    /api/v1/securities/orders             → ListSecuritiesOrders
POST   /api/v1/securities/orders             → SubmitSecuritiesOrder
GET    /api/v1/securities/orders/{id}        → GetSecuritiesOrder
DELETE /api/v1/securities/orders/{id}        → CancelSecuritiesOrder
```

All handlers in `handler.go` forward via `h.forward(w, r, "securities-service", "SecuritiesService/<Op>")`. The gateway's `main.go` maps `"securities-service"` to `http://securities-service:8089`.

Admin health endpoint (`GET /api/v1/admin/health`) includes `securities-service` at port 8089 in its service list.

## Blockers / Issues Found

1. **`go: no such tool "covdata"`** — The installed Go version does not include the `covdata` binary. This only affects coverage aggregation for the `cmd/` binary package; all library packages test and report coverage normally. Not a blocker.

2. **`pg_store.go` not covered by tests** — PostgreSQL store requires a live database. Store coverage (33.3%) reflects in-memory store only. This is acceptable for a dev-phase test suite but should be addressed before production readiness.

3. **Securities routes not explicitly in `handler_test.go`** — `TestRouteRegistration` checks `len(routes) >= 48` but does not assert the 9 securities routes by name. The routes are verified to exist via code inspection; a future task should add explicit securities route assertions to the test.
