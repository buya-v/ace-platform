## Implementation Decisions

### Markdown rendering (no dependencies)
Implemented `renderMarkdown()` as a string-replacement pipeline:
1. Fenced code blocks (``` ... ```) → `<pre><code>`
2. Inline code (`` ` ` ``) → `<code>`
3. Bold (`**text**`) → `<strong>`
4. Italic (`*text*`) → `<em>`
5. Bullet lines (`- item`) → accumulated `<ul><li>` groups

Bot messages render via `dangerouslySetInnerHTML`. User messages are plain text (XSS prevention).

### UX
- Floating button (56px round, gradient blue→purple) hidden when panel is open
- Panel: 400×500 px, fixed bottom-right, slide-up animation on open
- Typing indicator: three bouncing dots while awaiting API response
- Enter sends, Shift+Enter inserts newline; textarea auto-focuses on open
- Auto-scroll to latest message after each update
- Error rendered as a bot bubble when fetch fails

### API contract
`POST /api/architect/chat` with `{ message: string, history: {role, content}[] }` — expects `{ response: string }`.

### Responsive
On screens < 768 px: panel covers full viewport (width/height 100%, border-radius 0).

- No source code changes were required — pure build-and-deploy task.
- Login response uses PascalCase keys (`AccessToken`, `RefreshToken`) rather than snake_case — frontend and e2e tests must handle this.

## Key Decisions

1. **No `GetByName` on `RoleStore` interface** — The interface has only `Get(id)`, `List()`, `Create`, `Update`, `Delete`. `TestRoleStore_GetByName` iterates `List()` results, which is how application code would do it. This tests the interface as actually implemented.

2. **`ADMIN_FULL` as enumeration, not wildcard** — `privilege.go` uses exact string matching. There is no wildcard `ADMIN_FULL` token. `TestHasPermission_AdminFull` tests the pattern of putting all platform permissions on an ADMIN role, documenting the intended usage.

3. **Separate `newRoleTestServer` helper** — Rather than extending `newTestServerWithStores` (which would require changing its signature and all callers), a new role-specific helper was created. This avoids breaking the 80+ existing handler tests.

4. **`TestRoleHandlers_Unconfigured`** — Explicitly passes `nil` for `privilegeEngine` and `roleStore` to verify the `s.roleStore == nil` guard in each handler returns 503 (not a panic).

---

## Suggested Follow-ups

- **Silent role fetch errors**: `privilege.go` ignores errors from `roleStore.Get()` other than checking `rerr == nil`. If the store returns an unexpected error (e.g., timeout), the engine silently denies via the fallthrough path. Consider returning the store error explicitly for observability.
- **PermissionCheck middleware integration**: No test currently exercises `X-Participant-ID` + `PermAdminRoleManage` against the role endpoints via the `PermissionCheck` middleware. A future task should add a test where a participant with/without `ADMIN_ROLE_MANAGE` hits POST /roles.
- **Server coverage**: `internal/server` is at 50.7%. The role handler paths are now covered; the remaining gap is in many other handler files that lack tests.
- **`GetByName` convenience method**: Adding `GetByName(name string) (*types.Role, error)` to the `RoleStore` interface would make application code cleaner (currently callers must List + scan). The store test already demonstrates this lookup pattern.
