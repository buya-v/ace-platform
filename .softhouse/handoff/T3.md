---

## Decisions Made

- `provision_tenant` reuses `POST /platform/v1/tenants` (same as `create_tenant`) — the distinction is in the description and output formatting, which emphasises schemas and Kafka topics for the infrastructure-provisioning use case.
- `governance_tier` defaults to `"STANDARD"` via zod `.default()` on `create_tenant`; `provision_tenant` hard-codes `"STANDARD"` since provisioning is always an initial operation.
- `get_tenant_config` renders known sections (trading_hours, settlement, circuit_breakers, features) as labelled sub-blocks and falls back to raw JSON for unrecognised config shapes.
- API paths use `/platform/v1/...` (not `/api/v1/...`) consistent with the platform control-plane namespace.
