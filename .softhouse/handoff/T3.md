## Handlers Implemented

### `handleSubmitOrder` — POST /api/v1/securities/orders

Validation (in order, matching architecture spec section 4):

1. `instrument_id` required → 422 if empty
2. Instrument must exist via `InstrumentStore.Get` → 422 NOT_FOUND if missing
3. Instrument `trading_status` must be ACTIVE → 422 INSTRUMENT_NOT_ACTIVE
4. `side` must be BUY, SELL, or SHORT_SELL → 422 INVALID_FIELD
5. SHORT_SELL side → 422 SHORT_SELL_DISABLED ("short selling is not currently enabled")
6. `order_type` must be LIMIT, MARKET, STOP, or STOP_LIMIT → 422 INVALID_FIELD
7. `quantity > 0` → 422 INVALID_FIELD
8. `quantity % instrument.LotSize == 0` → 422 INVALID_LOT_SIZE ("quantity must be a multiple of lot_size (N)")
9. For LIMIT/STOP_LIMIT: `price > 0` → 422; then `math.Remainder(price, tick_size)` — if `abs(remainder) > 1e-9` → 422 INVALID_TICK_SIZE ("price must be a multiple of tick_size (N)")
10. For STOP/STOP_LIMIT: `stop_price > 0` → 422 INVALID_FIELD

Defaults applied after all validation passes:
- `ID` = `newUUID()` (crypto/rand UUID v4, reusing helper from handlers_instrument.go)
- `Status` = PENDING
- `FilledQuantity` = 0, `AvgFillPrice` = 0
- `CreatedAt` and `UpdatedAt` = `time.Now().UTC().Format(time.RFC3339)`
- `TimeInForce` = GTC if empty

Returns 201 with the created order on success; 409 CONFLICT if the store rejects a duplicate ID.

### `handleListOrders` — GET /api/v1/securities/orders

- Query params: `instrument_id`, `participant_id`, `status` passed to `OrderStore.List` as `store.OrderFilters`
- `side` param applied as in-memory filter after the store call (`OrderFilters` has no side field)
- `limit` (default 50) and `offset` (default 0) pagination applied in-memory
- Returns `{"data": [...], "total": <pre-pagination count>, "limit": N, "offset": N}`
- JSON array is never null (initialised to empty slice when result is nil)

### `handleGetOrder` — GET /api/v1/securities/orders/{id}

- Extracts ID via `extractLastSegment()` (shared helper from handlers_instrument.go)
- Returns 404 NOT_FOUND when `store.ErrNotFound`
- Returns 200 with the order on success

### `handleCancelOrder` — DELETE /api/v1/securities/orders/{id}

- Extracts ID via `extractLastSegment()`
- Fetches order first to check status: only PENDING or PARTIALLY_FILLED may be cancelled → 409 INVALID_STATE ("order cannot be cancelled in current status")
- Calls `orderStore.Cancel(id)` (which transitions status in the store)
- Fetches the updated order and sets `UpdatedAt` on the returned object
- Returns 200 with the cancelled order (not 204 — callers need the final order state)

## Architecture Decisions

1. **SHORT_SELL checked before order-type validations.** The side check (c) runs first so users get the short-sell disabled error immediately — before quantity/price validation.

2. **`math.Remainder` for tick-size check.** `math.Remainder(price, tick_size)` computes the IEEE 754 remainder (range `(-tick_size/2, tick_size/2]`) which handles floating-point rounding better than naive `math.Mod`. The 1e-9 epsilon guards against accumulated float64 error.

3. **Cancel returns 200 + order body, not 204.** The task spec says "Return 200 with cancelled order." The old stub returned 204 with no body. The new handler fetches the post-cancel order and returns it so callers can observe the terminal status.

4. **Side filter is in-memory.** `store.OrderFilters` has no side field (the store interface was set in T1). Filtering by side is applied in-memory on the slice returned by `orderStore.List`, consistent with how instrument search filtering is done in `handleListInstruments`.

5. **Server struct unchanged.** The `Server` struct already held both `instrumentStore` and `orderStore` (added in T1). No struct changes were needed.

6. **No new imports in server.go.** All order handler logic is in `handlers_order.go`. The `server.go` dispatchers only route, maintaining the separation established in T2.

## Suggested Follow-ups

1. **Handler tests** — Add `internal/server/handlers_order_test.go` using `net/http/httptest`. Cover: submit with missing instrument_id, instrument not found, instrument not active, invalid side, SHORT_SELL disabled, invalid order_type, quantity=0, lot-size violation, tick-size violation (LIMIT), stop_price missing (STOP), valid submit for each order type, list with all filter combos, get 404, get success, cancel on FILLED order (409), cancel on PENDING order (200).

2. **`UpdatedAt` on cancel is response-only.** `handleCancelOrder` sets `UpdatedAt` on the response object but does not persist it to the store. `InMemoryOrderStore.Cancel` only transitions status. A follow-up should have the store set `UpdatedAt = time.Now()` on every state transition.

3. **`OrderFilters.Side`** — When the store is backed by PostgreSQL, adding a `Side types.OrderSide` field to `store.OrderFilters` and implementing it at the query layer will be more efficient than in-memory filtering.

4. **SHORT_SELL locate validation** — The architecture spec (section 4.4) describes a full locate-check workflow. When the locate service is available, replace the blanket rejection with a proper locate lookup.

5. **`participant_id` sourced from JWT** — Currently `participant_id` must be provided in the request body. In production, it should be extracted from the authenticated user's JWT claims.
