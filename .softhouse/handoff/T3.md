### topicWithoutPrefix Refactor

```go
// Old — hardcoded "ace." prefix (4 chars)
if len(topic) > 4 && topic[:4] == "ace." {
    return topic[4:]
}

// New — generic first-component stripper; works for any {tenant_id}.{domain}.{event}
if idx := strings.Index(topic, "."); idx >= 0 {
    return topic[idx+1:]
}
```

## Part B — Tenant-Aware Observability

### Discovery

The `shared/internal/tenant` package already existed and was complete:
- `context.go` — `TenantID` type, `WithTenant`, `TenantFromContext`, `MustTenant`
- `middleware.go` — `TenantMiddleware` (enforces `X-GarudaX-Tenant` header, validates against whitelist)
- `observability.go` — `TenantLogger`, `TenantMetricLabels`

No new tenant types or files were created. The task wired the existing package into observability middleware.

### Files Changed

**`src/shared/internal/observability/middleware.go`** (TracingMiddleware):
- Imports `shared/internal/tenant`
- Checks `tenant.TenantFromContext(r.Context())` after trace setup
- If tenant present: echoes `X-GarudaX-Tenant` in response header
- Appends `slog.String("tenant_id", ...)` to both `request_started` and `request_completed` log entries

**`src/shared/internal/observability/metrics.go`** (MetricsMiddleware):
- Imports `shared/internal/tenant`
- Checks `tenant.TenantFromContext(r.Context())` after request completes
- If present: prefixes metric key with `{tenant_id}_` (e.g. `ace-commodities_GET_200`)

**`src/shared/internal/observability/tracing.go`** (TraceMiddleware):
- Imports `shared/internal/tenant`
- Checks `tenant.TenantFromContext(r.Context())` after span creation
- If present: echoes `X-GarudaX-Tenant` in response, appends `slog.String("tenant_id", ...)` to trace debug log

**`src/shared/internal/observability/logger.go`** (LoggerFromContext):
- Imports `shared/internal/tenant`
- `LoggerFromContext` enriches logger with `tenant_id` when present in context
- New helper `LoggerWithTenantID(base, tenantID string)` for background workers where tenant comes from config

## Decisions Made

1. **topicWithoutPrefix generic** — made tenant-agnostic (strips first dot-separated component) rather than hardcoding `"ace-commodities."`. This means it works for any `{tenant_id}.{domain}.{event}` pattern including future `mse-equities.*` topics.

2. **Metric key prefix** — used string prefix (`ace-commodities_GET_200`) rather than a proper Prometheus label because the existing `counterMap` is keyed by strings. This is a stopgap; see Follow-ups.

3. **DLQ tenant prefix hardcoded** — DLQ topics are still prefixed with `"ace-commodities.dlq."` hardcoded in each service's `consumer.go`. Correct for now; each service handles one tenant.

4. **Did not create new tenant types** — the `shared/internal/tenant` package was already complete and correct. Reused `TenantID`, `TenantFromContext` as-is.

---

## Suggested Follow-ups

- **DLQ parameterisation**: Accept tenant_id as a parameter to `ConsumerConfig` so DLQ prefix is `{tenant_id}.dlq.*`. Required before mse-equities consumers are wired.

- **Prometheus labels**: Replace the metric key prefix approach with a proper label dimension (`tenant_id="ace-commodities"`). The `counterMap` needs to support multi-dimensional labels.

- **TenantMiddleware rollout**: Verify each service's HTTP server wires `TenantMiddleware([]string{"ace-commodities"})` before auth middleware. The middleware exists in shared but may not be wired in all services.

- **Kafka consumer tenant injection**: Add `tenant.WithTenant(ctx, tenant.TenantID(parseTenantFromTopic(topic)))` in `ChannelConsumer.processRecord` so Kafka handlers have tenant context matching the topic's tenant prefix.
