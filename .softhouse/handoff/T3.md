## Decisions Made

1. **PUT `/instruments/{id}/status` vs separate halt/resume** — The task spec requires a single `PUT /{id}/status` endpoint. The architecture doc has separate `POST /halt` and `POST /resume`. The PUT is more REST-idiomatic (setting a field) and subsumes both operations via the `status` enum in the request body. Added `resume_with_auction` boolean to handle the auction flag from the resume case.

2. **Unified `Instrument` schema** — Rather than separate equity/bond/ETF schemas, a single `Instrument` schema with nullable bond-specific fields. This simplifies client code. Bond-specific fields (`coupon_rate`, `maturity_date`, etc.) are `nullable: true` with descriptive comments. A discriminator can be added later.

3. **Settlements: 3 endpoints per task spec** — `GET /`, `POST /cycle`, `GET /netting`. The architecture has 7 settlement routes. Additional routes (affirm, buy-in, get-by-id, list-fails) are noted as follow-ups.

4. **Corporate actions: 2 endpoints per task spec** — `GET /` and `POST /`. The architecture has 6. Additional routes (PATCH, process, entitlements) are noted as follow-ups.

5. **Server base URL** — `https://api.garudax.mn/api/v1/securities` as the base, paths are relative. Matches `T033_openapi.yaml` pattern. The gateway registers these under `/api/v1/securities/*`.

6. **Example values** — All schemas include `example:` drawn from the architecture spec's JSON examples (`SEC-EQ-001`, `MN0000012345`, `APU.UB`, `PART-001`, etc.) for consistency with the architecture doc.

7. **`PositionSummary` for `/positions/summary`** — New schema not explicitly defined in the architecture doc. Fields derived from the position model: total_market_value, total_unrealized_pnl, total_realized_pnl, long_position_count, short_position_count.
