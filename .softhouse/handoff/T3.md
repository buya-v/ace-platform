# T3 Handoff — Loading States & Toast Feedback

## Summary

Implemented two cross-cutting UX improvements across 17 admin-ui page files.

## Part A — DataGrid loading={isLoading}

Added `isLoading` to `usePolling` destructuring and wired `loading={isLoading}` to every `DataGrid` component. When data is loading for the first time, DataGrid shows 5 skeleton rows instead of an empty table.

Pages updated with DataGrid loading:
- **Participants.tsx** — `const { data, refresh, isLoading }` → `<DataGrid ... loading={isLoading} />`
- **CircuitBreakers.tsx** — same pattern
- **AuditLog.tsx** — same pattern
- **MarketPhase.tsx** — same pattern
- **MarginCalls.tsx** — `calls.isLoading` passed to DataGrid
- **RiskOverview.tsx** — `calls.isLoading` passed to DataGrid (margin calls grid)
- **WarehouseOverview.tsx** — `receipts.isLoading` and `deliveries.isLoading` passed to respective DataGrids
- **Positions.tsx** — `positions.isLoading` and `netting.isLoading` passed to respective DataGrids
- **OrderBook.tsx** — `trades.isLoading` passed to trades DataGrid

Pages that do NOT use DataGrid (custom tables or card layouts) — isLoading destructured but no DataGrid to wire:
- Surveillance.tsx — custom `<table>` element
- ComplianceAlerts.tsx — card layout
- FeeManagement.tsx — custom `<table>` element (already had isLoading)
- Tickets.tsx — custom `<table>` element
- SettlementStatus.tsx — custom history list

### DashboardHome skeleton
Added `import { Skeleton }` and conditional skeleton rendering in:
- System Status card: shows `<Skeleton variant="text" height="24px" count={2} />` while health.isLoading and no data yet
- Pending KYC card: shows `<Skeleton variant="text" height="32px" />` while participants.isLoading and no data yet

### SystemMonitoring skeleton
Added `import { Skeleton }` and shows `<Skeleton variant="card" height="60px" />` before the overall banner when loading with no data.

## Part B — Toast feedback on mutations

Added `import { useToast } from '../contexts/ToastContext'` and `const { showToast } = useToast()` to all pages with mutations. Wrapped every mutation in try/catch with human-readable success/error messages.

| Page | Mutations wired |
|---|---|
| Participants.tsx | approve → 'Participant approved', reject → 'Participant rejected' |
| CircuitBreakers.tsx | halt → 'Trading halted for {ticker}', resume → 'Trading resumed for {ticker}', edit limits → 'Circuit breaker limits updated' |
| Surveillance.tsx | resolve → 'Alert resolved' |
| ComplianceAlerts.tsx | resolve → 'Compliance alert resolved' |
| Tickets.tsx | assign → '...set to in progress', status change → 'Ticket updated to {status}', priority change → 'Ticket priority set to {priority}', add comment → 'Comment added' |
| MarketPhase.tsx | halt → 'Market phase changed to HALTED for {name}', resume → 'Market phase changed to TRADING for {name}', halt all → 'All N markets halted' |
| MarginCalls.tsx | trigger calculation → 'Margin calculation triggered' |
| Reports.tsx | generate market summary → 'Market summary report loaded', generate large trader → 'Large trader report loaded', download CSV → 'Report exported' |

**DashboardHome** already had toast wired for `triggerSettlementCycle` and `massCancel` — no changes needed.

## Decisions Made

1. **Pages with multiple DataGrids** (Positions, WarehouseOverview): passed the most semantically relevant poll's isLoading to each DataGrid rather than a combined OR — this avoids skeleton showing on the wrong grid when only one poll is loading.
2. **Surveillance/Tickets/FeeManagement custom tables**: these use `<table>` elements directly, not DataGrid. isLoading is destructured (available) but no DataGrid to wire. Not adding skeleton to raw tables as that would require substantial structure changes.
3. **DashboardHome**: Only shows Skeleton on initial load (when `!data`). Subsequent background refreshes don't trigger skeleton to avoid flickering on a dashboard that's already showing data.
4. **SettlementStatus**: No DataGrid or mutations — only `isLoading` added to destructuring for future use.
5. **Reports.tsx export**: Used 'info' toast type (not 'success') per task spec.

## Blockers Found

None. All 400 tests pass, TypeScript compiles with zero errors.
